# scripts/no-reserve-allowlist.txt — grandfathered std::vector::reserve sites.
#
# Format: `path:text` (one per line, relative to project root).  `text`
# is the EXACT trimmed source of the violating call line (leading and
# trailing whitespace stripped) — a content key, NOT a line number.
# Comments start with '#'.  Blank lines ignored.
#
# ── Why content-keyed, not line-keyed (fix-18) ──────────────────────
#
# A LINE-KEYED entry (`path:line`) pins an exemption to a source LINE
# NUMBER.  Any edit above the call — a concurrent agent inserting a
# member, a new `#include`, a reflowed comment — shifts the line and
# silently invalidates the entry: the real reserve reads as an
# un-suppressed violation AND the now-dangling entry trips the
# stale-entry gate.  That is the exact failure mode this file kept
# hitting (memory documents two recent regressions; the six Cipher.h
# entries had already drifted by 5 lines when fix-18 landed).
#
# A CONTENT-KEYED entry pins the exemption to the call line's TEXT.
# The text is stable across line shifts, so a content-keyed exemption
# keeps tracking the same call no matter how the file above it churns.
# fix-18 eliminates the line number as the drift vector entirely; the
# mint-pattern guard took the same approach (name-keyed) — see
# scripts/mint-pattern-allowlist.txt.
#
# Content-keyed semantics: `path` must match the candidate's file
# exactly; `text` must equal the trimmed source of the call line
# byte-for-byte.  Each call line in a given file is unique by its
# trimmed text, so one entry maps to exactly one site.  A future
# duplicate-text call in the same file would (correctly) be covered by
# the same waiver — they share one migration rationale by construction.
#
# Every entry is a TODO awaiting a tracked migration.  When the
# referenced fixy-A5-* / WRAP-* ticket lands, delete the line — the
# guard then enforces no regression.  Stale entries (no matching call
# text in the file) red CI at exit 2 so the allowlist drains in
# lockstep with the §IV migration.
#
# Inline suppression (preferred for new code): `// NO-RESERVE-OK:
# <reason>` on the call line exempts that single line directly.
#

# tooling is refactored (no dedicated ticket yet).
