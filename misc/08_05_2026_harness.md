## 1. Executive summary

This document specifies how Crucible discharges contract-style verification conditions
without an SMT solver, without a GCC plugin, and without an external proof assistant.
It establishes the **WP harness** (the macro layer that fires at consteval AND debug-mode
runtime) and the **VC catalog** (a curated set of `consteval` decision procedures that
production code cites by name). Together these constitute Crucible's verification-
condition discharge surface — Tier 0 and Tier 1 in the cost ladder.

The document is written for the maintainer and the LLM-pair-programmer who will
extend the surface over the next ~10 commits. It is not a research paper; it is a
working spec for a discipline already partially shipping (`safety/Pre.h`,
`safety/Post.h`, `safety/Refined.h`, `safety/RefinedAlgebra.h`) and to be completed
by a concrete catalog (`safety/Decide.h`) and ~30 production-site cut-overs.

The hard call this document makes — and justifies in §8 — is: **stop at Tier 1**.
No Z3 plugin. No bespoke SMT solver. The constexpr-direct-eval surface covers ~95 %
of Crucible's measured bug classes; the remaining ~5 % are better solved by
structural type disciplines (sessions, permissions, rows) than by any solver.

## 2. Theoretical context (90 seconds)

### 2.1 Hoare logic

A Hoare triple `{P} c {Q}` reads: *if precondition P holds before command c, then
postcondition Q holds after*. P and Q are predicates over the program state; c is
a piece of code. The triple is a **partial-correctness** assertion (assumes
termination); strengthening to **total correctness** adds a `decreases` measure.

For Crucible, P is the implicit conjunction of: parameter validity (`x > 0`,
`p != nullptr`, `len ≤ N`), object invariants (the wrapped types' construction-
time guarantees), and ambient context (effects row admissibility). Q is what the
function promises about its return value and observable side effects.

### 2.2 Predicate transformer semantics

Dijkstra's WP calculus computes the *weakest* precondition that, given a command
c and a desired postcondition Q, ensures `{wp(c, Q)} c {Q}`.

```
wp(skip, Q)        = Q
wp(x := e, Q)      = Q[e/x]
wp(c1; c2, Q)      = wp(c1, wp(c2, Q))
wp(if b then c1 else c2, Q) = (b ⇒ wp(c1, Q)) ∧ (¬b ⇒ wp(c2, Q))
wp(while b inv I do c, Q)   = I ∧ (∀state. (I ∧ b ⇒ wp(c, I)) ∧ (I ∧ ¬b ⇒ Q))
```

The WP calculus reduces program correctness to predicate logic — at which point
a solver (or a constexpr decision procedure) can discharge the obligation.

For our purposes, **wp is computed by hand at the predicate's authoring site**.
The author writes the precondition that, when assumed, lets the body's algorithm
satisfy the post. The mechanism then enforces it.

### 2.3 Verification conditions

A **verification condition** (VC) is the formula that, if proven, establishes
Hoare-triple validity. For a function spec'd `requires P, ensures Q` with body
`c`, the VC is `P ⇒ wp(c, Q)`.

If the VC is *valid* (true in every model), the function is correct relative to
its spec.

In Crucible's harness, VCs are **named, decidable formulas over bounded integer
arithmetic + bitvector operations + finite set inclusion**. They are not
quantifier-alternated. They are not general first-order. They fit on a screen.
This is by design.

### 2.4 The discharge spectrum

Discharge mechanisms, ordered by strength and cost:

| Mechanism | What it covers | Cost | Crucible adoption |
|---|---|---|---|
| Type-system check (`Refined<P, T>`) | Single-leaf bounds at construction | Zero runtime; ≤ 1 µs compile | ✅ shipping |
| `static_assert` over compile-time constants | Closed-form arithmetic over literals | Zero runtime | ✅ shipping (~600 sites) |
| `CRUCIBLE_PRE / POST` direct evaluation | Function-entry / -exit predicates | Zero NDEBUG; debug runtime check | ✅ shipping (Tier 0) |
| Curated `Decide.h` decision procedures | Composed bounds, transitivity chains, set algebra | Zero NDEBUG; debug runtime check | 📋 Tier 1 (this doc) |
| Bespoke Presburger solver | Open-form linear arithmetic over unbounded ints | 100 ms to many seconds compile | ❌ rejected |
| Z3 via GCC plugin | Full SMT-LIB2 fragment | Plugin build + 1–10 s per VC | ❌ rejected |
| F\* round-trip | Full higher-order logic with extraction | Months of engineering, separate proof language | ❌ rejected |

The tier we adopt — direct evaluation over a curated catalog — is the
strongest mechanism whose discharge cost is bounded by C++ compiler evaluation,
not by a separate solver. That bound is what makes it shippable in a
nanosecond-budget codebase.

## 3. The state of the art in C++

### 3.1 P2900 contracts and the GCC 16.1.1 consteval bypass (per build)

GCC 16.1.1 ships P2900R14 contracts (`pre()`, `post()`, `contract_assert`).
The bypass behavior depends on whether the build includes the May 4 2026
cache-fix commit `ac1cdcdad9d`. Two builds, two different truths:

| Shape | Distro `gcc16-16.1.1-1.fc43` (May-1 snapshot) | Patched build (`+ac1cdcdad9d`) |
|---|---|---|
| 1. `pre(x > 0)` on `int x` (scalar by-value) | BYPASSED | FIRES |
| 2. `pre(s.nz())` on `S const&` | BYPASSED | FIRES |
| 3. `pre(s.nz())` on `S const` | BYPASSED | FIRES |
| 4. `pre(s.nz())` on `S` by-value | BYPASSED | FIRES |
| 5. `pre(s != nullptr && s->nz())` on `S const*` | FIRES | FIRES |
| 6. `post(r: r > 0)` on scalar return | BYPASSED | FIRES |
| 7. `post(r: r.v > 0)` on struct return field | BYPASSED | FIRES |

The mechanism on May-1 distro: when a `constexpr` function's body folds to a
literal (e.g. `return c.uuid.lo ^ c.uuid.hi` for a default-init `c`), GCC's
constexpr evaluator caches the folded result *without invalidating the cache
when a contract violation is detected*. Subsequent calls return the cached
literal — silently — and the contract clause is bypassed. This was the root
cause of the original Crucible cog_neg fixture compiling cleanly when it
should have failed.

The patched build's cherry-pick (commit `ac1cdcdad9d` on master, May 4 2026)
adds **4 lines** to `gcc/cp/constexpr.cc`:

```cpp
/* Only cache a result without contract violations.  */
if (!any_contracts_violated)
  cxx_eval_call_expression_cache (entry, result);
```

This invalidates the cached result when a contract fires, so the next call's
contract evaluation runs against the actual predicate rather than the cached
literal. All 7 shapes now fire correctly at consteval on the patched build.

**Conclusion** (revised 2026-05-08 PM):

- On the patched build we use, `pre()`/`post()` clauses CAN be the foundation
  of consteval-testable contracts. The cog_neg fixture works without
  CRUCIBLE_PRE.
- On the un-patched distro build, the bypass is real and CRUCIBLE_PRE is
  load-bearing.
- For codebase portability across both builds, we keep CRUCIBLE_PRE/POST
  shipped. They cost nothing in NDEBUG and are functionally identical to
  native `pre()`/`post()` on the patched build, but they're the only mechanism
  that works on the un-patched build.

**Caveat — the cache fix introduced its own regression**: see §13.6. Always-
true post-clauses on constexpr functions whose body folds to a literal now
fail with `"contract condition is not constant"` instead of silently passing.
This is a wash with the bypass it fixes (you trade a silent false-pass for a
loud false-fail), but it's a real cost of the patch.

### 3.2 LLVM's discipline

LLVM ships ~436 `assert(cond)` callsites in `Support/` headers alone. The
`llvm_unreachable(msg)` macro covers the "this case can't happen" axis. Both are
runtime-only (NDEBUG eliminates them); neither fires at consteval.

LLVM's `LLVM_BUILTIN_TRAP` shim and the `LLVM_UNREACHABLE_OPTIMIZE` CMake flag
are the two patterns worth cribbing. Neither requires Crucible-specific work
(we already have equivalents in `Platform.h`); both inform our cost-model
design.

LLVM has no analog of `CRUCIBLE_PRE` (consteval-firing), `CRUCIBLE_POST`
(post-state checking), or `Refined<P, T>` (type-encoded refinement).

### 3.3 mp-units / strong types

`mp-units` encodes physical quantities in the type system: `quantity<si::metre> x`
cannot be added to `quantity<si::second> y`. The contract is: "this value is in
units of metres, and any operation that does not preserve dimensional correctness
is a compile error."

Crucible's `Tagged<T, source::*>` is the analog axis (provenance), and
`Refined<P, T>` is the analog axis (value bounds). Same philosophy: the type
encodes the contract.

### 3.4 Boost.Contract, GSL

Boost.Contract: pre-P2900, full Eiffel-style `pre`/`post`/`inv` macros, runtime-
only, heavy. Historical precedent for the spelling.

GSL: `Expects()` / `Ensures()` macros. Same shape as `assert()`, no consteval.

Neither contributes design ideas Crucible doesn't already have.

### 3.5 What's missing across the entire C++ ecosystem

No shipping C++ project we audited has all four of:
1. Refinement types at construction (Crucible has `Refined<P, T>`)
2. Function-entry pre that fires at consteval AND debug-runtime
3. Function-exit post that fires at consteval AND debug-runtime
4. A named, citable catalog of decision procedures composing 1–3

Crucible's harness, when complete (Tier 0 + Tier 1), is novel by composition.
The components are individually proven (LLVM macros, mp-units strong types, JML
postconditions, Rust quantifiers); the integration is not.

### 3.6 The four P2900 evaluation semantics

GCC 16's `-fcontract-evaluation-semantic=` flag accepts four values, each with
a distinct runtime profile. All four are empirically verified working on the
patched build. The default is `enforce`.

| Semantic | Predicate evaluated? | Handler called? | Program continues? | Use case |
|---|---|---|---|---|
| `ignore` | No (DCE-eliminated) | No | Yes | NDEBUG production hot paths where the contract is documentation only |
| `observe` | Yes | Yes (logs only) | Yes | Production telemetry — fire violations to a sink, keep running |
| `enforce` (default) | Yes | Yes (logs) | No (calls `std::terminate` after handler returns) | CI / debug builds — fail loud |
| `quick_enforce` | Yes | No (skips handler) | No (`std::terminate` directly) | Performance-sensitive enforce paths where handler overhead matters |

**Empirical verification** (POC source in `/tmp/contract-poc/`, 2026-05-08):

```
$ for sem in ignore observe enforce quick_enforce; do
    g++-16p -std=c++26 -fcontracts -fcontract-evaluation-semantic=$sem ...
    ./poc
done

ignore     → A(-1) returns -2 silently (predicate elided)         exit 0
observe    → handler logs "[h] sem=2 expr=x > 0", program continues  exit 0
enforce    → handler logs "[h] sem=3 expr=x > 0", terminate         exit 134 (SIGABRT)
quick_enforce → no handler call, terminate                          exit 134
```

**Crucible's adoption pattern**:

- Default builds (`default` preset, debug + tests): `enforce` — the CMakePresets default, fail-fast on violation.
- `release` preset: `enforce` — production binaries also fail fast; performance is fine because contracts are mostly compiled out via `[[assume]]` propagation.
- `crucible_perf` static lib: `-fcontract-evaluation-semantic=ignore` — required because the static lib doesn't link `handle_contract_violation`. See `feedback_loader_tu_contract_semantic.md` memory for the rationale.
- Hypothetical Augur production telemetry preset (deferred per `feedback_augur_dependency_order.md`): `observe` — would let violations log to a Cipher event channel without crashing the runtime. The handler would emit a structured event; the program continues running on a degraded-but-known path. The hypothetical use case is a contracts-as-runtime-monitoring story, but Augur ships after Forge/Mimic/CNTP, so this preset is not yet needed.

**Cross-TU semantic mixing**: there is no `#pragma GCC contract_evaluation_semantic`
and no per-function attribute. Semantic is per-TU compile flag *only*. A static
library compiled with `=ignore` and a consumer compiled with `=enforce`
co-link cleanly — the contracts on the library's function bodies are
elided, the consumer's body-level assertions still fire. This is the
mechanism behind Crucible's `crucible_perf` carve-out.

### 3.7 The seven GCC 16 contract feature flags

Surveyed from `gcc/c-family/c.opt`:1903-1968 in `releases/gcc-16` HEAD.
All seven are stable and documented. Defaults shown are GCC's defaults.

| Flag | Values | Default | What it does |
|---|---|---|---|
| `-fcontracts` | (boolean) | OFF (auto-ON under `-std=c++26` since 2026-03-19, commit 62a3c3d5951) | Master enable for P2900 contracts. Crucible passes it explicitly because we use C++26 and want the behavior pinned, not implicit. |
| `-fcontract-evaluation-semantic=` | `ignore` / `observe` / `enforce` / `quick_enforce` | `enforce` | Per-TU semantic. See §3.6. |
| `-fcontracts-conservative-ipa` | (boolean) | ON | Disables IPA optimizations between functions in contract assertions. Workaround for PR121936 (still open) where IPA can incorrectly propagate `noreturn` from contract-terminating handlers into callers. Keep ON. |
| `-fcontract-checks-outlined` | (boolean) | OFF | Builds contract checks as outlined helper functions (`fn.pre`, `fn.post` clones) instead of inlining the checks into the body. Saves text size when many call sites share the same check pattern; costs one indirect call per check. Worth measuring on a release build with hundreds of contracts; not yet adopted in Crucible. |
| `-fcontract-disable-optimized-checks` | (boolean) | OFF | Disables the optimizer's elision of redundant contract checks. Useful for debugging; never for production. |
| `-fcontracts-client-check=` | `none` / `pre` / `all` | `none` | Caller-side checking: emit contract checks at the call site rather than (or in addition to) the callee body. Useful when the callee is in a separate TU with `-fcontracts-evaluation-semantic=ignore` but the caller wants to verify. | `-fcontracts-definition-check=` | `on` / `off` | `on` | Enable/disable contract checks on the definition (callee) side. Pairing knob for `client-check`. |

Plus the constexpr-related flags (also relevant for contract evaluation):

| Flag | Default | What it does |
|---|---|---|
| `-fconstexpr-depth=N` | 512 | Maximum recursion depth in constexpr evaluation. |
| `-fconstexpr-cache-depth=N` | 8 | Maximum recursion depth in the constexpr-evaluation cache. The cache that PR c++/124241 fixed. |
| `-fconstexpr-fp-except` | OFF | Allow constexpr FP exceptions to propagate. Off by default; on means `1.0/0.0` fails to constant-evaluate. |

### 3.8 Hard rules learned from GCC source (April-May 2026 audits)

Surveyed across `gcc/cp/contracts.cc`, `gcc/cp/constexpr.cc`, `gcc/cp/parser.cc`,
`libstdc++-v3/include/std/contracts`, and the May 2026 ChangeLog. These rules
fire as compile errors on the patched build; treat them as constraints, not
preferences.

1. **Virtual functions cannot have contracts.** GCC emits *"contracts cannot be
   added to virtual functions"*. Workaround: put the contract on a non-virtual
   public-API entry point that delegates to the virtual `do_*` worker.

2. **By-value parameter referenced in post must be `const`.** A `post (r: r > x)`
   where `x` is `int` (not `int const`) emits *"a value parameter used in a
   postcondition must be const"*. Mark by-value parameters `const` if you
   reference them in post; or use const-ref.

3. **Contract on declaration must match contract on definition.** A forward-decl
   with `pre(...)` and a definition with `post(...)` (or with no contracts) emits
   *"mismatched contract specifier in declaration"*. Either declare without
   contracts and define with them, or replicate the same set on both sides.
   Cherry-pick `b8b5f25c5a1` (PR c++/123456 area) is the relevant rule
   enforcement.

4. **Lambda contracts work** (since PR c++/124648, 2026-04-XX). The const-by-value
   rule applies (rule 2). Implicit captures referenced *only* in the contract
   assertion scope are now properly handled (PR c++/124824 area, April 2026).

5. **`contract_assert(cond)` is a parser keyword**, not a macro. Mid-body
   assertion. Same evaluation-semantic dispatch as `pre` / `post`. Behaves
   correctly at consteval on patched.

6. **`__builtin_observable_checkpoint()` is the optimizer fence** for partial
   correctness (P1494R5 + P3641R0). See §4.7.

7. **No pragma form, no per-function attribute** for evaluation_semantic. Per-TU
   compile flag is the only mechanism. Documented at §3.6.

8. **`__contracts_unchecked { ... }` does NOT exist in GCC 16.1.1.** Despite
   appearing in P3287R0 design discussion, the syntax is unimplemented.
   Workaround: factor recursion-safety-sensitive code into a separate function
   without contracts.

9. **The 4-line cache-fix commit `ac1cdcdad9d` is master-only** as of
   2026-05-08. Releases/gcc-16 has 78 backports for other fixes but not this
   one. Crucible cherry-picks it (see §0); the upstream may eventually backport
   in a 16.x point release.

10. **`std::observable_checkpoint()` is the C++26 wrapper** for `__builtin_observable_checkpoint()`
    in libstdc++ via P1494R5+P3641R0 implementation (commit 9056b5faa87 + 1e84849cb2a, both
    in releases/gcc-16). FTM `__cpp_lib_observable_checkpoint == 202506L`.

11. **The replaceable-contract-violation-handler ABI is ELF-only** (libstdc++
    gates it on `__ELF__`). On Linux/BSD the weak-symbol replacement works; on
    Mach-O / PE-COFF the macro `__cpp_lib_replaceable_contract_violation_handler`
    is undefined. Crucible's `src/ContractHandler.cpp` weak-handler approach is
    Linux-only by this constraint; document this if cross-platform builds are
    ever attempted.

12. **`std::contracts::contract_violation` carries 7 fields** per
    `<contracts>` header: assertion_kind (pre/post/assert), evaluation_semantic
    (ignore/observe/enforce/quick_enforce), detection_mode (predicate_false /
    evaluation_exception), comment (predicate text), `_M_src_loc_ptr` (a
    `std::source_location::__impl*`), version, vendor extension pointer. The
    structured object is the canonical violation payload — Crucible's
    `crucible::detail::contract_failed(expr, file, line, fn)` shim (planned per
    CONTRACT-005/006) will likely just construct one of these and forward to
    `handle_contract_violation`.

13. **Active development cadence (Jan-May 2026)**: 6 contract-related PRs
    (124241 cache fix, 124648 lambda captures, 124824 / 124855 / 125179 /
    125184 / 125206 reflection-via-contract interactions). Subsystem is in
    bug-fix-only mode on releases/gcc-16; new feature work targets GCC 17.
    Implication: pin the patched build for Crucible and don't chase trunk
    until at least stage 3 of GCC 17 (~Nov 2026).

## 4. Crucible's WP harness (Tier 0, shipped)

### 4.1 The if-consteval-trap pattern

The mechanism that makes consteval-firing contracts work in GCC 16.1.1 despite
the P2900 bypass:

```cpp
#ifdef NDEBUG
  #define CRUCIBLE_PRE(cond)                                            \
      do {                                                              \
          if consteval {                                                \
              if (!(cond)) [[unlikely]] { __builtin_trap(); }           \
          }                                                             \
          [[assume(cond)]];                                             \
      } while (0)
#else
  #define CRUCIBLE_PRE(cond)                                            \
      do {                                                              \
          if (!(cond)) [[unlikely]] {                                   \
              if consteval { __builtin_trap(); }                        \
              else         { std::abort(); }                            \
          }                                                             \
          [[assume(cond)]];                                             \
      } while (0)
#endif
```

The discriminator `if consteval` (P1938R3) is a compile-time branch: at consteval
time, the constexpr evaluator descends into the consteval branch. `__builtin_trap()`
is non-constexpr, so descent fails with "non-constant condition" — exactly the
diagnostic we want a `static_assert` to surface.

At runtime, the consteval branch is statically false; the entire body is dead
code. NDEBUG eliminates the trap path entirely; only `[[assume(cond)]]` survives,
and that emits no instructions — only an optimizer hint.

### 4.2 `CRUCIBLE_PRE(cond)` specification

**File**: `include/crucible/safety/Pre.h`

**Form**: `CRUCIBLE_PRE(boolean-expression)`

**Semantics** (per build mode):

| Build mode | Compile-time check | Runtime check | Optimizer hint | Code size |
|---|---|---|---|---|
| `-DNDEBUG` | YES (consteval evaluator hits trap on violation) | NO (eliminated) | YES (`[[assume(cond)]]`) | Zero bytes |
| `!-DNDEBUG` | YES (same) | YES (`std::abort()` on violation) | YES | One branch + cold call |

**Empirical cost** (measured on a 3-fn TU, GCC 16.1.1 -O1):
- Release-mode (NDEBUG): **1312 bytes**
- Debug-mode: **7568 bytes**

The 6256-byte delta is the runtime check + diagnostic infrastructure. Per-call
overhead in release: zero cycles. Per-call overhead in debug: one predicted-not-
taken branch.

**Parameter-shape coverage**: all seven probed shapes plus mid-body invariant
assertion (shape 9). See §A for the empirical probe.

### 4.3 `CRUCIBLE_POST(retvar, cond)` specification

**File**: `include/crucible/safety/Post.h`

**Form**: `CRUCIBLE_POST(name-of-return-variable, boolean-expression)`

Postcondition macro. Forwards to `CRUCIBLE_PRE` after a `(void)retvar` cast that
suppresses unused-variable diagnostics if the predicate happens not to reference
`retvar` directly.

The reason `Post.h` is a separate header (not embedded in `Pre.h`) is project
convention: one concept per file. The runtime mechanism is identical.

**Idiomatic usage**:
```cpp
constexpr T fn(int x) noexcept {
    CRUCIBLE_PRE(x > 0);
    T r = compute(x);
    CRUCIBLE_POST(r, r.is_well_formed());
    return r;
}
```

### 4.4 Cost model summary

The CRUCIBLE_PRE/POST contract pair is designed for **production ≡ optimizer
hint** + **debug ≡ assertion**. The cost model is:

```
NDEBUG → [[assume(cond)]]                 // 0 instructions
!NDEBUG → if (!cond) abort();             // 1 branch + cold path
         + [[assume(cond)]]               // optimizer hint preserved

CONSTEVAL (any build) → if (!cond) trap;  // surfaces as compile error
```

The triple alignment — production zero-cost, debug runtime-checked, consteval
compile-checked — is what makes the macros usable as both a runtime safety net
AND a static-test-fixture surface.

### 4.6 Composition with `[[assume]]`

After the runtime check (or its consteval equivalent) succeeds, the macro emits
`[[assume(cond)]]`. This propagates the invariant to the optimizer, which may:

- Eliminate redundant bounds checks downstream
- Vectorize loops whose iteration counts are now known finite
- Switch-densify when an enum value is bounded
- Devirtualize indirect calls when the receiver is now known non-null

The optimizer hint is the second-order win of contract discipline. Every
`CRUCIBLE_PRE(positive(x))` callsite tells the compiler `x > 0`, which downstream
cascade-removes overflow-check branches in arithmetic, signed-comparison
defensive paths, and impossible-case error returns.

This is *exactly* what F\* extraction-to-C achieves via VC discharge: post-VC-
proof, the invariant is a free fact for codegen. We get it without the solver.

### 4.7 Composition with `__builtin_observable_checkpoint` (deferred)

C++26 ships P1494R5 + P3641R0 ("Partial program correctness"), implemented in
GCC 16 via `__builtin_observable_checkpoint()` (compiler intrinsic, no header
required) and `std::observable_checkpoint()` (C++26 wrapper, in `<utility>`,
gated by `__cpp_lib_observable_checkpoint == 202506L`).

**What it does**: emits a *sequence point* that the optimizer is forbidden to
reorder code across with respect to UB-time-travel. Anything before the
checkpoint must have its observable side effects produced before the checkpoint
"completes" (in the abstract machine); anything after may exhibit UB without
poisoning the prior code.

**Why it composes with contracts**: aggressive UB-based optimization is
formally allowed to use `[[assume(cond)]]` to delete code BEFORE the assume
under "if cond is false here, the prior code couldn't have happened" reasoning.
This is technically sound under the C++ abstract-machine semantics but produces
spooky-action-at-a-distance bugs. The observable-checkpoint primitive marks a
boundary: "everything observable before this point must remain observable
regardless of what UB happens after". Pairing it with a runtime contract check
prevents the optimizer from reasoning backwards across the check.

**Why we don't yet adopt it**: in practice, GCC 16's optimizer is
non-aggressive about backward-UB-reasoning past `[[assume]]`. The theoretical
hole is real but unobserved; no Crucible miscompile has been traced to it. The
checkpoint primitive emits zero machine code (it's a compile-time barrier),
so adoption cost is zero — but adoption complexity (every CRUCIBLE_PRE site
needs a checkpoint inserted before the `[[assume]]`) is non-trivial.

**Hardening pattern, if/when adopted**: the macro becomes:

```cpp
#define CRUCIBLE_PRE(cond)                                            \
    do {                                                              \
        if (!(cond)) [[unlikely]] {                                   \
            if consteval { __builtin_trap(); }                        \
            else         { std::abort(); }                            \
        }                                                             \
        __builtin_observable_checkpoint();   /* NEW: epoch fence */   \
        [[assume(cond)]];                                             \
    } while (0)
```

Tracking task: not yet filed. Add when we observe the first miscompile or
when a security-sensitive contract path shows up where backward-UB reasoning
is a real concern. Until then, the doc-only awareness of the option is
sufficient.

## 5. Crucible's VC catalog (Tier 1, planned)

### 5.1 Why a curated catalog

The existing `Refined<P, T>` predicate library — `positive`, `non_zero`, `aligned<N>`,
`in_range<L,H>`, `bounded_above<M>`, `power_of_two`, `length_ge<N>`, `bounded_below<M>`,
`exact_size<N>`, `divisible_by<N>` — discharges leaf-level VCs at construction
time. CRUCIBLE_PRE discharges arbitrary boolean expressions at function entry.

What's missing is a **named catalog of common composed obligations** — the VCs
that recur across the codebase and that engineers currently re-encode as inline
boolean expressions. Naming them:

1. Makes them grep-able. `decide::no_overflow_mul` finds every site that depends
   on the multiplication-doesn't-overflow obligation.
2. Centralizes the implementation. A subtle bug in the bound check is fixed in
   one place, not 20 inline sites.
3. Makes them testable. A property-based test for `decide::no_overflow_mul`
   covers every production caller, not just the call site under test.
4. Documents the formal predicate. The catalog entry says exactly what's being
   discharged.

The catalog is the missing tier. It is **not** new logic — every piece of it
already exists as inline expressions or hand-rolled loops. The catalog is the
*naming* and the *centralization*.

### 5.2 `safety/Decide.h` structure

### 5.3 The decision procedures

Every procedure follows the convention:

- `[[nodiscard]] consteval bool name(args) noexcept;`
- One-line VC docstring at the top: `// VC: <formal predicate in standard math notation>`
- One-paragraph implementation rationale
- Pointer to the production sites that should cite it

#### 5.3.1 `no_overflow_mul` / `no_overflow_sum`

```cpp
// VC: a × b ∈ [T::min, T::max]
//
// Discharges the arithmetic-doesn't-overflow obligation for a single
// signed/unsigned multiplication.  Used by:
//   - MerkleDag::compute_storage_nbytes (sizes[d] product chain)
//   - PoolAllocator::is_initialized (pool_bytes computation)
//   - KernelCache slot count × bucket size sanity
template <std::integral T>
[[nodiscard]] consteval bool no_overflow_mul(T a, T b) noexcept;

// VC: ∀ k: |Σ_{i ≤ k} xs[i]| ≤ cap
//
// Discharges sum-of-bounded-series invariants where partial sums must
// also fit (the strict bound, not just the final sum).  Used by:
//   - PoolAllocator init slot offset accumulation
//   - MerkleDag region size walk
//   - TraceGraph CSR offset chain
template <std::integral T>
[[nodiscard]] consteval bool no_overflow_sum(std::span<T const> xs, T cap) noexcept;
```

#### 5.3.2 `all_in_range` / `strictly_increasing`

```cpp
// VC: ∀ i: Lo ≤ xs[i] < Hi
//
// Discharges range-constraint obligations over an entire span without
// hand-rolling a bounds-check loop.  Used by:
//   - TensorMeta::sizes[8] elements (each ≥ 0)
//   - SymbolTable::range_lower / range_upper field validation
//   - Forge phase trip-count bounds
template <typename T, T Lo, T Hi>
[[nodiscard]] consteval bool all_in_range(std::span<T const> xs) noexcept;

// VC: ∀ i: xs[i] < xs[i+1]
//
// Discharges strict-monotonicity obligations.  Stronger than weakly_increasing
// (no equality).  Used by:
//   - Cipher.LogEntry step_id sequence (no duplicates)
//   - Transaction count_ progression
//   - Vigil step_ counter
template <typename T>
[[nodiscard]] consteval bool strictly_increasing(std::span<T const> xs) noexcept;
```

#### 5.3.3 `is_power_of_two_le` / `factorization_eq`

```cpp
// VC: n > 0 ∧ has_single_bit(n) ∧ n ≤ Cap
//
// Discharges power-of-two capacity invariants.  Bit-count is via
// std::has_single_bit (C++20).  Used by:
//   - RecipePool::capacity_
//   - ExprPool::int_cache_ size
//   - SwissCtrl::kGroupWidth (must equal 16 on AVX2, 64 on AVX-512)
template <std::integral T, T Cap>
[[nodiscard]] consteval bool is_power_of_two_le(T n) noexcept;

// VC: Π factors = product (with overflow guard at each multiply)
//
// Discharges parallelism factor consistency.  The 5D parallelism config
// has TP × DP × PP × EP × CP == world_size as an invariant; this VC names it.
// Used by:
//   - 5D parallelism configuration validation
//   - ExecutionPlan worker_count × per_worker_batch == total_batch
template <std::integral T>
[[nodiscard]] consteval bool factorization_eq(std::span<T const> factors, T product) noexcept;
```

#### 5.3.4 `intervals_pairwise_disjoint`

```cpp
// VC: ∀ i ≠ j: [start_i, end_i) ∩ [start_j, end_j) = ∅
//
// Discharges sweep-line non-overlap obligations.  Used by:
//   - MemoryPlan slot offset assignment (slots can't share bytes)
//   - BackgroundThread pipeline phase scheduling
//   - Cipher cold-tier blob layout
//
// Implementation: O(n²) comparisons over the input span.  For our use cases
// (slot count typically ≤ 1024), this discharges in microseconds at consteval.
[[nodiscard]] consteval bool intervals_pairwise_disjoint(std::span<Interval const> ivs) noexcept;
```

#### 5.3.5 `tier_replaces` / `row_subset`

```cpp
// VC: from's numerical tolerance is ≥ to's numerical tolerance,
//     i.e., from is a sound substitute for to in any caller's recipe slot.
//
// The numerical tier order is: BITEXACT_STRICT < BITEXACT_TC < ORDERED < UNORDERED.
// "from replaces to" means we can swap a less-tolerant recipe in for a more-
// tolerant one with the caller still safe.
//
// Discharges:
//   - Forge Phase E.RecipeSelect substitution decisions
//   - KernelCache fallback chain (when a vendor backend can't compile a
//     specific recipe, fall to one this tier-replaces it)
[[nodiscard]] consteval bool tier_replaces(NumericalTier from, NumericalTier to) noexcept;

// VC: Row1 ⊆ Row2 (every effect in Row1 appears in Row2)
//
// Restated as a Decide.h entry of the existing Subrow concept, so contract
// sites can cite a named VC instead of a multi-template-parameter constraint.
//
// Discharges:
//   - All ctx-bound mint factories (CtxFitsX<X, Ctx>)
//   - Pipeline composition stages (CtxFitsPipeline)
//   - Forge phase row admission
template <typename Row1, typename Row2>
[[nodiscard]] consteval bool row_subset() noexcept;
```

#### 5.3.6 `fmix_preserves_non_zero` (domain-specific)

```cpp
// VC: ∀ in ≠ 0: fmix64(in) ≠ 0
//
// Crucible's content-hash function is a 64-bit Murmur-style fmix.  It must
// preserve non-zero-ness because zero is the sentinel for "uninitialized" /
// "empty slot" in KernelCache, MerkleDag, TraceGraph.
//
// Discharge: this is a property of the specific multiplication constants in
// fmix64; verified by exhaustive bit-fold-checking + a finite-domain probe
// of representative inputs at consteval.
//
// Discharges:
//   - cog::content_hash distinctness
//   - MerkleDag::merkle_hash propagation
//   - TraceGraph::content_hash invariant
[[nodiscard]] consteval bool fmix_preserves_non_zero(std::uint64_t in) noexcept;
```

#### 5.3.7 Composition combinators

```cpp
// VC: P(x) ∧ Q(x)
template <auto P, auto Q>
[[nodiscard]] consteval bool conjunction(auto const& x) noexcept {
    return P(x) && Q(x);
}

// VC: P(x) ∨ Q(x)
template <auto P, auto Q>
[[nodiscard]] consteval bool disjunction(auto const& x) noexcept {
    return P(x) || Q(x);
}

// VC: Pre(x) ⇒ Post(x)
template <auto Pre, auto Post>
[[nodiscard]] consteval bool implies(auto const& x) noexcept {
    return !Pre(x) || Post(x);
}

// VC: Lo ≤ x < Hi ∧ x ≡ 0 (mod Step)
template <auto Lo, auto Hi, auto Step>
[[nodiscard]] consteval bool aligned_in_range(auto x) noexcept;
```

These mirror `safety::all_of`, `safety::any_of`, `safety::implies` from
`RefinedAlgebra.h`. The duplication is deliberate: Decide.h is the catalog
of *VC names*; RefinedAlgebra.h is the algebra of *predicate composition for
Refined<P, T>*. Same lambdas, two namespaces, two doc surfaces.

### 5.4 Per-procedure complexity bound

Every procedure in Decide.h must satisfy:

1. **O(n) or O(n²) at most** in the input size (where n is bounded by the
   caller's static array length).
2. **Discharge in microseconds at consteval** for typical inputs (n ≤ 1024).
3. **Zero allocation** (consteval permits no heap).
4. **No recursion deeper than log(n)** (consteval has step limits).
5. **Returns bool, no exceptions** (consteval forbids throw under no-exceptions).

If a hypothetical decision procedure can't satisfy these, it doesn't belong in
Decide.h — it belongs in a session/permission/row discipline that catches the
obligation structurally rather than by evaluation.

### 6.2 Migration order

The order is dictated by:
- **Cascade safety**: migrate sites whose change cannot cascade-break a downstream contract first.
- **HS14 fixture compliance**: ≥ 2 negative-compile fixtures per new soundness gate.
- **Verifiability**: `ctest --preset default` must pass after every commit.

Recommended sequence:

1. Ship Decide.h foundation (commit 2 in §12 ladder)
2. Migrate single-site, single-obligation cases first (Arena, SymbolTable,
   TraceGraph indexed access)
3. Then composed obligations (PoolAllocator init, MemoryPlan)
4. Then domain-specific (cog::content_hash, Cipher, MerkleDag)
5. Lastly, the row_subset rebrand of existing concepts (compatibility-only,
   no behavior change)

### 6.3 Migration anti-pattern

Do **not** in a single commit:
- Migrate >5 production sites
- Add new Decide.h procedures and migrate sites
- Touch unrelated files

Each migration commit should be: one Decide.h cite swap + the corresponding
HS14 fixture(s), nothing else. Greppable, bisectable, revertable.

## 7. Test discipline

### 7.1 Positive runtime + consteval coverage

For each Decide.h procedure, ship in `test/test_decide.cpp`:

1. **Positive consteval test**: `static_assert(decide::name(known_good_input))`.
   This proves the procedure returns true on a known valid input AND that the
   procedure itself is callable from a `static_assert` context.
2. **Negative consteval test**: `static_assert(!decide::name(known_bad_input))`.
   The negation pattern; confirms the procedure returns false on a known
   invalid input, again at consteval.
3. **Runtime exhaustive probe**: a `for` loop over a small input universe (e.g.,
   all uint8_t values for a single-byte parameter) that asserts the procedure
   matches a reference implementation. Catches off-by-one in the procedure
   itself.
4. **Property-based fuzz**: random inputs from a deterministic Philox-seeded
   stream, comparing against the slow reference. This is in `test/fuzz_decide.cpp`
   (separate target, runs under fuzz preset).

### 7.2 HS14 negative-compile fixtures

For each Decide.h procedure, ship ≥ 2 `test/safety_neg/neg_decide_*.cpp` fixtures
demonstrating distinct misuse classes:

1. **Domain-violation fixture**: passes an input that violates the procedure's
   precondition (e.g., a negative value to `is_power_of_two_le`). The
   `static_assert(decide::name(bad_input))` must fail compilation.
2. **Composition-violation fixture**: composes two procedures whose obligation
   ranges don't overlap (e.g., `decide::all_in_range<10, 20>(span_of_5s)`).
   Different mismatch class than #1; surfaces composition errors that single-
   procedure checks miss.

For procedures with template parameters (e.g., `all_in_range<T, Lo, Hi>`),
add at least one **template-substitution fixture** that violates the constraint
on Lo < Hi at the type level.

Total HS14 fixture count: ~20 procedures × 2 = 40 minimum. Realistically 50+
once the composition-violation fixtures are exhausted.

### 7.3 Property-based testing layer

`test/fuzz_decide.cpp`:

```cpp
// For each Decide.h procedure, deterministic Philox-seeded fuzz against a
// slow reference implementation.  Run for 10K iterations per procedure.
// Catches subtle sign/overflow/edge-case bugs in the procedure itself that
// HS14 fixtures might not surface.
//
// Output: silent on success, abort on first divergence with the offending
// input dumped to stderr.  Re-runnable with the same seed for bug isolation.

TEST_FUZZ(no_overflow_mul) {
    for (auto [a, b] : philox_pairs<int64_t>(10'000)) {
        auto fast = decide::no_overflow_mul(a, b);
        auto slow = reference_no_overflow_mul(a, b);  // big-int based
        ASSERT_EQ(fast, slow);
    }
}
```

### 7.4 The exhaustive-finite-domain pattern

For domain-specific procedures whose input universe is small (`tier_replaces`
over the four NumericalTier enum values, or `coprime` over u8 × u8), ship an
**exhaustive consteval table check**:

```cpp
// At consteval time, iterate every (from, to) pair and assert the procedure
// agrees with a hand-computed truth table.  Compile-time only; fails compile
// on disagreement.
static_assert(consteval_exhaustive_check_tier_replaces());
```

This is the strongest possible discharge: not "we tested 10K random cases" but
"we proved it on every single input." Available for any procedure whose input
universe fits in a constexpr loop.

## 8. Beyond Tier 1: the deliberately-unbuilt

This section documents the choices we are NOT making, with rationale. Including
them here prevents future engineers from reopening these questions without the
context that was already weighed.

### 8.1 Tier 2: Z3 via GCC plugin (rejected)

**What it would buy**: SMT-LIB2 fragment over linear arithmetic + bitvectors +
arrays + ADTs. Discharge of cross-function obligations the constexpr surface
can't reach (e.g., "every caller of mint_X passes a Ctx whose row admits Y").

**What it would cost**:
- ~3 weeks of engineering for VC extraction from GCC AST
- New runtime dependency on libz3 in build environment
- Plugin ABI fragility across GCC point releases
- 1–10 second per-VC discharge time at compile (vs. microseconds for direct
  evaluation)
- Z3 timeout / incompleteness on hard fragments

**What kills it**:
- ~5 sites total in Crucible would benefit. Cost-per-site is ~3 days of
  engineering. Compare against Refined<>+CRUCIBLE_PRE+Decide.h, which gives
  ~30 sites at ~3 hours each.
- Frama-C/WP is the cautionary tale: 15 years of engineering on VC extraction
  for C; the team's bandwidth is consumed by keeping VC generation correct
  through language evolution. Crucible doesn't have that bandwidth.
- The cross-function obligations we have are all caught structurally by
  session types, permission tokens, and effect rows. Adding a solver doesn't
  catch new bugs; it duplicates structural enforcement at higher cost.

### 8.2 Tier 3: bespoke SMT solver (rejected)

**What it would buy**: A small constexpr Presburger solver with no external
dependency. Could discharge pure-LIA obligations at consteval.

**What it would cost**:
- ~1 month of careful engineering (DPLL(T) + Omega test)
- ~3000 LOC of constexpr C++
- Slower than Z3 by orders of magnitude
- Strictly weaker than Z3 (no bitvectors, no arrays, no ADTs)

**What kills it**:
- We don't have unbounded LIA obligations. Every Crucible loop has a bounded
  trip count. Every index has a static upper bound. The Decide.h direct-eval
  surface covers every Presburger obligation we actually have.
- Building a solver is a research project. We are not researchers; we are
  engineers shipping a runtime.

### 8.3 Tier 4: Daikon-style invariant inference (rejected)

**What it would buy**: Automatic discovery of likely invariants by observing
runtime behavior, then synthesizing pre/post clauses.

**What it would cost**: Months of work. Brittle. Generates noise.

**What kills it**: Crucible designs invariants up front. We know what each
function should require and ensure because we wrote the spec. We don't need
inference; we need enforcement.

### 8.4 Where the scope ends

**Crucible discharges VCs for**:
- Single-leaf bounds at construction time (Refined<>)
- Function-entry preconditions (CRUCIBLE_PRE)
- Function-exit postconditions (CRUCIBLE_POST)
- Composed bounds and transitivity chains (Decide.h)
- Set-inclusion and effect-row admission (Subrow / row_subset)
- Permission and resource ownership (Linear, Permission)
- Protocol typestate (Session)

**Crucible does NOT discharge VCs for**:
- Floating-point reasoning (no IEEE 754 axioms in the catalog)
- Concurrent invariants under arbitrary interleaving (TSan + permission
  discipline catch these structurally)
- Information-flow security (we have ConstantTime<> + Secret<> wrappers; no
  taint analysis)
- Termination of unbounded loops (we don't have unbounded loops)

The decision boundary is: if a VC class doesn't fit constexpr-direct-eval over
bounded inputs, we don't catch it with the harness. We catch it with another
discipline (cross-vendor CI, TSan, Tagged provenance) or we accept that we
don't catch it and rely on measurement (the L0 axiom: "soundness via
measurement, not proof").

## 10. Composition with the rest of Crucible

### 10.1 Refined<P, T> overlap (no double-enforce)

**Rule**: if a function takes `Refined<P, T> x` as a parameter, do **not** also
write `CRUCIBLE_PRE(P(x.unwrap()))`. The Refined wrapper enforced P at
construction time; double-checking is dead code.

```cpp
// Wrong — Refined<positive, int> already proves x > 0.
constexpr int wrong(safety::Refined<safety::positive, int> x) noexcept {
    CRUCIBLE_PRE(x.unwrap() > 0);  // redundant
    ...
}

// Right — the type system carries the obligation.
constexpr int right(safety::Refined<safety::positive, int> x) noexcept {
    int n = x.unwrap();  // n > 0 is now a free fact for the optimizer
    ...
}
```

A grep-based CI check (`scripts/check-refined-pre-subsumption.sh`) flags
double-enforcement for review. Lazy ship; not blocking.

### 10.2 effects::Row<...> orthogonality

The effect-row axis (Cap, IO, Block, Init, Bg, Test) is enforced by Subrow
concept at concept-substitution time. Decide.h's `row_subset` is a renamed
view of the same machinery, exposed for diagnostic naming.

CRUCIBLE_PRE does **not** enforce row admission — that's the concept's job,
not the body's. CRUCIBLE_PRE handles value-level invariants only.

### 10.3 Permission<Tag> / Linear<T> orthogonality

Ownership invariants are caught structurally by the move-only Permission tokens
and Linear wrappers. CRUCIBLE_PRE does **not** check "does this caller hold
the permission?" — the type system already proves that the caller couldn't have
gotten the function's input without the permission.

### 10.4 Session<Proto> orthogonality

Protocol invariants (e.g., "send before recv", "close before drop") are caught
by Session typestate. CRUCIBLE_PRE does **not** check protocol order — the type
system enforces it via the move-only handle threading.

The net rule: **CRUCIBLE_PRE handles what the structural disciplines can't
encode**. Value bounds, value relations, post-state-vs-pre-state diffs, span
quantifications. Everything that the type system already encodes is left to the
type system.

## 13. Anti-patterns

Things to avoid as we extend the harness:

### 13.1 Don't add a procedure for a single-site invariant

Decide.h is a **catalog of recurring obligations**. If an invariant is only
needed in one place, leave it as inline `CRUCIBLE_PRE(...)`. The catalog earns
its keep through reuse; one-off entries are noise.

### 13.2 Don't silently relax a procedure's specification

If a Decide.h procedure's docstring says "VC: a × b ∈ [T::min, T::max]",
the implementation MUST decide that VC. Adding an `else return true;` "as a
relaxation" silently weakens every cite-site's contract. Procedures are
deletable but not weakenable.

### 13.3 Don't use Decide.h procedures as runtime-only checks

Every Decide.h procedure MUST be `consteval` (not `constexpr`). This forces
the discharge into the consteval surface. If a use case demands runtime
discharge over arbitrary input, that's a different procedure — name it
differently and ship it elsewhere.

### 13.4 Don't compose Decide.h with non-decide predicates

Mixing a Decide.h procedure with a runtime-only helper inside a CRUCIBLE_PRE
loses the consteval guarantee. Either both are consteval (and the whole
expression discharges at consteval) or both are runtime-only.

### 13.5 Don't migrate without HS14 fixtures

Every Decide.h procedure ships with ≥ 2 negative-compile fixtures. Adding a
procedure without fixtures is a soundness gap: nothing proves the procedure
fires at consteval, and a regression that silently relaxes it goes undetected.

### 13.6 Don't write always-true `post()` on foldable-body constexpr

This is the regression introduced by the very cache-fix commit `ac1cdcdad9d`
that we cherry-pick. Diagnostic on patched build: *"contract condition is not
constant"*. Reproducer:

```cpp
// FAILS to compile on patched build (over-invalidation of contract cache)
[[gnu::const]] constexpr int8_t h2_tag(uint64_t hash) noexcept
    post (r: r >= 0)
{
    return static_cast<int8_t>(hash >> 57);  // body folds to a literal
                                             // for any constexpr-known hash
}
```

The mechanism: when `h2_tag` is called from `static_assert(h2_tag(0) == 0)`,
the body folds to the literal `0`. The patched constexpr evaluator now
re-evaluates the post-clause every call (cache invalidation works correctly),
but the post-clause asks "is `0 >= 0`?" — and the cache-fix patch
over-invalidates the *predicate's* result too, treating it as
"not constant" instead of "constant true".

**Three workarounds, in preference order**:

1. **Drop the post() entirely if static_asserts at the boundary already prove
   it.** SwissTable.h `h2_tag` had `static_assert(h2_tag(0)==0)`,
   `static_assert(h2_tag(UINT64_MAX)==127)`, `static_assert(h2_tag(0x8000…)==64)`
   right below the function — those three boundary-input proofs are STRONGER
   than the post (they pin specific values; the post only pins a one-sided
   inequality). The post was redundant documentation. Drop it.

2. **Convert to CRUCIBLE_POST.** Same diagnostic as native post but routed
   through the macro. Caveat: incompatible with `[[gnu::const]]` because
   CRUCIBLE_POST in debug calls `crucible::detail::contract_failed` which has
   side effects, violating const-purity.

3. **Rewrite the predicate so it isn't trivially-always-true.** E.g., for
   `h2_tag` the bit-7-clear invariant could be stated as
   `post (r: (static_cast<uint8_t>(r) & 0x80) == 0)` — semantically identical
   to `r >= 0` but expressed via bitmask, which the constexpr evaluator
   doesn't fold to a constant in the same way.

**Trip-pattern detection**: a native `post (r: P(r))` clause is at risk on
patched GCC iff:
- (a) the function is `constexpr`, AND
- (b) the body folds to a literal under at least one argument set, AND
- (c) `P(r)` is provably-always-true for that folded value.

Any one of (a)/(b)/(c) missing → the site is safe. The other four native posts
in Crucible's production tree (Arena, Transaction ×2, Types) all fail (b) —
their predicates reference runtime state — and survived the cache fix unchanged.

**This is not a bug we filed upstream**. The wash is: cache fix closes a
silent-false-pass bug (worse) and opens a loud-false-fail bug (better).
Loud-false-fail is grep-able and fixable; silent-false-pass is invisible.
Net: positive trade, even though it costs us the SwissTable.h post-clause.

## 14. Glossary of WP/VC terminology

Crucible jargon, with cross-references:

- **Hoare triple** — `{P} c {Q}`: precondition, command, postcondition. The
  basic unit of program correctness reasoning.
- **WP / weakest precondition** — `wp(c, Q)` is the weakest predicate that,
  when assumed before `c`, guarantees `Q` after.
- **VC / verification condition** — the formula that, if proven, establishes
  a Hoare triple. For a function `requires P, ensures Q` with body `c`, the
  VC is `P ⇒ wp(c, Q)`.
- **Discharge** — the act of proving a VC. In Crucible, by direct constexpr
  evaluation of the relevant Decide.h procedure.
- **Refinement type** — a type tagged with a predicate: `{x : T | P(x)}`.
  Crucible's `Refined<P, T>` is the encoding.
- **Predicate transformer** — a function from postconditions to preconditions.
  WP is the canonical predicate transformer.
- **Frame condition** — what the function does NOT modify. In Crucible,
  enforced structurally by `Linear<T>`, `Permission<Tag>`, and the absence of
  shared mutable state.
- **Decreases / variant** — a measure that strictly decreases on each loop
  iteration, proving termination. In Crucible, replaced by `Monotonic<T,
  std::greater>` on loop counters or by bounded iteration over `std::array`.
- **Triple subsumption (Liskov substitutability)** — overriding function's
  precondition is weaker, postcondition is stronger. In Crucible, structurally
  enforced by template parameter constraints (the override's concept must
  imply the base's).

## 15. Appendix A: GCC 16.1.1 consteval bypass — empirical probe

**Probe source** (`/tmp/probe_full.cpp`):

constexpr int p1(int x) noexcept pre (x > 0) { return x * 2; }

constexpr unsigned long long p4(S s) noexcept pre (s.nz()) { return s.lo; }
// Probe 5: pre on struct by-pointer (the only one that works)
constexpr unsigned long long p5(S const* s) noexcept pre (s != nullptr && s->nz()) { return s->lo; }
// Probe 6: post on scalar return
constexpr int p6(int const x) noexcept post (r: r > 0) { return x + 1; }

## 16. Appendix B: Decide.h reference implementation sketch

A subset of the catalog, showing the documentation + implementation
discipline for the catalog. **NOT** the final commit; this is the design
template for commit 3 of the ladder in §12.

```cpp
// SPDX-License-Identifier: Apache-2.0
// crucible — safety/Decide.h
//
// Curated catalog of verification-condition decision procedures.
//
// Every procedure in this header satisfies:
//   * `[[nodiscard]] consteval bool name(args) noexcept`
//   * Documented VC predicate in standard math notation
//   * O(n) or O(n²) at most in input size
//   * Discharges in microseconds at consteval for typical inputs
//   * Zero allocation, no recursion deeper than log(n)
//
// Cite sites use the procedures via CRUCIBLE_PRE / CRUCIBLE_POST or as
// `Refined<decide::name, T>` predicates.
//
// See: misc/08_05_2026_harness.md for the full design document.

#pragma once

#include <crucible/Types.h>
#include <crucible/safety/Pre.h>

#include <concepts>
#include <cstdint>
#include <span>

namespace crucible::safety::decide {

// ── Numerical bounds ──────────────────────────────────────────────

// VC: a × b ∈ [T::min, T::max]
//
// Discharges the multiplication-doesn't-overflow obligation for a single
// signed/unsigned multiplication.  Used by:
//   - MerkleDag::compute_storage_nbytes
//   - PoolAllocator::is_initialized
//   - KernelCache slot count × bucket size
template <std::integral T>
[[nodiscard]] consteval bool no_overflow_mul(T a, T b) noexcept {
    using L = std::numeric_limits<T>;
    if (a == 0 || b == 0) return true;
    if constexpr (std::signed_integral<T>) {
        if (a > 0 && b > 0) return b <= L::max() / a;
        if (a < 0 && b < 0) return b >= L::max() / a;  // careful: T(-1) division
        if (a > 0) return b >= L::min() / a;
        return a >= L::min() / b;
    } else {
        return b <= L::max() / a;
    }
}

// VC: ∀ k: |Σ_{i ≤ k} xs[i]| ≤ cap, with no overflow at any partial sum.
//
// Discharges sum-of-bounded-series.  Used by PoolAllocator slot offset chain.
template <std::integral T>
[[nodiscard]] consteval bool no_overflow_sum(std::span<T const> xs, T cap) noexcept {
    T acc = 0;
    for (T x : xs) {
        // additive overflow check
        if (x > 0 && acc > cap - x) return false;
        if (x < 0 && acc < std::numeric_limits<T>::min() - x) return false;
        acc += x;
    }
    return acc <= cap;
}

// ... (etc., see commits 4-8 in the ladder)

}  // namespace crucible::safety::decide
```
