### 0.4 Decision log

The user has explicitly committed to "superpowers in C++ with frontier applied type theory" and authorized refactoring up to 10% of existing code to support them. This pins the following decisions, which prior sketches left open:

- **D1.** All ~25-30 wrapper categories ship in the v1 catalog. No deferrals. Each chosen category is justified in §4 by both an academic citation and a production call site.
- **D2.** `Computation<R, T>` becomes a thin alias for `Graded<Modality::Relative, EffectRowLattice<OsUniverse>, T>`. Existing call sites see no change (the alias preserves the surface). The substrate sees uniformity. The reserved `Modality::Relative` slot (`algebra/Modality.h:51`) is finally colonized.
- **D3.** `Row<Es...>` gains a `Lattice`-conforming adapter (`EffectRowLattice<Universe>`) without losing its compile-time-set semantics. The existing `Row<Effect::Bg>` continues to work; new code can also use the value-level lattice surface.
- **D4.** The `KernelCacheKey` is extended additively from `(content_hash, device_capability)` to `(content_hash, device_capability, row_hash)`. Existing entries continue to work as `Row<>` (empty-row default). No invalidation cascade.
- **D5.** PermissionEffect is NOT a row category. CSL permissions stay parallel substrate per `safety/Safety.h:42-90` SESSIONS-AUDIT decision and the 27_04_csl_permission_session_wiring §2.4-§2.5 architectural commitment. Sessions stay structural typestate. Three substrates compose orthogonally; we do NOT collapse them into one ProductLattice.
- **D6.** FOUND-E (diagnostic infrastructure) MUST land in Month 3, before further wrapper proliferation. Without it, 19-wrapper failures generate 5000-line backtraces. This is non-negotiable.
- **D7.** Lean proofs for new lattices ship as part of the wrapper PR, not as a separate Phase. The `lean/Crucible/` development grows by ~12 modules in concert with the C++ work, not after.
- **D8.** The 9 deliberately-not-graded wrappers (Machine, ScopedView, OwnedRegion, Pinned, Checked, ConstantTime, NotInherited, Simd, Workload, per `safety/Safety.h:48-86`) STAY structural. The substrate completion does NOT migrate them. They compose with Graded wrappers via documented bridges (already the discipline in `bridges/`).
- **D9.** No breaking changes to public APIs. The 10% refactor surface is the union of: (a) `Computation` becoming an alias, (b) `Row` gaining a Lattice instance, (c) cache key extension, (d) Cipher::record_event gaining a `requires` clause, (e) ~50 hot-path call sites adopting row-typed signatures. ## 1. Thesis — superpowers via frontier applied type theory

### 1.1 The one-paragraph pitch

Crucible already ships graded modal types, quantitative type theory, Concurrent Separation Logic, multi-party session types, and Met(X) effect rows — but as separate substrates. The substrate completion unifies them under one calling convention: **every type-level fact about computation in Crucible is a `GradedWrapper`-conforming wrapper, every wrapper carries its lattice via reflection, every downstream component reads behavior through one substrate, and the KernelCache uses the row as part of its key.** Six months of focused work; ~10K LoC; ~25-30 new wrappers; the substrate is consumed by Forge / Mimic / Vigil / Keeper / Cipher / Augur / Canopy / Vessel / Distribution. After landing, Crucible is the first production C++26 codebase with full Met(X) row arithmetic + Graded modal type composition + GradedWrapper-introspectable dispatcher + content-addressed federation cache, validated by mechanized Lean proofs. The closest analogs (Linux sparse: 10 annotations; MLIR effect interfaces: IR-level only; F* in HACL*: not C++; ZIO at Netflix: typeclasses not rows; Haskell `effectful`: not C++) each ship one ingredient. The combination has not been done at production scale anywhere.

### 1.2 The five superpowers, named

**Superpower 1 — Compile-time DetSafe.** The 8th axiom (CLAUDE.md §II.8) is the only un-fenced one today, validated by `bit_exact_replay_invariant` CI tests. After substrate completion, `Cipher::record_event` requires `Subrow<event_row, Row<Effect::Pure, Effect::PhiloxRng>>` at the type level. Accidentally recording a `MonotonicClockRead` into the replay log is a compile error at the call site, not a CI failure 12 hours later.

**Superpower 2 — Universal cache vocabulary.** Every L1 / L2 / L3 KernelCache entry is keyed on `(content_hash, device_capability, row_hash)`. Two organizations running Llama-70B share L1 entries because rows are vendor-neutral and content-addressable across machines. Federation works without protocol negotiation.

**Superpower 3 — Reflection-driven dispatcher.** FOUND-D's dispatcher is generic over `GradedWrapper`-conforming parameters (`algebra/GradedTrait.h:222-269` already specifies the contract). One ~500-line dispatcher consumes 25-30 wrappers via reflection. Per-lowering-target row preconditions enforce hot-path / cold-path / replay-safe contracts at compile time.

**Superpower 4 — Row-typed Forge IR.** IR001 nodes carry rows describing structural properties (`Pure`, `Commutative`, `FpReassociation`, `ReductionOrderChange`). REWRITE phase fires row-disjoint rules in parallel via interaction-net pattern (Lafont 1990). FUSE phase reads cost-row to decide eligibility. IR002 KernelNodes carry rows describing recipe tier, vendor target, fusion eligibility. Cross-vendor numerics CI extends to row-equivalence verification.

**Superpower 5 — Augur drift attribution per-axis.** Today Augur reports "5% slower" as one scalar. After substrate completion, drift correlates with row dimension: "HotPath axis dropped from Hot to Warm; the cost overrun is consistent with bg-thread scheduling latency, not kernel-time degradation." The recommendation engine cites specific axes; remediation becomes targeted, not opaque.

### 1.3 Why this is "frontier"

The five algebraic substrates Crucible relies on — graded modal types (Orchard-Liepelt-Eades 2023, [arXiv:2309.04324]), quantitative type theory (Atkey FLoC 2018), Met(X) modal effects (Tang-Lindley POPL 2026, [arXiv:2507.10301]), Concurrent Separation Logic (O'Hearn 2007), multi-party session types (Honda 1998, HYC 2008, BSYZ22) — are all from the last 5-25 years of PL research. The combination of all five in one production C++ codebase has not been done. The supporting C++26 features (P2900 contracts, P2996 reflection, P1306 expansion statements, P3491 `define_static_array`, P2795 erroneous behavior for uninit reads) only landed in GCC 16, which only stabilized in March 2026.

The substrate is genuinely first-of-its-kind. The novel part is not any individual ingredient — each has prior art. The novel part is the **integration**: row arithmetic + graded modalities + linear permissions + multi-party sessions + interaction-net rewriting + content-addressed federation, all in one C++26 codebase, all validated by mechanized Lean proofs. The substrate authors (the Crucible team, present and past) have already shipped 8343 lines of substrate code and ~1300 Lean theorems. The completion is finishing what they started.

### 1.4 What the substrate gives that no individual piece does

- **Composition is mechanical.** Wrapper-nesting (Stale<Tagged<Refined<positive, Computation<Row<Bg>, T>, source::FromUser>>>) composes 5 axes of behavior at zero runtime cost via EBO. No individual algebra gives you all 5 axes.
- **Cross-cutting properties get a name.** "Hot-path safe" is one wrapper. "Replay-safe" is one wrapper. "Recipe-tier-pinned" is one wrapper. Today these are 8 different mechanisms (see §1.6 below).
- **Dispatcher is one piece of code.** FOUND-D ships once; consumes 25-30 wrappers via GradedWrapper reflection. No per-wrapper specialization in the dispatcher.
- **Cache federation is automatic.** Rows are vendor-neutral, content-addressable, stable across machines. L1 cache entries become naturally shareable.
- **Bug reports cite specific axes.** "DetSafe leak at Cipher.h:431 — row contains MonotonicClockRead but Cipher requires Pure." Not "test_bit_exact failed; investigate."

### 1.5 Honest scope

Six months. ~10K LoC. ~25-30 new wrappers. ~50 production call site migrations. ~12 new Lean modules. Three substrate boundaries acknowledged and respected: (1) the 9 deliberately-not-graded wrappers stay structural; (2) CSL permissions stay parallel substrate (not a row axis); (3) sessions stay structural typestate (not graded). The 10% refactor surface is precisely enumerated in §3. The 90% of existing substrate that this builds on is enumerated in §2.

### 1.6 The eight mechanisms today, collapsed to one

Today, "what behavior happens at this function call" is encoded across:

1. Runtime fields on structs (`NumericalRecipe.determinism`, `KernelNode.kind`, `Vigil::Mode`, `Cipher::Tier`)
2. Permission tags (`Permission<Tag>`, `PermSet<Tags...>`, `SharedPermissionPool<Tag>`)
3. Session typestate (`SessionHandle<Proto>`, `PermissionedSessionHandle<Proto, PS, Resource>`)
4. Cap-tag values (`effects::Alloc`, `effects::IO`, `effects::Block`, `effects::Bg/Init/Test` contexts)
5. Comment discipline ("hot path", "fg thread", "ARM dmb cost is 1 cycle here")
6. Code review enforcement ("we do not seq_cst here", "no syscall on hot path")
7. CI tests (`bit_exact_replay_invariant`, `cross_vendor_step_invariant`, `fleet_reshard_replay`)
8. Augur runtime thresholds (10% drift over 100 samples)

These are EIGHT MECHANISMS for the SAME KIND OF FACT. Every component has to translate between them. New components have to learn all eight. Cross-cutting properties (DetSafe, hot-path safety, recipe tier) are scattered across all eight.

After substrate completion: ONE substrate (`Graded<M, L, T>` + `GradedWrapper` concept + `Row<Es...>` lattice bridge), ONE introspection surface (FOUND-D's `signature_traits<FnPtr>`), ONE diagnostic surface (FOUND-E's per-category errors), ONE cache key extension, ONE Lean foundation. Mechanisms 1, 4, 5, 6 collapse into Graded wrappers. Mechanisms 2, 3 stay parallel substrates (composing via documented bridges). Mechanism 7 becomes confirmation, not primary defense (the type system catches the failures CI used to detect). Mechanism 8 becomes per-axis drift attribution (Augur reads rows).

## 2. The substrate today — 90% is shipped

This section enumerates what already exists. Every claim cites the file:line. The substrate completion builds on these; refactor surface is §3.

### 2.1 The Graded foundation

Per `algebra/Graded.h:124-410, 448-656, 718-880` (1158 lines total).

`Graded<ModalityKind M, Lattice L, typename T>` is the foundational primitive. Three storage regimes:

- **Primary template** (`Graded.h:124-410`) — separate `inner_` and `grade_` fields, EBO collapses each independently via `[[no_unique_address]]` (`Graded.h:141-142`). Used when value and grade are independent (e.g., `Stale<T>` carrying value + staleness counter).
- **Value-equals-grade specialization** (`Graded.h:448-656`) — when `T == L::element_type`, single field. Used by `Monotonic<T, Cmp>` where the watermark IS the value. `sizeof(Graded<...>) == sizeof(T)`, half the primary's storage cost.
- **Derived-grade specialization** (`Graded.h:718-880`) — when `L::grade_of(T const&)` exists (LatticeDerivesGrade concept, `Graded.h:676`), single field with computed grade. Used by `AppendOnly<T>` where grade derives from container length.

Public API:
- `Graded(value, grade)` explicit construction; `Graded::at_bottom(value)` for bounded lattices.
- `peek() const&`, `consume() &&`, `grade() const` access.
- `peek_mut() &`, `swap()` mutable access (gated by `AbsoluteModality<M> || std::is_empty_v<grade_type>`, `Graded.h:287`).
- `extract() &&` (Comonad counit, gated by `ComonadModality<M>`).
- `inject(value, grade)` (RelativeMonad unit, gated by `RelativeMonadModality<M>`).
- `weaken(new_grade)` (lattice widening, `pre (L::leq(grade_, new_grade))`, `Graded.h:358-381`).
- `compose(other)` (lattice join, `Graded.h:388-409`).

Diagnostics: `modality_name()`, `lattice_name()`, `value_type_name()` (P2996R13 `display_string_of`, with documented TU-context-fragility discipline at `Graded.h:156-186`).

Layout invariant macro `CRUCIBLE_GRADED_LAYOUT_INVARIANT` (`Graded.h:928-948`) verifies `sizeof + alignof + trivial_destructor + trivial_copy` parity per alias. Used by every wrapper to prove zero overhead.

### 3.2 `Row<Es...>` gains a `Lattice`-conforming adapter

`EffectRow.h` adds (~150 LoC) a parallel `EffectRowLattice<Universe>` that satisfies the `Lattice` concept:

```cpp
namespace crucible::effects {

template <typename Universe>
struct EffectRowLattice {
    using element_type = std::bitset<Universe::cardinality>;
    static constexpr element_type bottom() noexcept { return {}; }
    static constexpr element_type top() noexcept { return ~element_type{}; }
    static constexpr bool leq(element_type a, element_type b) noexcept {
        return (a & ~b) == element_type{};  // a ⊆ b
    }
    static constexpr element_type join(element_type a, element_type b) noexcept {
        return a | b;
    }
    static constexpr element_type meet(element_type a, element_type b) noexcept {
        return a & b;
    }

    // Singleton sub-lattice At<Atoms...> mirrors Conf::At, Tolerance::At,
    // QttSemiring::At — used when an op's effect set is fixed at the type level.
    template <typename Universe::atom_t... Atoms>
    struct At {
        using element_type = element_type;
        static constexpr element_type value = bits_of<Atoms...>();
    };
};

// OsUniverse (from Capabilities.h's Effect enum):
struct OsUniverse {
    using atom_t = Effect;
    static constexpr std::size_t cardinality = effect_count;  // = 6
};

// The pre-existing Row<Es...> stays as the ergonomic compile-time-set form;
// it becomes a thin alias resolving through the lattice's At<Atoms...>.
}
```

The pre-existing `Row<Effect::Bg>` continues to work; new code can also use `EffectRowLattice<OsUniverse>::element_type` directly when runtime grades are needed (rare). Birkhoff's representation theorem (`Lattice.h:269`) guarantees the bitset embedding is sound — distributive lattice over a finite set IS a sublattice of the powerset.

### 3.3 KernelCache key extension

Today the cache key is `(content_hash, device_capability)`. The extension is additive:

```cpp
struct KernelCacheKey {
    ContentHash       content_hash;
    DeviceCapability  device_capability;
    RowHash           row_hash;          // NEW — defaults to Row<>::hash() = 0
};
```

Existing entries with `row_hash == 0` continue to work as "empty-row" entries. New entries carry the row of the kernel. Two kernels with same content but different rows are different cache entries. Federation across organizations works because rows are vendor-neutral and content-addressable.

### 3.4 `Cipher::record_event` row gating

Today `Cipher::record_event` (Cipher.h) trusts the caller. template <typename E_os, ::crucible::safety::DetSafeTier dst>
```cpp
auto Cipher::record_event(

        Computation<E_os, Event>> event) noexcept

        row_union_t<E_os, Row<Effect::Alloc, Effect::AppendsReplayLog>>,

```

Equivalently, using the F*-style `Pure<T>` named alias from §4.7:

```cpp
auto Cipher::record_event(Pure<Event> event) noexcept
```

The F* tradition (Project Everest's HACL*, EverCrypt) ships this pattern in production crypto code; the Crucible alias makes the same surface available in C++26 directly.

## 4. The wrapper proliferation — the new alphabet

### 4.1 The wrapper-author template

Codified from `safety/Stale.h` (the canonical model). Every new wrapper follows:

```cpp
template <typename T /*, optional grade parameter */>
class [[nodiscard]] WrapperName {
public:
    // Public type aliases (GradedWrapper concept C2-CHEAT-5 conformance)
    using value_type   = T;
    using lattice_type = SomeLattice;
    using graded_type  = ::crucible::algebra::Graded<
        ::crucible::algebra::ModalityKind::Absolute /* or Comonad / RelativeMonad / Relative */,
        SomeLattice,
        T>;
    static constexpr ::crucible::algebra::ModalityKind modality =
        ::crucible::algebra::ModalityKind::Absolute;

private:
    graded_type impl_;

public:
    // Construction: explicit (value, grade), in_place (std::in_place, grade, args...),
    // factory methods (fresh / at_top / at_bottom / at(value, grade)).
    constexpr WrapperName() noexcept(...) : impl_{T{}, lattice_type::bottom()} {}
    constexpr WrapperName(T value, typename lattice_type::element_type g) noexcept(...);

    // Defaulted copy/move/destroy (or deleted copy for Linear-style).
    constexpr WrapperName(const WrapperName&)            = default;
    constexpr WrapperName(WrapperName&&)                 = default;
    constexpr WrapperName& operator=(const WrapperName&) = default;
    constexpr WrapperName& operator=(WrapperName&&)      = default;
    ~WrapperName()                                       = default;

    // Forwarded diagnostics (CHEAT-3 conformance — must equal substrate's strings)
    [[nodiscard]] static consteval std::string_view value_type_name() noexcept {
        return graded_type::value_type_name();
    }
    [[nodiscard]] static consteval std::string_view lattice_name() noexcept {
        return graded_type::lattice_name();
    }

    // Access (forward to impl_)
    [[nodiscard]] constexpr T const& peek() const& noexcept { return impl_.peek(); }
    [[nodiscard]] constexpr T consume() && noexcept(...) { return std::move(impl_).consume(); }
    [[nodiscard]] constexpr typename lattice_type::element_type
        grade() const noexcept { return impl_.grade(); }

    // Operations (compose, weaken — semantics per modality).

    // For semiring lattices: TWO named methods (lattice join + semiring mul),
    // per Stale<T>'s combine_max + compose_add discipline.

    // Self-test block (mandatory): GradedWrapper concept satisfaction,
    // CRUCIBLE_GRADED_LAYOUT_INVARIANT macro, runtime smoke test.
};

static_assert(::crucible::algebra::GradedWrapper<WrapperName<int>>);
CRUCIBLE_GRADED_LAYOUT_INVARIANT(WrapperName, int);
```

Every new wrapper ships with:

1. The wrapper header (~250 LoC including self-test)
2. The lattice header (~120-200 LoC, often via `ChainLatticeOps<EnumT>`)
3. A Lean module proving lattice axioms (~100-200 lines)
4. Negative-compile witnesses (~50 lines, in `test/safety_neg/`)
5. One production call site adopting it (per-wrapper acceptance test)

Total per wrapper: ~600 LoC + Lean. 25 new wrappers × 600 = ~15K LoC ceiling. Realistic ~10-12K including infrastructure.

### 4.7 Composition rules — wrapper-nesting, not 17-axis ProductLattice

The substrate's actual composition mechanism is **wrapper-nesting**, NOT mega-product-lattice. Each Graded instantiation is one algebraic slice; composition is at the wrapper layer:

```cpp
// A function that needs HotPath + DetSafe + Numerical + Bg context returns:
HotPath<Hot,
    DetSafe<Pure,
        NumericalTier<BITEXACT,
            Computation<Row<Effect::Bg>, ResultTensor>>>>
```

Each layer EBO-collapses if its grade is empty (type-level singleton). The 4-deep nest is `sizeof(ResultTensor) + 4 bytes for wrapper machinery + alignment` in the worst case; zero overhead when grades are singletons.

The dispatcher (FOUND-D) reads the wrapper STACK via reflection — `is_graded_wrapper_v<W>` plus `W::lattice_type` introspection. Per-wrapper concept detection. The dispatcher is generic over the wrapper stack, not specialized per (axis × axis × ... × axis) combination.

This is the algebraic shape that resolves the "23-axis ProductLattice" tension from prior analysis: ProductLattice composes TWO axes that genuinely need to live at the same wrapper layer (Budgeted, EpochVersioned, NumaPlacement, RecipeSpec). Wrapper-nesting composes EVERYTHING ELSE. The substrate is **fibered**, not product-multiplied.

### 4.7 F*-style effect annotations — named compositions

The Crucible substrate ships individual wrappers (DetSafe, HotPath, Wait, etc.); the F* tradition (Microsoft Research, Project Everest's HACL* / Vale / EverParse / EverCrypt — production crypto code shipping in Mozilla Firefox + Linux kernel + Wireguard via KreMLin extraction) ships a NAMED EFFECT LATTICE: `Pure ⊑ Ghost ⊑ Tot ⊑ Div ⊑ ST ⊑ All`. The F* effect names are recognized by every PL-trained engineer; we adopt them as named aliases over the underlying wrappers.

This gives Crucible's substrate three benefits:

1. **Recognizability**. Engineers from F* / Lean / Coq / Idris background read `Pure<int>` and immediately understand "total, no observable effects, terminating". The substrate's per-wrapper composition (`Progress<Terminating, DetSafe<Pure, Computation<Row<>, int>>>`) is technically equivalent but verbose.
2. **A canonical effect-lattice ordering**. F*'s `Pure ⊑ Ghost ⊑ Tot ⊑ Div ⊑ ST ⊑ All` is decades of validated PL design. Adopting it gives Crucible a NATURAL ORDERING on the umbrella effect categories, complementary to the per-axis chain lattices.
3. **Production prior art at scale**. HACL* + EverCrypt ship F*-extracted C in Wireguard (every modern Linux kernel) and Mozilla Firefox's NSS — billions of users. The pattern works in practice.

#### 4.7.1 The aliases — F*-style names over Crucible wrappers

```cpp
namespace crucible::effects {

// Pure: total, no observable effects, terminating.
// = empty effect row + Terminating progress + Pure DetSafe.
// The bottom of F*'s effect lattice.
template <typename T>
using Pure = ::crucible::safety::Progress<
    ::crucible::safety::ProgressClass::Terminating,
    ::crucible::safety::DetSafe<
        ::crucible::safety::DetSafeTier::Pure,
        Computation<Row<>, T>>>;

// Tot: terminating, but OS effects allowed (alloc / IO / block).
// = Terminating progress over arbitrary OS row + Pure DetSafe.
// One step up from Pure: side effects allowed but termination guaranteed.
template <typename E_os, typename T>
using Tot = ::crucible::safety::Progress<
    ::crucible::safety::ProgressClass::Terminating,
    ::crucible::safety::DetSafe<
        ::crucible::safety::DetSafeTier::Pure,
        Computation<E_os, T>>>;

// Div: may diverge.  No termination guarantee but otherwise pure.
// = MayDiverge progress + Pure DetSafe + empty OS row.
template <typename T>
using Div = ::crucible::safety::Progress<
    ::crucible::safety::ProgressClass::MayDiverge,
    ::crucible::safety::DetSafe<
        ::crucible::safety::DetSafeTier::Pure,
        Computation<Row<>, T>>>;

// Ghost: type-level only, computationally erasable.
// = QTT grade-0 (Atkey 2018), the linear semiring's "irrelevant" grade.
// The Lean-side certificate of QTT-grade-0 marks it as erasable for
// downstream Lean proofs; in C++ runtime cost is sizeof(T) but the
// substrate flags it as ghost via reflection (FOUND-D reads it).
template <typename T>
using Ghost = ::crucible::algebra::Graded<
    ::crucible::algebra::ModalityKind::Absolute,
    ::crucible::algebra::lattices::QttSemiring::At<
        ::crucible::algebra::lattices::QttGrade::Zero>,
    T>;

// ST: stateful (heap manipulation allowed).
// = Computation with Alloc effect (the only OS-effect needed for heap mutation).
// Composes with Refined<...> for stronger pre/postconditions over the heap.
template <typename T>
using ST = Computation<Row<Effect::Alloc>, T>;

// All: top of the effect lattice — no constraint.
// The escape hatch.  Equivalent to T itself (no wrapping).
template <typename T>
using All = T;

}  // namespace crucible::effects
```

#### 4.7.2 The F* effect-lattice ordering

```
Pure ⊑ Ghost ⊑ Tot ⊑ Div ⊑ ST ⊑ All
```

A function declared `Pure<T>` satisfies any consumer asking for `Tot<E_os, T>` (Pure is a Tot with empty OS row). Any `Tot` satisfies any `Div` (Tot is a Div that happens to terminate). Any `Div` satisfies `ST` (ST allows divergence + heap). And `All` is the top.

This ordering is enforced via concept-overloaded specialization on the dispatcher (FOUND-D). A `Pure`-pinned call site rejects `ST`-rowed callees at compile time. The `requires` clauses are mechanical:

```cpp
template <typename T> concept IsPure  = /* matches the Pure<...> alias shape */;
template <typename T> concept IsTot   = IsPure<T> || /* matches Tot<...> */;
template <typename T> concept IsDiv   = IsTot<T>  || /* matches Div<...> */;
template <typename T> concept IsST    = IsDiv<T>  || /* matches ST<...> */;
template <typename T> concept IsAll   = true;  // trivial top
```

#### 4.7.3 Why F*-style aliases AND per-axis wrappers (not just one)

The two surfaces are complementary:

- **F*-style aliases** are HORIZONTAL — they group multiple per-axis wrappers under one named composition. Engineers reach for `Pure<T>` when they want "the standard guarantee".
- **Per-axis wrappers** are VERTICAL — they expose individual axes for fine-grained constraints. Engineers reach for `HotPath<Hot, NumericalTier<BITEXACT, T>>` when they need a specific axis combination not captured by F* names.

The per-axis wrappers are the foundation; F*-style aliases are the ergonomic surface. Both ship; users pick the right level for the call site.

#### 4.7.4 Comparison with F* proper

F*'s effect system has features Crucible's substrate does NOT replicate:

- **Effect handlers and lifts** — F* has a `lift` operation between effects (e.g., `Pure t ↪ Tot t`). Crucible's `weaken` operation on Graded provides the same surface for individual axes; cross-alias lifts are the F* concept aliasing chain (IsPure → IsTot → IsDiv → IsST → IsAll).
- **Named effect bindings** — F* has `let x = e in ...` propagating effects. C++26 has automatic-storage-duration variables; the wrapper composition handles propagation.
- **Effect inference** — F* infers effect types from function bodies. Crucible explicitly does NOT (per §15.3 — author-declared rows only).
- **Verification conditions discharged to Z3** — F*'s effect system feeds Z3 with WP-style verification conditions. Crucible has the `verify` preset for residual integer-arithmetic obligations only; effect composition is mechanical at the type level.

The F*-style aliases give Crucible the SURFACE familiarity without claiming the full F* verification framework. The substrate is C++26-native; the aliases are a recognition layer.

#### 4.7.5 Adoption discipline

- New code SHOULD reach for F*-style aliases (`Pure<T>`, `Tot<E, T>`, `Div<T>`, `ST<T>`) where the common cases fit.
- Per-axis wrappers (`HotPath<>`, `DetSafe<>`, `NumericalTier<>`) are the right choice when constraint precision matters more than recognizability.
- `Ghost<T>` is reserved for proof-only / lemma values. Comes with a static_assert verifying QTT grade-0 at construction.
- `All<T>` is the explicit escape hatch — using it documents "no row constraint asserted here".

The dispatcher (FOUND-D) reads BOTH surfaces. F*-style aliases collapse via concept refinement (`IsPure → IsTot → ...`) into the underlying per-axis wrapper introspection.

## 5. The bridge — Row<Es...> meets Lattice

This is the algebraic unification month. ~150 LoC for the bridge, ~50 LoC for the alias preservation.

### 5.1 The gap as it exists today

`effects/EffectRow.h:56-194` ships `Row<Effect... Es>` as a structural template — different `Es...` packs are different types. Operations `row_union_t`, `row_difference_t`, `row_intersection_t`, `Subrow<R1, R2>` are TYPE-LEVEL templates returning types. The `Lattice` concept (`Lattice.h:69-76`) requires VALUE-LEVEL `static constexpr bool leq(element_type, element_type)`.

The mismatch:
- **Row** has type-level operations on type-level effect sets
- **Lattice** has value-level operations on value-level `element_type`

To bridge: define an `EffectRowLattice<Universe>` whose `element_type` is a value-level encoding (bitset) of the row's effect set. Keep `Row<Es...>` as the ergonomic compile-time-set form, but make it a thin alias resolving through the lattice.

### 5.2 The EffectRowLattice<Universe> adapter

```cpp
namespace crucible::effects {

// A "Universe" parameterizes the lattice with a closed catalog of atoms.
// OsUniverse uses Capabilities.h's Effect enum.  Future per-category
// universes (DetSafeUniverse, HotPathUniverse, etc.) follow the same
// pattern but live with their respective lattices.
struct OsUniverse {
    using atom_t = Effect;
    static constexpr std::size_t cardinality = effect_count;  // = 6
};

template <typename Universe>
struct EffectRowLattice {
    using element_type = std::bitset<Universe::cardinality>;

    [[nodiscard]] static constexpr element_type bottom() noexcept {
        return element_type{};
    }
    [[nodiscard]] static constexpr element_type top() noexcept {
        element_type all;
        all.flip();
        return all;
    }
    [[nodiscard]] static constexpr bool leq(element_type a, element_type b) noexcept {
        return (a & ~b).none();  // a ⊆ b
    }
    [[nodiscard]] static constexpr element_type join(element_type a, element_type b) noexcept {
        return a | b;
    }
    [[nodiscard]] static constexpr element_type meet(element_type a, element_type b) noexcept {
        return a & b;
    }
    [[nodiscard]] static consteval std::string_view name() noexcept {
        return "EffectRowLattice";
    }

    // At<Atoms...>: singleton sub-lattice for type-level pinning.  Mirrors
    // ConfLattice::At<Conf>, ToleranceLattice::At<Tolerance>, QttSemiring::
    // At<QttGrade> — empty element_type, EBO-collapses to zero bytes.
    template <typename Universe::atom_t... Atoms>
    struct At {
        using element_type = std::integral_constant<
            std::uint64_t,
            bits_of<Atoms...>()  // consteval bitset → uint64
        >;
        [[nodiscard]] static constexpr element_type bottom() noexcept { return {}; }
        [[nodiscard]] static constexpr element_type top()    noexcept { return {}; }
        [[nodiscard]] static constexpr bool         leq(element_type, element_type) noexcept { return true; }
        [[nodiscard]] static constexpr element_type join(element_type, element_type) noexcept { return {}; }
        [[nodiscard]] static constexpr element_type meet(element_type, element_type) noexcept { return {}; }
    };
};

// Lattice-conformance assertions
static_assert(::crucible::algebra::Lattice<EffectRowLattice<OsUniverse>>);
static_assert(::crucible::algebra::BoundedLattice<EffectRowLattice<OsUniverse>>);
static_assert(::crucible::algebra::verify_distributive_lattice<EffectRowLattice<OsUniverse>>(
    EffectRowLattice<OsUniverse>::bottom(),
    EffectRowLattice<OsUniverse>::top(),
    /* third witness */));

}  // namespace crucible::effects
```

### 5.4 Backward compatibility

The pre-existing `Computation<Row<Effect::Bg>, int>::mk(7)` interface continues to compile unchanged. The class-method form is preserved by making `Computation` a class template that internally holds the `Graded<...>` — OR by making it a true alias and providing the methods as free functions plus a CRUCIBLE_LEGACY-tagged class wrapper for back-compat:

Two viable migration paths:
1. **Alias with façade methods** — `Computation` stays a class template whose body is `Graded<...> impl_;` plus the existing monadic API forwarding. No call-site changes. ~50 LoC of forwarding glue.
2. **True alias plus free functions** — `Computation` becomes a `using` declaration; existing `.mk()` / `.extract()` calls migrate to `mk(comp)` / `extract(std::move(comp))`. Requires sweep of test/effects.cpp + 3 effects_neg/ fixtures. ~50 call-site touches.

Path 1 is recommended for v1 (zero call-site churn). Path 2 is the eventual cleanup once the codebase has internalized the substrate.

### 5.6 The composition: `Computation<...>` nests with other wrappers

The whole point: after the bridge, `Computation` is just one Graded instance. Wrapper-nesting composes:

```cpp
struct KernelCacheKey {
```

L1 (IR002 snapshot, vendor-portable), L2 (IR003* snapshot, intra-vendor), L3 (compiled bytes, per-chip) all use the extended key. Three lookup levels, one consistent extension.

### 8.2 Federation: rows are vendor-neutral

The federation story per FORGE.md §23.2 ("L1 — IR002 snapshot cache (the federation goldmine)"):

> Two Crucible installations running the same model produce identical IR002. Each independently compiles IR003* for its local hardware, but the IR002 lowering work only happens once globally. Meta compiling Llama-8B for H100 enriches Google's L1 cache for the same model on v5p.

- Two installations running the same model with DIFFERENT ROWS get distinct L1 entries (correct — they have different effect requirements, so they want different kernels).
- The federation network's content-addressable storage uses `(content_hash, row_hash)` as the key. Vendor caps come into play only at L2/L3.

The row IS vendor-neutral by construction — every wrapper in §4 is defined without reference to specific silicon. Federation works because rows match across machines.

### 8.5 Row hash composition — how nested wrapper stacks become a 64-bit key

The cache key needs `row_hash` to be:
- **Stable across processes** (same wrapper stack → same hash, regardless of compiler version, linker order, TU inclusion)
- **Stable across machines** (federation: Org A and Org B produce the same hash for the same wrapper stack)
- **Composable** (the hash is computed from the wrapper stack's structure, not from a flat enumeration)
- **Cheap to compute** (every cache lookup pays this cost)

The composition rule:

```cpp
namespace crucible::reflect {

// For a wrapper stack like Stale<Tagged<Refined<positive,
//                                            Computation<Row<Bg>, T>,
//                                            source::FromUser>>>,
// the row_hash is a fold over the per-wrapper hash contributions:
//
//   row_hash(Stack) = fmix64(
//       per_wrapper_hash(outermost_wrapper),
//       row_hash(inner_stack))
//
// per_wrapper_hash(W) = fmix64(
//     stable_name_of<W::lattice_type>,    // canonical lattice name
//     stable_grade_hash<W>)                // grade value (consteval-collapsed
//                                          // for type-level singletons)
//
// Recursion bottoms out at the wrapped value's stable_type_id.
//
// stable_name_of<L> uses canonicalize_pack<Ts...> to sort template
// argument packs alphabetically — so Tagged<T, A> + Tagged<T, B> and
// Tagged<T, B> + Tagged<T, A> produce the same hash even if instantiation
// order differs across TUs.

template <typename WrapperStack>
constexpr std::uint64_t row_hash = /* recursive fmix64 fold */;

}  // namespace crucible::reflect
```

#### 8.5.1 Per-axis hashes are stable

Each lattice's `stable_name_of<L>` is canonical: it canonicalizes template argument packs (FOUND-E10), strips compiler-version-specific qualifications (the TU-context-fragility documented at `algebra/Graded.h:156-186` is handled via the `.ends_with()` discipline), and sorts type-level packs by lexicographic order on the canonicalized names.

`stable_grade_hash<W>`:
- For type-level singleton grades (e.g., `ToleranceLattice::At<BITEXACT>`): the grade is encoded in the type, so `stable_name_of<W>` includes it; no separate grade hash needed.
- For runtime-valued grades (e.g., `Stale<T>` carrying a runtime staleness counter): hashes the grade value bytes directly. The cache key is computed at write time when the grade value is known.

#### 8.5.2 Wrapper-nesting commutativity

Two wrappers compose via nesting: `Stale<Tagged<T>>` vs `Tagged<Stale<T>>` — are these the same row_hash?

**No, by design.** The wrapper-nesting order encodes evaluation order: `Stale<Tagged<T>>` is "a Tagged<T> value carrying a staleness grade" while `Tagged<Stale<T>>` is "a Stale<T> value carrying a Trust provenance tag". These have DIFFERENT semantics and SHOULD have different cache keys.

The CANONICAL ORDER for composition (defined in §4.7 F*-style discipline):

```
HotPath ⊃ DetSafe ⊃ NumericalTier ⊃ Vendor ⊃ ResidencyHeat ⊃
  CipherTier ⊃ AllocClass ⊃ Wait ⊃ MemOrder ⊃ Progress ⊃
  Stale ⊃ Tagged ⊃ Refined ⊃ Secret ⊃ Linear ⊃ Computation
```

(Outer wrappers carry "higher-level" properties; inner wrappers are "closer to the value".)

Wrapper authors are EXPECTED to construct stacks in this order. Out-of-order stacks compile fine but produce DIFFERENT row_hashes. A linter (FOUND-E enforcement) emits a warning when a stack deviates from canonical order.

The canonical order is documented in CLAUDE.md (after the substrate completion lands). The hash composition is deterministic given canonical order; deviation is the user's choice and signals "I want this specific composition order to be cached separately".

#### 8.5.3 Cache invalidation on Universe extension

What happens when we add atom #N+1 to a Universe (e.g., `Effect::Refute` joins OsUniverse)?

- The Universe's cardinality grows from N to N+1.
- The bitset's encoding changes.
- All `row_hash` values that depended on bit positions in the OS row CHANGE.

This invalidates the cache for OS-row-keyed entries. To avoid this:

- Atoms are appended-only. New atoms get NEW bit positions (N, N+1, ...). Existing atoms keep their positions.
- The hash function is computed over the canonical bitset — `Row<Bg>` always hashes to the same value because `Effect::Bg` always has the same bit position.
- Adding `Effect::Refute` at position N+1 does NOT affect the hash of `Row<Bg>` (still at position 3) — only NEW rows mentioning Refute get new hashes.

So cache invalidation on Universe extension is bounded: only rows that mention the new atom get new hashes. Existing entries continue to work.

This is the standard content-addressing discipline — APPEND-ONLY UNIVERSES. Per §14.2 open question, removing or renumbering atoms is a major-version event for that Universe; appending is additive.

### 8.6 Worked example: federation across organizations

Two organizations training Llama-70B:

- Org A (Meta) compiles for H100 + Trainium fleet. Emits kernels with `Row<Effect::Bg, NumericalTier::BITEXACT_TC, Vendor::NV, ...>`.
- Org B (Google) compiles for TPU v5p fleet. Emits kernels with `Row<Effect::Bg, NumericalTier::BITEXACT_TC, Vendor::TPU, ...>`.

The L1 cache federates content-addressed entries. For every kernel where the recipe and structural attrs match (identical IR002 hash) AND the row matches (identical row_hash), the L1 entry is shared. Org B's TPU compile pulls Org A's L1 entry, then independently compiles to IR003*-TPU.

The compile time on Org B for shared kernels: ~1μs (cache hit lookup + load) instead of ~100ms (cold compile through Forge phases A-E). At scale, this is hours of compile time saved per fleet.

Why this works: rows are content-addressable by definition. Same row → same hash → same key. Vendor caps come into play only at L2/L3 (per-vendor IR003* + per-chip bytes).

## 12. Risks and mitigations

### 12.1 Compile-time burden — R1

19+ wrappers nestable per signature, even with EBO, generates large template instantiation graphs. Realistic risk of cc1plus blowing past 4GB RSS on the heaviest TUs.

**Mitigation**: bench in Month 2 with 8 wrappers stacked. Use `test/test_wrapper_compile_burden.cpp` as the regression test. If RSS exceeds 3 GB on the heaviest TU, factor out per-category lattices into separate header modules to amortize. The existing `-j8` build cap (per `feedback_build_parallelism` user memory) provides headroom.

### 12.2 Diagnostic blast radius — R2

19-wrapper failures generate huge backtraces without FOUND-E. Today (with 11 wrappers + Met(X)) failures already produce 200-line errors; 25+ wrappers without FOUND-E would push to 2000+ lines.

**Mitigation**: FOUND-E lands in Month 3, BEFORE further wrapper proliferation in Month 4. Decision D6.

### 12.3 Migration cost — R3

~50 hot-path call sites get row-typed signatures in Month 4. Each is mechanical but requires careful review (every parameter type changes; every `Subrow` check needs validation).

**Mitigation**: per-call-site PRs, not bulk migration. Each PR ships with a per-axis acceptance test. The `test_migration_verification` harness tracks adoption progress.

### 12.4 The deliberately-not-graded boundary — R4

9 wrappers + sessions stay structural. Some of them carry effect-shaped properties (ConstantTime carries timing-shape; Pinned carries address-stability). The substrate proposal does NOT migrate them.

**Mitigation**: documented in CLAUDE.md L0 + this doc D8. Bridges in `bridges/` (existing pattern from MachineSessionBridge, CrashTransport, RecordingSessionHandle) are the composition mechanism.

### 12.5 Dispatcher consumption lag — R5

Months 1-3 ship rows that nobody reads yet. Risk of stalled momentum if FOUND-D slips.

**Mitigation**: parallel-track FOUND-D from Month 1. Reading surface ships in Month 3. Full consumption in Month 6. The reading surface is independent of wrapper proliferation; dispatcher implementation can lag behind wrappers by one month without breaking anything.

### 12.6 Path-dependency lock-in — R6

23 lattice categories committed in v1. Future refinement requires either backward-compat shim or breakage.

**Mitigation**: stay one design level more abstract than the catalog — concept-driven access patterns (`HotPathSafe<callee>` concept rather than `Subrow<E_in, hot_path_only_grade>` direct check). Per-axis `PowerSubrow` vs `ChainOver` syntax to keep the propagation direction explicit (T2 from prior analysis). Powerset axes are extensible by adding atoms; chain axes are NOT extensible without breakage — bias toward powerset where possible.

### 12.7 Algebraic hazards — R7

Some categories don't fit cleanly:
- ReliabilityEffect (M_3 diamond, non-distributive) — DROPPED from v1; recover via PhiEffect's `safe`/`live+` chains
- FairnessEffect — semantic carrier, not lattice — captured via PhiEffect refinement
- AssociationEffect — composes orthogonally with sessions, not a row axis — stays in session typestate

**Mitigation**: per-category algebraic classification (§4.8) precisely identifies which categories are Lattice / Semiring / Chain / Product / OUT-OF-SCOPE.

### 12.8 The Cipher write fence's adoption cost — R8

Adding `requires Subrow<event_row, Row<DetSafe::PhiloxRng>>` to `Cipher::record_event` will break every existing call site that records non-Philox-rowed events. Audit cost.

**Mitigation**: Month 4 includes a "DetSafe call site audit" task — every Cipher::record_event caller is reviewed and either (a) migrated to be DetSafe-pure, (b) explicitly marked NON-DETERMINISTIC and routed to a different log (telemetry rather than replay log), or (c) flagged as DetSafe bug to be fixed. Estimated ~20 call sites to audit. Per-call PR; CI gates on the new requires.

## 14. Open questions

### 14.1 Linearity in ProductLattice

`ProductLattice<L1, L2>` requires every `L_i` to satisfy `Lattice<L_i>` (`ProductLattice.h:131`). Linear permissions ride QttSemiring (which is a `Semiring`, not a `Lattice` per `Lattice.h:96-111`). The substrate distinguishes Lattice from Semiring deliberately.

**Resolution path**: PermissionEffect is NOT a row axis (Decision D5). Linear<T> stays as a separate Graded instance with QttSemiring; it composes via wrapper-nesting, not via ProductLattice. No bridge needed.

### 14.2 Universe extensibility

Each `EffectRowLattice<Universe>` has a closed-cardinality `bitset<N>` element type. Adding atoms to a Universe (e.g., adding `Effect::Refute` to OsUniverse) breaks the cardinality.

**Resolution path**: per-category Universes are STABLE catalogs. New atoms are additive only; cardinality grows. Hash stability is preserved because the bit position is determined by enum value, not by enumeration order. Adding atoms to a Universe is a major version bump for that category's Lattice but DOES NOT break wrappers in other categories.

### 14.3 Per-process row registry

For the dispatcher to introspect rows at runtime (not just compile time), it needs a runtime registry mapping atom enum values → display names. Each Universe ships its own.

**Resolution path**: `Capabilities.h` already has `effect_name(Effect)` consteval string emitter. Each new Universe ships an analogous emitter. No central registry needed; reflection-driven per-Universe.

### 14.4 Cross-process row hashing

For the federation cache, two processes must agree on the same `row_hash` for the same row. P2996 reflection's `display_string_of` is TU-context-fragile (per `algebra/Graded.h:156-186`).

**Resolution path**: `stable_function_id<FnPtr>` uses canonicalized type names + sorted template argument packs (per FOUND-E10). The hash function is FNV-1a / fmix64 — deterministic across processes. The federation protocol spec (Phase 7+) will pin the canonical form.

### 14.5 Refinement implications across categories

Today `safety/Refined.h` has `implies_v<P, Q>` for cross-predicate refinement (positive ⇒ non_negative). The dispatcher could use this to safely narrow a function's input refinement when composing two stages. Open: does the dispatcher apply implication transitively?

**Resolution path**: ship implications as direct (one hop) for v1. Add transitive closure later if a use case emerges. Per 27_04 §9.2 recommendation.

### 14.6 Async subtyping for session-typed rows

Session types ship sync subtyping (Gay-Hole 2005); precise async subtyping (GPPSY23) is unshipped (Task #348). When session-typed rows compose with effect rows, the session subtyping affects what's admissible.

**Resolution path**: defer until L6.5 SISO async subtyping (Task #348) ships. The substrate completion does not depend on this.

### 14.7 Bridges for the remaining 6 deliberately-not-graded wrappers

`bridges/` ships 3 bridges today: `MachineSessionBridge.h`, `CrashTransport.h`, `RecordingSessionHandle.h`. The other 6 deliberately-not-graded wrappers (ScopedView, OwnedRegion, Pinned, Checked, ConstantTime, NotInherited, Simd, Workload — minus Machine which has its session bridge) compose with Graded wrappers via ad-hoc patterns. After the substrate completion, ~3-5 may want explicit `bridges/<wrapper>Bridge.h` headers to formalize the composition.

Specifically:

- **`OwnedRegion<T, Tag>` bridge** — when an OwnedRegion contains Graded values (e.g., `OwnedRegion<HotPath<Hot, T>, Tag>`), the bridge documents how the row propagates through `parallel_for_views<N>` body invocations.
- **`Pinned<T>` bridge** — Pinned address-stability composes with Graded via "the pinned address IS the row's identity". The bridge specifies the pin discipline.
- **`ConstantTime<T>` bridge** — CT timing-shape composes with Graded; specifically the `Secret<T>` ↔ `ConstantTime<T>` interaction is delicate (CT operations on secret values must not branch on secret bits). The bridge ships the interaction rules.

**Resolution path**: track these as `bridges/` work in Phase 7+. Per-bridge ~200-400 LoC. Three bridges = ~1000 LoC. The substrate completion does NOT block on these; they ship as the wrappers gain composition partners.

### 14.8 Quotient combinators

The 27_04 wiring plan §17 lists `ContentAddressed<T>` quotient combinator (SEPLOG-L1, Task #361). When two protocols deliver the same payload up to content hash, they're equivalent up to quotient.

**Resolution path**: `ContentAddressed<T>` becomes a Graded wrapper in v2. Lattice TBD; likely `Graded<Absolute, ContentHashLattice, T>` with derived-grade (regime-3). Estimate 1 month after v1 ships.

## 15. What we explicitly do NOT do

Documenting non-decisions is as important as documenting decisions. These prevent future iterations from re-litigating settled choices.

### 15.1 No effect handlers, no JIT, no runtime dispatch

The substrate is plain C++26 templates + reflection + concepts + contracts. No clang plugin, no embedded DSL, no JIT compilation step, no runtime effect-handler dispatch.

Crucible's effect rows are zero-cost phantom types. The §3 Met(X) extension preserves zero-cost. There is nothing for a JIT to specialize. Compare to the `Tracing JIT for effect handlers` (Gaißert-Bolz-Tereick-Brachthäuser OOPSLA 2025) — that work targets RPython-based meta-tracing for Eff / Effekt / Koka effect handlers; Crucible's `Effect` enum atoms are zero-byte tags, not handlers needing runtime dispatch. The §3 extension preserves this.

### 15.2 No flow-sensitive rows

Crucible's checker is point-in-time: at construction the predicate holds; the type system does not reason about flow. Per `THREADING.md L772` documentation.

The substrate completion does NOT add flow-sensitive row analysis. A function whose row is `Row<Pure>` is not refined to `Row<>` mid-body even if no impure operation occurs. The row reflects the WORST case of the body.

This is the conservative design choice. Flow-sensitive refinement requires either (a) full dataflow analysis at compile time (expensive, language-design territory) or (b) external SMT discharge via Z3 (covered by the existing `verify` preset for residual integer-arithmetic obligations, NOT for row refinement).

### 15.3 No row inference (rows are author-declared)

Function authors DECLARE their function's row in the signature. The compiler does NOT infer rows from function bodies. This is the same discipline as the existing `effects::Bg` / `effects::Init` / `effects::Test` cap-tag form — author-declared, never inferred.

Inferring effect rows from C++26 function bodies would require parsing the body's AST + classifying every operation by effect — basically a static analyzer the size of clang-tidy. Out of scope.

### 15.4 No cross-language row interop

Rows are a C++26 substrate. The Vessel adapters (PyTorch, JAX, native frontend) cross language boundaries. Rows do NOT propagate through those boundaries.

Future work (Phase 7+): a Vessel-side row declaration mechanism (e.g., Python decorator `@cr.declare_row(DetSafe::Pure)`) for user code. v1 substrate completion does not include this.

### 15.5 No abstract cost dimensions

Per 27_04 §1.2 explicit commitment: "no `Cost<bytes, ns_per_item>`, no `[[crucible::compute_ns(N)]]`, no `LatencyBudget`, no FLOP-intensity targets. Cost is modeled via concrete hardware facts only — bytes, cache sizes, NUMA distances."

The substrate completion preserves this. Wrappers like `Budgeted<{BitsBudget, PeakBytes}, T>` carry CONCRETE byte / cache budgets. Wrappers do NOT carry "expected nanoseconds" or "expected FLOPs/byte" as type-level dimensions. Honest per CLAUDE.md §VIII.

### 15.6 No DSL on top of Met(X)

Per 27_04 §8.1: "The foundation is plain C++26. Users write regular C++26 functions. Parameter types are the spec. There is no embedded DSL the user must learn — no `[[crucible::parallel]]` attribute syntax to memorize, no `pipe(...) | map(...) | filter(...)` combinator chain, no `make_computation<Spec...>(body)` spec pack."

The substrate completion preserves this. New wrappers compose via standard C++26 type construction. The user-facing API question stays deliberately deferred (per 27_04 §0.2). The substrate ENABLES any of the four prior dispatcher candidates (signature-as-spec, type-spec, dataflow combinators, annotated-imperative); choice deferred.

### 15.7 No optimistic API surface

Every substrate API is conservative: requires explicit type-level commitments (Permission tags, OwnedRegion lifetimes, Refined predicates, Effect rows, the new wrappers). Refuses to dispatch when commitments aren't in place. The dispatcher emits a structured diagnostic (FOUND-E), not a default-best-effort. Users who want default-best-effort fall through to manual orchestration via the existing direct primitives.

### 15.8 No global mutable state in the dispatcher

The dispatcher does NOT register call sites with a global scheduler, does NOT maintain a global thread pool that must be initialized at process startup, does NOT have hidden lifetime dependencies on a singleton. Each dispatch is a self-contained computation; the underlying primitives (jthreads, Permission roots, OwnedRegions, GradedWrappers) are constructed and destroyed within the call's lifetime.

Per 27_04 §8.7 explicit commitment.

### 15.9 No migration of the deliberately-not-graded 9 wrappers

Per Decision D8 and `safety/Safety.h:42-86` policy block. The 9 wrappers (Machine, ScopedView, OwnedRegion, Pinned, Checked, ConstantTime, NotInherited, Simd, Workload) STAY structural. They compose with Graded wrappers via documented bridges (already shipped in `bridges/` for some). The substrate completion does NOT migrate them.

This is a load-bearing boundary. Future iterations should NOT propose migrating these — the SESSIONS-AUDIT decision is documented and binding.

### 15.10 No PermissionEffect as a row axis — the algebraic argument

Per Decision D5 and 27_04_csl_permission_session_wiring §2.5. CSL permissions stay parallel substrate. Sessions stay structural typestate. The three substrates (Graded values, CSL permissions, sessions) compose orthogonally. The substrate completion does NOT collapse them into one ProductLattice.

The DEEPEST reason is algebraic, not just architectural. The `Lattice` concept (`algebra/Lattice.h:69-76`) requires `join` to be IDEMPOTENT (`a ∨ a = a`), per the verifier `verify_idempotent_join` (`Lattice.h:154-156`). Linear permissions explicitly forbid this:

```cpp
// What a "PermissionLattice" would have to satisfy:
auto p1 = Permission<MyTag>{};       // OK — minted once
auto p2 = Permission<MyTag>{};       // FORBIDDEN — linearity violation
auto joined = join(p1, p2);          // can't even SPELL the operands
//                                   // (p1 was already moved-from)
```

The lattice axiom `join(a, a) = a` requires `a` to be present TWICE — but Permission<Tag> deletes copy and the move-from-self pattern is undefined. So idempotence isn't just untrue for permissions; it's UNSPELLABLE. Permissions live in QttSemiring (`Lattice.h:96-111`) — a Semiring whose `mul` is NOT idempotent (grade-1 × grade-1 = grade-2, not grade-1). The substrate's deliberate disjunction of `Lattice` and `Semiring` concepts (separate definitions at `Lattice.h:69-76` vs `Lattice.h:96-111`) reflects this fundamental difference.

PermissionEffect appearing in prior brainstorms was a category error. The right model is `PermissionedSessionHandle<Proto, PermSet, Resource>` (FOUND-C v1, shipped) — three substrates threaded through one handle, not one substrate carrying three axes. The substrate authors saw this two months ago when they shipped QttSemiring as a Semiring separate from the Lattice family.

### 15.11 No ReliabilityEffect as a row axis

Per the prior algebraic analysis. `{AtMostOnce, AtLeastOnce, ExactlyOnce}` form Birkhoff's M_3 diamond (non-distributive). Adding it as a row category would break `verify_distributive_lattice` for the affected axis.

**Resolution**: recover at-most-once / at-least-once via PhiEffect's `safe`/`live+` chains (which ARE distributive). Drop ReliabilityEffect from the v1 catalog.
