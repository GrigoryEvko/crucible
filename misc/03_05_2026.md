## §0. TL;DR

The Crucible substrate is **far more shipped than consumed**. ~14000 lines of session-types theory, 28 algebra lattices, ~80 safety wrappers, full Met(X) row machinery, FOUND-C v1 PermissionedSessionHandle, 5 mint factories, 18-cheat probe — all green. **The Fixy prerequisite verdict: Phase 0 is 5/6 closed.** P0-1 (Fn.h), P0-3 (DimensionTraits.h), P0-4 (Reflectable.h → Reflected.h), P0-5 (examples/fn/*), P0-6 (HS14 fixtures) all shipped. **P0-2 (CollisionCatalog.h, #1096) is the lone unshipped Phase 0 deliverable** — the spec is fully written in fixy.md §24.2 (12 rules). ## §1. Methodology

10 elite-bug-hunter agents launched in parallel on 03 May 2026, each with a non-overlapping integration-audit mandate spanning a distinct slice of the codebase. Each agent received:

- A self-contained ~350-word brief (smart-colleague style, no shared conversation context).
- A specific file/path roster to investigate.
- 5-7 investigation hints listing concrete questions to answer.
- A strict report format: top 5 findings with severity/files/missing/enabler/effort + 2-sentence strategic takeaway.
- A read-only injunction (no modifications).
- Output budget: under 700 words per report.

The 10 agent mandates partitioned the integration-audit space along these axes:

| # | Agent | Domain |
|---|---|---|
| 1 | Five-axis session theory bonding | The 5 orthogonal session-type concepts (Linear binary, MPST, Subtyping, Crash-stop, Higher-order) and their cross-bonding |
| 2 | Permission × Session integration | CSL permissions × session typing — every Permissioned* primitive, every Session* combinator |
| 3 | Lattice × Session × Wrapper | 28 algebra lattices × 24 session headers × ~80 wrappers — composition gaps |
| 4 | Met(X) Row × Sessions × Permissions × Concurrent | Effect-row propagation across the substrate's call surfaces |
| 5 | Concurrent × cache-tier × workload | The runtime concurrency story — BPF telemetry, AdaptiveScheduler, cache-tier rule |
| 6 | Bridge layer audit | bridges/ tree (5 headers) plus cross-axis bridges that DON'T exist |
| 7 | Cipher × Sessions × CipherTier × Federation | Cipher persistence integration with the discipline primitives |
| 8 | Algebra × wrapper-nesting × Graded | Substrate-internal — Graded adoption, wrapper-nesting, row_hash, Is*.h coverage |
| 9 | Perf BPF × telemetry × Augur × visualization | Observability stack — what feeds where, and what's orphaned |
| 10 | Production substrate adoption gap | Vigil, BackgroundThread, TraceRing, MetaLog, KernelCache, Cipher, vessel_api — substrate consumption rate |

Every agent had access to all tools (Read, Bash, grep, etc.). Agents did not communicate with each other; some findings overlap between agents and that overlap is meaningful — the same primitive named as missing by multiple independent agents is a stronger signal than a single citation.

Total artifacts examined across all 10 agents: ~275 header files, 14755 lines of session/* code alone, ~5000 lines of bench/test infrastructure, the full task list (350+ entries), 9 BPF programs, 5 visualization headers, 8 diagnostic headers.

Total agent runtime: ~30 minutes wall clock (parallel), ~3 hours of agent-time across the 10 agents.

Verdict format: each finding is tagged **CRITICAL / LOAD-BEARING / HIGH / MEDIUM / LOW**. CRITICAL means a real bug or a structural soundness gap that already affects production correctness. LOAD-BEARING means the absence blocks a cascade of downstream work. HIGH/MEDIUM/LOW are forward-looking opportunity tiers.

## §2. The Fixy frame

fixy.md (2190 lines, 24 sections) is the strategic doc that defines what Crucible's substrate must support to compile Fixy programs. The 50 agent findings make sense ONLY when read against that frame — without it, "missing SwmrSession" is just a backlog ticket; with it, it's a Phase 0 substrate prerequisite for the entire downstream pipeline.

### §2.1 What Fixy is

Fixy is a **typed compute DSL embedded in C++26**, NOT a separate language with its own compiler infrastructure. Three surfaces lower to the same substrate:

1. **Python decorators** (researcher path) — `@cr.kernel.attention(...)`, AST-reflected to IR000 directly, never goes through `fixyc`.
2. **C++26 Fn templates** (engineer path) — direct `mint_fn<...>(ctx, body)` calls. The substrate IS the surface.
3. **`.fixy` source files** (purist path) — `fixyc` transpiles to C++26.

The transpiler is structurally lightweight: ~8.6K LoC for parser + elaborator + codegen + diagnostic translator. The C++26 substrate it emits to (safety/, effects/, permissions/, sessions/, algebra/, concurrent/, handles/, bridges/) is what we have today plus what's missing.

### §2.2 The radical architectural split

fixy.md §3 mandates a 70/30 split:

| Stays in `crucible/` (~30%) | Moves to `fixy/` (~70%) |
|---|---|
| `safety/` — Linear, Refined, Tagged, Secret, NumericalTier, DetSafe, ... | `fixy/ir/` — Expr, ExprPool, Graph, MerkleDag, SymbolTable, Ops, CKernel, NumericalRecipe, RecipePool, RecipeRegistry, TensorMeta, TypesML |
| `effects/` — Row, Computation, Capability, ExecCtx | `fixy/runtime/` — TraceRing, MetaLog, BackgroundThread, IterationDetector, TraceGraph, TraceLoader, Lower, CrucibleContext, ReplayEngine, Transaction, RegionCache, Serialize, SchemaTable, Arena, PoolAllocator, Vigil, DimHash, Philox, CallSiteTable, Saturate, StorageNbytes |
| `permissions/` — Permission, SharedPermission, PermissionFork, ReadView, FairSharedPermissionPool | `fixy/forge/PhaseA-PhaseL/` — 12-phase IR optimization pipeline |
| `sessions/` — Session, SessionContext, SessionGlobal, SessionAssoc, SessionCrash, SessionSubtype, ... | `fixy/mimic/{nv,am,apl,tpu,trn,cpu}/` — per-vendor backends |
| `handles/` — FileHandle, OneShotFlag, PublishOnce, LazyEstablishedChannel, Once | `fixy/frontends/{pytorch,jax,native}/` — adapter trees |
| `algebra/` — Graded, Lattice, Modality + 28 lattices | `fixy/stdlib/` — Std/{Tensor,Kernel,Vigil,Recipe,Collective,Data,HPC,Sim,Math,Logic,Cluster}.fixy |
| `concurrent/` — Topology, AdaptiveScheduler, queues, Permissioned* primitives | |
| `bridges/` — adapter primitives | |
| `cipher/` — content-addressed persistence | |
| `canopy/`, `cntp/`, `keeper/`, `augur/`, `operator/`, `rt/` — distributed substrate | |

After the split, **Crucible knows nothing about kernels, recipes, tensors** — those become Fixy concepts. Crucible's responsibility shrinks to "run any `.fixy-pkg` artifact across a fleet of Keepers, route data through Cipher and CNTP, recover from failures, monitor drift via Augur." Every agent finding either (a) belongs to substrate that stays in `crucible/`, or (b) belongs to code that moves to `fixy/` and must work correctly there.

### §2.3 The 20-dim × 5-Tier × 12-collision spec

fixy.md §24.1 specifies a **20-dimension grade vector** (down from FX's 22; drops Clock Domain and FP Order). Every Fixy binding carries this vector; the C++26 type system + concept gates discharge it.

| Tier | Composition law | Dimensions | Substrate header |
|---|---|---|---|
| F | Foundational (concept gates) | Type (1), Refinement (2) | safety/Refined.h, safety/Fn.h |
| S | Commutative semiring (par=+, seq=*, 0 annihilator) | 15 dims: Usage (3), Effect (4), Security (5), Lifetime (7), Provenance (8), Trust (9), Observability (11), Complexity (13), Precision (14), Space (15), Overflow (16), Mutation (18), Reentrancy (19), Size (20), Staleness (22) | safety/{Linear,Tagged,Secret,Refined,Borrowed,...}.h |
| L | Lattice with validity (par=join, seq=meet) | Representation (10) | safety/{Bits,...}.h |
| T | Typestate (transitions; no par/seq) | Protocol (6) | sessions/Session.h + family |
| V | Versioned (compatibility check at each site) | Version (21) | safety/EpochVersioned.h |

**12 collision rules** (fixy.md §24.2) catalog the dimension compositions the kernel rejects on soundness grounds. These are the next deliverable: P0-2 #1096 `safety/CollisionCatalog.h`. The 12 codes:

| Code | Composition | Reason |
|---|---|---|
| I002 | classified × Fail(E) | error payload may carry secret; catch arm leaks classification |
| L002 | borrow × Async | borrow lifetime cannot bridge await suspension |
| E044 | CT × Async | async scheduling defeats constant-time guarantee |
| I003 | CT × Fail on secret | fail(e) on secret-dependent condition leaks branch via control flow |
| M012 | monotonic × concurrent | concurrent monotonic update without atomic is racy |
| P002 | ghost × runtime | ghost values are erased at codegen; no runtime representation |
| I004 | classified × async × session | classified value over async session may leak via timing |
| N002 | decimal × overflow(wrap) | wrap is meaningless for exact decimal types |
| L003 | borrow × spawn (unscoped) | unscoped spawn may outlive captured borrows |
| M011 | linear × Fail (cleanup) | linear value live at fail site without cleanup leaks |
| S010 | Staleness × CT | runtime freshness check defeats constant-time guarantee |
| S011 | Capability × Replay | ephemeral capability does not survive replay |

Wall-clock dimensions (Latency, Energy, Power, WallClock, BitsTransferred) are **deliberately excluded** — the compiler cannot prove physical bounds; annotation-only dimensions create false guarantees. SCHED_DEADLINE / realtime priority remain as OS-level scheduling-class hints, NOT type-system claims.

### §2.5 The strategic verdict

The substrate is more capable than its consumers know how to use. Every Phase 0 deliverable except CollisionCatalog has shipped, but production code (Vigil, BackgroundThread, Cipher, KernelCache) hasn't been refactored to consume the discipline primitives. ### §3.1 Agent 1 — Five-axis session theory bonding audit

**Mandate.** Crucible's substrate bonds together 5 orthogonal session-type theory concepts: (1) Linear binary sessions (Honda 1998), (2) Multiparty session types / MPST (Honda-Yoshida-Carbone 2008, Scalas-Yoshida 2019, Hou-Yoshida-Kuhn 2024), (3) Synchronous subtyping (Gay-Hole 2005), (4) Crash-stop semantics (BSYZ22, BHYZ23), (5) Higher-order delegation (Honda 1998 throw/catch). Plus side-cars: payload subsort, content-addressed quotient, checkpoint/rollback, CT crypto, declassify, event log, permission flow, patterns. Agent identified where the 5 theory concepts FAIL TO BOND.

**Investigation surface.** All 24 sessions/*.h headers (14755 lines), test/test_session_*.cpp, cross-references via grep.

#### Finding 1.1 — SessionDelegate × SessionCrash gap
- **Severity:** LOAD-BEARING
- **Concepts:** Higher-order delegation × Crash-stop
- **Where:** `sessions/SessionDelegate.h:135-140` (`all_offers_have_crash_branch<Delegate<T,K>>` recurses K only)
- **What's missing:** When carrier hands off `Delegate<T, K>`, the framework explicitly skips T's crash-safety check ("recipient's problem"). But there is NO `delegated_crash_propagation<T, RecipientTag, CarrierK>` metafunction certifying that if T's recipient crashes, K can recover. There is also no rule for `Delegate<Stop, K>` (delegating an already-crashed channel) — `compose_t<Delegate<Stop, K>, Q>` is unspecified. Honda 1998's throw/catch semantics required the carrier to learn of recipient failure; BHYZ23 did not extend to higher-order delegation.
- **Test evidence:** `test_session_delegate.cpp` has 0 references to Crash/Stop.
- **Enabler shipped?** Yes — Stop combinator + `all_offers_have_crash_branch` exist; can be specialized.
- **Effort:** medium.

#### Finding 1.2 — SessionEventLog × {Crash, Checkpoint, Delegate} incomplete enum
- **Severity:** LOAD-BEARING
- **Concepts:** Event-log × Crash-stop × Higher-order × Checkpoint
- **Where:** `sessions/SessionEventLog.h:135-149` (`SessionOp` enum is {Send, Recv, Select, Offer, Close})
- **What's missing:** SessionOp enumerates only the 5 L1 combinators. It cannot record `Stop` (peer crash), `Checkpoint::base/rollback`, `Delegate`/`Accept`, or branch-choice protocol changes. Bit-exact replay of a session that crashed or rolled-back is therefore impossible — the log is silent on the most important transitions. `SessionCheckpoint.h` has 0 cross-references to `SessionEventLog.h` despite both being typed.
- **Test evidence:** `test_session_event_log.cpp` exercises only Send/Recv.
- **Enabler shipped?** Yes — enum is open; Stop, CheckpointedSession::base()/rollback() are defined.
- **Effort:** small (extend enum + dispatch in `RecordingSessionHandle`).

#### Finding 1.3 — SessionGlobal × SessionCrash projection bug **(REAL BUG)**
- **Severity:** HIGH (correctness bug, not just gap)
- **Concepts:** MPST × Crash-stop
- **Where:** `sessions/SessionGlobal.h:191-198` (`StopG<Peer> ↾ R = End` for R≠Peer)
- **What's missing:** Projection of `StopG<Peer>` produces `End` for every non-Peer role. This is wrong when other roles have pending sends to Peer — they should project to `Stop` (or to a crash-handling Offer). The rule disagrees with `all_offers_have_crash_branch_v` enforcement: a global `Choice<R, Peer, ...>` followed by `StopG<Peer>` projects R to a clean `End` instead of forcing R into a mandatory-crash branch. Round-trip Γ → G → projected → Γ' ⊑_s G therefore loses crash-safety information.
- **Test evidence:** No test in `test_session_global.cpp` enforces "non-Peer projection of StopG must be Stop or contain a crash-handling Offer."
- **Enabler shipped?** Yes — StopG, Stop, projection metafunction.
- **Effort:** medium (refine `project_case_v` for stop nodes; add reachability check).

#### Finding 1.4 — `is_permission_balanced_v<Γ, InitialPerms>` is GHOST API **(REAL BUG)**
- **Severity:** HIGH (documented surface that does not exist)
- **Concepts:** CSL × MPST (the L9 ⊥ L2 axis)
- **Where:** `sessions/SessionContext.h:20`, `sessions/PermissionedSession.h:40,114,495`
- **What's missing:** Four header doc-blocks state `is_permission_balanced_v<Γ, InitialPerms>` "ships standalone" — but `grep -rn "is_permission_balanced_v" include/` finds ONLY documentation strings, no template definition. This is the Γ-level analogue to per-handle PSet balance (the `PermissionedSessionHandle` enforces per-channel; the Context analogue does not exist). The CSL × MPST bonding is only the per-handle slice; the per-context slice is ghost-documented.
- **Enabler shipped?** Partly — per-handle balance shipped; Context enumeration shipped. Need fold over `Context::entries` summing permission deltas per Tag.
- **Effort:** medium.

#### Finding 1.5 — SessionPatterns × {Crash, Subtype, Delegate} unverified
- **Severity:** MEDIUM
- **Concepts:** Patterns × all 3 (Crash, Subtype, Delegate)
- **Where:** `sessions/SessionPatterns.h:1-90`
- **What's missing:** Patterns are documented as "self-verifying" with static_asserts for "dual involution, well-formedness, structural expansion" — but NOT subtype refinement, NOT crash-safety against unreliable peers, NOT delegate compatibility. `TwoPhaseCommit_Coord` mentions "crash branches arrive with SessionCrash.h" (line 379, 2026 promise) yet ships unprotected. `RequestResponse_Server` against an unreliable Client is not provably crash-safe at the pattern level. `test_session_patterns.cpp` has 0 grep hits for Subtype/Crash/Delegate.
- **Enabler shipped?** Yes — every primitive exists.
- **Effort:** small per pattern, ~15 patterns × 3 axes = medium aggregate.

**Strategic takeaway (Agent 1):** The session subsystem ships every theoretical *primitive* but skimps on the *cross-axis algebraic laws* that bond the 5 theories — most starkly between Delegate × Crash (1.1) and EventLog × everything-except-Send/Recv (1.2), both of which break Crucible's bit-exact-replay-and-recover guarantee for any non-trivial protocol. The next session-types PR should make the EventLog SessionOp enum *complete* (Stop/Checkpoint/Rollback/Delegate/Accept) and ship the missing `is_permission_balanced_v<Γ, …>` definition.

### §3.2 Agent 2 — Permission × Session integration completeness

**Mandate.** Crucible's L0 substrate ships CSL permissions in `include/crucible/permissions/` (Permission.h, PermSet.h, PermissionFork.h, ReadView.h, FairSharedPermissionPool.h). The framework's central thesis is `PermissionedSessionHandle<Proto, PermSet, Resource, LoopCtx>` in sessions/PermissionedSession.h (1319 lines) — three-axis composition: Graded value invariants × CSL permissions × Session protocols. Find every place where the integration is INCOMPLETE.

**Investigation surface.** sessions/PermissionedSession.h, sessions/SessionPermPayloads.h, sessions/SpscSession.h, concurrent/Permissioned*.h (8 primitives), permissions/*.h.

#### Finding 2.1 — SwmrSession façade for AtomicSnapshot — MISSING **(KEYSTONE)**
- **Severity:** LOAD-BEARING
- **Primitives:** `concurrent/PermissionedSnapshot.h`, `concurrent/AtomicSnapshot.h`, `safety/SwmrWriter.h` / `SwmrReader.h`, `permissions/FairSharedPermissionPool.h`, `sessions/PermissionedSession.h`
- **What's missing:** No `swmr_session.h` analogous to `SpscSession.h`. SAFEINT-§18 (24_04 doc) specifies `class SwmrSession : Pinned<...>` (~200 LOC), and `SessionContentAddressed` (#361) explicitly assumes a `SwmrSession<CompiledKernel, KernelCacheReaderTag>` exists for KernelCache broadcast — but `grep -rn "SwmrSession"` returns ZERO hits in `include/`. PermissionedSnapshot has zero references to Session/`namespace proto`. SwmrWriter/Reader endpoints exist (`safety/SwmrWriter.h`) but never compose into a typed `Loop<Send<ContentAddressed<T>, Continue>>` protocol with SharedPermissionPool refcount evolution on Recv.
- **Entry point:** `include/crucible/concurrent/PermissionedSnapshot.h:1` (no session namespace block); would add `sessions/SwmrSession.h` mirroring `sessions/SpscSession.h:215`.
- **Enabler shipped?** YES — PermissionedSnapshot ships, SharedPermissionPool ships, mint_permissioned_session ships, ContentAddressed ships.
- **Effort:** ~250 LOC + 1 test file. Mechanical mirror of SpscSession.h with `WriterProto = Loop<Send<ContentAddressed<T>, Continue>>` and ReaderProto using `Borrowed<T, ReaderTag>`.

#### Finding 2.2 — PSH × Delegate composition — SessionDelegate ignores PermSet evolution
- **Severity:** LOAD-BEARING
- **Primitives:** `sessions/SessionDelegate.h` (909 lines), `sessions/PermissionedSession.h`, `sessions/SessionPermPayloads.h`
- **What's missing:** PermissionedSession.h has ZERO friend `mint_permissioned_session` or `step_to_next` specialization for `Delegate<T, K>` / `Accept<T, K>` (`grep -n "Delegate\|Accept" PermissionedSession.h` returns 0). When a session endpoint is delegated as a payload, the inner Proto's own PermSet should transfer with it (consume from delegator's PS, insert into accepter's PS). Honda 1998 §6 demands this. SessionPermPayloads only handles plain T payloads via Transferable/Borrowed/Returned; a `DelegatedSession<T, X>` payload marker is missing. `test_session_delegate.cpp` has zero Permission references.
- **Entry point:** `sessions/PermissionedSession.h:1041` (only Send/Recv/Loop/Select/Offer/End specializations exist — Delegate missing); `sessions/SessionPermPayloads.h:99` (Transferable family doesn't include DelegatedSession marker).
- **Enabler shipped?** PARTIAL — PSH framework + SessionDelegate ship as parallel subsystems; the bridge marker (`DelegatedSession<InnerProto, InnerPS>`) and PSH dispatch on Delegate<...> head are missing.
- **Effort:** ~200 LOC + 4 negative-compile fixtures (HS14). Enables CNTP layered nesting (the canonical use case in SessionDelegate.h §F.6) with permission discipline preserved across handoff.

#### Finding 2.3 — `mint_session` returns bare SessionHandle, never PSH — mint family fragmented
- **Severity:** HIGH
- **Primitives:** `sessions/SessionMint.h:215`, `sessions/PermissionedSession.h:897`
- **What's missing:** `mint_session<Proto>(ctx, res)` returns `SessionHandle<Proto, Resource>` with NO PermSet. `mint_permissioned_session<Proto>(res, perms...)` accepts Permission tokens but DOES NOT take a Ctx (no `CtxFitsProtocol` gate). Per the Universal Mint Pattern (CLAUDE.md §XXI), every cross-tier mint should follow ONE shape: `mint_X(Ctx const&, args...) requires CtxFitsX`. There is no canonical `mint_session<Proto, Perms...>(ctx, res, perms...)` that ctx-validates AND consumes Permission tokens AND returns PSH. Production code wanting both row-typed Ctx checking AND CSL discipline must call two functions and lose composability.
- **Entry point:** `sessions/SessionMint.h:215` returns `mint_session_handle<Proto>(...)` (bare); `sessions/PermissionedSession.h:897` takes no Ctx.
- **Enabler shipped?** YES — both factories exist; needs unification.
- **Effort:** ~80 LOC: add `mint_permissioned_session<Proto, Perms...>(Ctx const&, Resource, Permission<Perms>&&...)` overload in SessionMint.h that gates on `CtxFitsProtocol<Proto, Ctx>` AND forwards to PSH constructor. 4 negative-compile fixtures.

#### Finding 2.4 — CrashTransport wraps bare SessionHandle only — no CrashWatchedPSH
- **Severity:** HIGH
- **Primitives:** `bridges/CrashTransport.h:309/353/427/501/626` (all 5 specializations hold `SessionHandle<...> inner_`, NOT `PermissionedSessionHandle<...>`), `sessions/PermissionedSession.h`
- **What's missing:** When a peer dies mid-protocol, a PSH holding `Permission<RemoteWorkerPerm>` needs to either (a) gracefully transfer the remaining permission back to the surviving party or (b) declassify it via a documented policy. Today `CrashWatchedHandle` returns the recovered Resource but the held PermSet is silently dropped — equivalent to a permission leak (LeakSafe violation at the type level). No `CrashEvent<PeerTag, RecoveredPerms...>` payload exists.
- **Entry point:** `bridges/CrashTransport.h:309` — `inner_` field is SessionHandle, not parameterized over PSH variant.
- **Enabler shipped?** PARTIAL — CrashWatchedHandle + Stop combinator ship; PSH-aware overlay missing.
- **Effort:** ~300 LOC + tsan test for permission-recovery race.

#### Finding 2.5 — ChaseLevDeque + MetaLog + ChainEdge — three concurrent primitives with ZERO session façade
- **Severity:** MEDIUM
- **Primitives:** `concurrent/PermissionedChaseLevDeque.h`, MetaLog (no Permissioned variant), Forge ChainEdge GPU semaphore (no header in concurrent/ at all)
- **What's missing:** (a) PermissionedChaseLevDeque has the asymmetric linear-owner × fractional-thieves discipline already encoded in permissions but exposes no `owner_session` / `thief_session` namespace — would need `Loop<Select<Send<T, Continue>, Recv<T, Continue>>>` for owner and `Loop<Recv<Borrowed<T, ThiefTag>, Continue>>` for thieves. (b) `MetaLog.h` has no `PermissionedMetaLog.h` even though it is the parallel SPSC to TraceRing; SAFEINT-K1 calls for it. (c) ChainEdge (semaphore-driven plan sequencing per CRUCIBLE.md §11.9.3) has no concurrent/ header at all — pure GPU primitive lacks any `PermissionedChainEdge<...>` with `Send<SemaphoreSignal, End>` protocol.
- **Entry point:** `concurrent/PermissionedChaseLevDeque.h:154` (deque_tag namespace exists, no session namespace); `MetaLog.h:1` (no permission integration); ChainEdge missing entirely.

### §4.4 Pattern P4 — Permission discipline doesn't carry rows

**Cited by:** Agent 4 (#4.3, #4.4), Agent 6 (#6.2)

**The story.** CSL permissions and Met(X) effect rows are two disjoint type-system axes today. They should compose: `Permission<Tag>` ownership often conveys an effect-row obligation (e.g., disk-spilled region acquisition needs `Effect::Block`). They don't:

- `Permission<Tag>` has no row association — `payload_row<Permission<Tag>>` returns `Row<>` (Agent 4 #4.4)
- `mint_permission_fork<Children...>` spawns std::jthreads (unconditionally `Effect::Bg`) but takes no Ctx, no row admission check (Agent 4 #4.3)
- `CrashTransport` drops PermSet on peer death — equivalent to permission leak at the type level (Agent 6 #6.2 — the LeakSafe BUG)

**The opt-in trait pattern.** A `permission_row<Tag>` trait that opts specific tags into row association:
```cpp
template <typename Tag>
struct permission_row { using type = effects::Row<>; };  // default

template <>
struct permission_row<DiskSpilledRegion> {
    using type = effects::Row<effects::IO, effects::Block>;
};
```

Then `mint_permission_fork` becomes ctx-bound, requires `Subrow<permission_row_t<P>, Ctx::row>` for parent's permission, and threads child Ctx with the appropriate Bg row.

**The CrashWatchedPSH (#6.2) is the bug.** When `OneShotFlag::peek()` fires, the dead peer's `Permission<PeerTag>` should NOT vanish — it should either transfer to a survivor's PermSet via `permission_inherit<DeadTag, SurvivorTag>` or declassify via a documented policy. Today the wrapper returns `CrashEvent<PeerTag, Resource>` with the permissions silently dropped. This is a TYPE-LEVEL LeakSafe violation per CLAUDE.md axiom 7.

## §5. Real bugs vs forward-looking gaps

The 50 findings include 4 REAL BUGS — code that is broken NOW, not infrastructure that's missing. These deserve immediate attention separate from the forward-looking gaps. The remainder are forward-looking opportunity tiers (CRITICAL / LOAD-BEARING / HIGH / MEDIUM / LOW).

### §5.1 BUG 1 — SessionGlobal::project<StopG<Peer>, R> returns End for non-Peer roles

**Source:** Agent 1 #1.3 (HIGH severity)
**File:** `sessions/SessionGlobal.h:191-198`
**Class:** Correctness bug — type system claims a guarantee it does not deliver

**Description.** Projection of `StopG<Peer>` onto a non-Peer role R returns `End` even when R has pending sends to Peer. Should return `Stop` (or a crash-handling Offer wrapping the remainder) when those roles had pending interactions with the crashed Peer.

**Consequence.** Round-trip Γ → G → projection → Γ' loses crash-safety information. The `is_subtype_sync` check on the projected local type passes silently because it sees `End`, but the original Γ entry had the role mid-protocol against the dead Peer. Implementations that honor the projection (correctly per the rule) ship code that doesn't handle the crash branch. The `all_offers_have_crash_branch_v` enforcement disagrees with the projection rule.

**Fix.** Refine `project_case_v` for stop nodes:
1. For each non-Peer role R, walk R's pending interactions with Peer in the global type up to the StopG point.
2. If any pending interaction exists, project to `Stop` for R (or surround the remainder with a mandatory-crash Offer per BHYZ23).
3. Add reachability check: a role with no pending Peer interactions still projects to End (correct).

**Test.** Add to `test_session_global.cpp`: enforce "non-Peer projection of StopG must be Stop or contain a crash-handling Offer when pending interactions exist." This test will fail today, exposing the bug.

**Effort.** Medium. ~150 LOC in `project_case_v` + 3 new self-test fixtures.

### §5.2 BUG 2 — `is_permission_balanced_v<Γ, InitialPerms>` is documented but undefined

**Source:** Agent 1 #1.4 (HIGH severity)
**Files:** `sessions/SessionContext.h:20`, `sessions/PermissionedSession.h:40,114,495`
**Class:** Ghost API — four documented mentions, zero template definitions

**Description.** Four header doc-blocks state `is_permission_balanced_v<Γ, InitialPerms>` "ships standalone." `grep -rn "is_permission_balanced_v" include/` returns ONLY documentation strings, no template definition. This is the Γ-level analogue to per-handle PSet balance. Per-handle balance shipped via PermissionedSessionHandle; the Context analogue does not exist.

**Consequence.** Code that wants to verify "this typing context Γ has balanced permission flow" cannot do so. The L9 ⊥ L2 axis (CSL × MPST bonding) is only the per-handle slice; the per-context slice is ghost-documented. Users reading the docblock would write code that fails to compile when they reach the trait.

**Fix.** Ship the trait. Implementation strategy: fold over `Context::entries`, sum permission deltas per Tag (consume from Send<Transferable>, produce from Recv<Transferable>, etc.), check sum equals `InitialPerms`. Mechanism is mechanical given SessionPermPayloads.h's per-marker dispatch.

**Effort.** Medium. ~120 LOC + 5 self-test fixtures (balanced/unbalanced/empty/multi-tag/cross-context cases). Likely lands as part of SwmrSession (P1 keystone) since SwmrSession provides the per-Γ-entry permission-fold pattern.

### §5.3 BUG 3 — CrashTransport silently drops PermSet on peer death

**Source:** Agent 6 #6.2 (HIGH severity)
**File:** `bridges/CrashTransport.h:309/353/427/501/626` (all 5 specializations)
**Class:** LeakSafe TYPE-LEVEL violation

**Description.** When `OneShotFlag::peek()` fires inside `CrashWatchedHandle`, the dead peer's `Permission<PeerTag>` tokens are LOST. The wrapper returns `CrashEvent<PeerTag, Resource>` but no permissions are reclaimed back to a survivor's PermSet. Per CLAUDE.md axiom 7 (LeakSafe), a `Permission<Tag>` token that vanishes from the type system is equivalent to a leaked resource — the runtime invariant "every Permission has exactly one owner" is violated structurally.

**Consequence.** Code paths that expect to inherit a crashed peer's resources (the reincarnation case from CRUCIBLE.md §L13/§L14) cannot statically prove they have authority over the inherited resources. The recovery path silently runs without the permission tokens it should have received.

**Fix.** Define `permission_inherit<DeadTag, SurvivorTag>` factory in `permissions/PermissionFork.h`. CrashWatchedHandle returns `CrashEvent<PeerTag, Resource, Permission<SurvivorTag>...>` where the SurvivorTag list is computed by the inheritance lattice. The lattice itself is design work: which dead peers' permissions inherit to which survivors? Per-tag opt-in via `inherits_from<DeadTag, SurvivorTag>` trait specialization.

**Effort.** Large (1 week). Type machinery + inheritance lattice design + tsan test for permission-recovery race + 4 neg-compile fixtures.

### §5.4 BUG 4 — 6 of 9 BPF programs compile but never load (dead code)

**Source:** Agent 9 #9.1 (CRITICAL severity)
**Files:** `lock_contention.bpf.c`, `pmu_sample.bpf.c`, `sched_switch.bpf.c`, `sched_tp_btf.bpf.c`, `syscall_latency.bpf.c`, `syscall_tp_btf.bpf.c`
**Class:** Dead code — kernel-side complete, userspace consumer missing

**Description.** Each of these 6 BPF programs defines complete event/map structures with `BPF_F_MMAPABLE` ring buffers wire-spec'd in `common.h:117-160`. They compile cleanly. They are never loaded — `bpf_object__open` is called only for `sense_hub.bpf.c` (`bench/bpf_senses.cpp:47`).

**Consequence.** Engineering investment in 6 BPF programs (perhaps ~1500 lines of BPF-side code) sits unused. The `pmu_sample_event::ip` field — a textbook Augur substrate for bottleneck attribution — never crosses the kernel/userspace boundary. WorkloadProfiler (#322) cannot be implemented without these signals.

**Fix.** Promote `bench/bpf_senses` to `include/crucible/perf/SenseHub.h` with a generic `bpf_load<Program>(target_tgid)` template. Add per-program userspace consumers in `include/crucible/perf/{LockContention, PmuSample, SchedSwitch, SyscallLatency}.h`. Wire into rt/Policy.h and the (planned) AdaptiveScheduler.

**Effort.** ~1 week per program for full integration. ~2 days for the loader template + first program (sched_switch).

## §8. Open questions

These are design decisions surfaced by the audit that need resolution before implementation can proceed. They are NOT engineering tickets; they are policy questions for the project lead.

### §8.1 Permission inheritance lattice (BUG #5.3 design)

When a peer holding `Permission<PeerTag>` dies, which survivors inherit which permissions? Three options:
- **Per-tag opt-in trait specialization** — every tag declares its `inherits_from<DeadTag, SurvivorTag>` spec at definition site. Most explicit; requires foresight.
- **Topology-driven inheritance** — the surviving Keepers' Canopy view determines who inherits. Most flexible; harder to reason about statically.
- **Policy-declared inheritance** — the `.fixy-pkg` artifact declares an inheritance policy at deployment. Moves the decision to operations.

**Recommendation:** Per-tag opt-in (option 1). It's most consistent with the existing CSL discipline (every tag is a type, decisions about that tag belong at the tag declaration site).

### §8.2 Federation peer trust model (Agent 7 #7.5)

For cross-organizational federation, which trust mechanism is the source of truth?
- **Self-signed peer certificates** — each org signs its own peer cert; trust is per-pair handshake.
- **CA-issued peer certificates** — a trusted root CA signs all federation peers. More familiar; introduces a TCB element.
- **Web-of-trust** — peers vouch for other peers transitively. Most decentralized; hardest to reason about.

**Recommendation:** Self-signed for v1 federation. The cross-org gap is design work (no one has specified the FederatedPeer tag tree per fixy.md §15); start simple, add CA later if needed.

### §8.3 Wall-clock dimension framework (deliberate exclusion per fixy.md §24.1)

fixy.md §24.1 explicitly excludes wall-clock dimensions (Latency, Energy, Power, WallClock, BitsTransferred) on grounds that "the compiler cannot prove physical bounds; annotation-only dimensions create false guarantees." But operational reality has these signals everywhere — Augur drift detection, deadline watchdog, BPF telemetry.

**Question:** Should there be a separate annotation framework (NOT type-system enforcement) for wall-clock observations that operationally inform runtime decisions but make no compile-time claims?

**Recommendation:** YES, but in the runtime/observability layer (Augur), not in the type system. Document the annotations as "operational hints, not guarantees" and route through the existing Augur drift detection.

### §8.4 Live evolution of running vigils (fixy.md §22 deferred question)

A vigil running in production may need to evolve: model architecture changes, new stages added, capability requirements updated. The existing live-reshard mechanism (CRUCIBLE.md §12) handles topology changes; live updates to the vigil's structure are not specified.

**Recommendation:** Defer until Phase 7+. Most production deployments restart for vigil structure changes; the use cases that need live evolution are research-grade.

### §8.5 Multi-tenancy model (fixy.md §22 deferred question)

A single Crucible fleet may host multiple vigils for multiple users. Resource isolation, capability quotas, cross-tenant data flow restrictions are not specified. Current model assumes one vigil per fleet.

**Recommendation:** Defer until Phase 6 distributed runtime is proven for single-tenant. Multi-tenancy is largely additive (per-vigil capability quotas + per-tenant Cipher tier limits) and can land as Phase 8+ work.

### §8.6 Reproducibility across compiler versions (fixy.md §22 deferred question)

The L1 / L2 / L3 Cipher cache is keyed on (content_hash, chip_caps_signature). The content hash includes the source's IR001 form; a change to the compiler that changes the IR001 form invalidates the cache even if the source is unchanged. The version-skew handling is not specified.

**Recommendation:** Phase 9 work (federation across organizations needs this; single-org doesn't). Add a `compiler_version_compat: range` field to cache entries.

### §8.7 The Augur namespace shape (Agent 9 #9.5)

5+ headers reference `Augur::*` as a consumer that doesn't exist. Phase 6 ships Augur, but the API surface is undefined.

**Recommendation:** Ship a 50-line `augur/Predictor.h` skeleton in Phase 6 (or earlier) with `record_metric()` as the doorway. The substrate references can then point to a real (if minimal) namespace.

## §9. Sequencing notes

This section explains why the §7 ranked backlog is in this order, not another. The ranking optimizes for unblocking cascades, not for individual finding severity.

### §9.1 Why P0-2 is rank 1, not rank 2 or 3

P0-2 closes Phase 0. Without Phase 0 closure, every Fixy phase is gated. The strategic doc (misc/02_05_2026.md "Sequencing rule") puts P0-2 explicitly first. Even if Pattern P1 (SwmrSession) has more downstream cascade unblocks, P0-2 has architectural precedence: it is the LAST piece of Phase 0.

The risk of doing P0-2 second: if SwmrSession lands first and reveals P0-2 design issues, P0-2 might churn. But the spec for P0-2 is fully written in fixy.md §24.2 (12 explicit rules); the design risk is low.

### §9.2 Why SwmrSession is rank 2, not rank 3

SwmrSession is the single most-cited audit finding (4 agents). Its cascade unblocks:
- KernelCache R32 migration (#414)
- AtomicSnapshot for Augur metrics (#387)
- ContentAddressed-Cipher publish (Agent 7 #7.2)
- `is_permission_balanced_v<Γ, …>` ghost API closure (Agent 1 #1.4)
- RecipeRegistry read-only-after-init pattern (#979)

Stage/Pipeline row-check (rank 3) has only ONE downstream cluster (METX-AUDIT/VERIFY + FOUND-I11..I15). Stronger cascade = higher rank.

### §9.3 Why bug fixes are interleaved, not bulk-first

A natural impulse is "fix all bugs first, then move forward." We don't, because:
- BUG #5.1 (SessionGlobal projection) blocks Phase 4 (when crash-safety starts being tested in production). It's a Phase 4 prerequisite, not an immediate blocker.
- BUG #5.2 (`is_permission_balanced_v` ghost API) is naturally fixed alongside SwmrSession (rank 2) since SwmrSession provides the per-Γ-entry permission-fold pattern.
- BUG #5.3 (CrashWatchedPSH) is rank 6 because it gates Phase 6 (distributed crash-stop), and the design (permission inheritance lattice) is itself an open question (§8.1).
- BUG #5.4 (BPF dead code) is rank 9 because the BPF SenseHub bridge is the right vehicle to address it.

Each bug is fixed when its gating phase is the next thing to unblock. This interleaves fixes with forward progress instead of bulk-first.

### §9.4 Why Vigil.h is rank 7 (worked-example precedence)

Agent 10 #10.1 (Vigil.h pending_region_ migration) is a 2-day fix that closes 2 WRAP-Vigil tasks. By raw effort/impact ratio, it should be rank 1. It's rank 7 because:
- It's a Phase 3 prerequisite (production migration template), not Phase 0/1
- The CASCADE unblocked by other higher-ranked items is larger
- But once Phase 3 is the next phase to attack, this becomes the FIRST thing to ship

This is the "worked-example precedence" pattern: cheap, layout-neutral migrations of canonical cases come BEFORE bulk migrations of similar cases. Establishes the template.

### §9.5 Why the Cipher sweep is rank 10 (5-PR aggregate)

Agent 7's entire report (5 findings) maps to a Cipher integration sweep. As an aggregate, it's ~3 weeks of work — comparable to several other ranked items. We rank it last because:
- It's a Phase 6 prerequisite, the LATEST phase among the audit prereqs
- All 5 enablers are already shipped; the work is integration only
- Each sub-PR is independent — they can ship as bandwidth allows

The Cipher sweep is BIG and IMPORTANT but it's not BLOCKED. It can wait.

## §11. What was NOT audited

The 10 agents partitioned the integration-audit space along the axes named in §1, but several areas were only superficially examined or skipped entirely. This section names those gaps so a future audit run can target them deliberately.

### §11.1 Visualization tree (vis/) — touched only by Agent 9

`include/crucible/vis/` ships ~1672 LOC across 5 headers (TraceVisualizer, SugiyamaLayout, NetworkSimplex, SvgRenderer, BlockDetector). Agent 9 #9.2 noted that visualization was forensic-only (offline `.crtrace` consumption) but did not audit the layout algorithms themselves, the SVG output format, or potential interaction with the diag/ catalog. GAPS-092 partially closes the live-trace bridge by adding `vis/MerkleDagAdapter.h` and `vis/LiveTraceVisualizer.h` over the real `TraceNode` / `RegionNode` root surface; full visualization audit remains open. A dedicated visualization-audit agent could examine: layout-algorithm correctness on degenerate inputs (cycles, single nodes, ~10K-node graphs), SVG-output reproducibility (deterministic ordering of overlapping nodes), the BlockDetector heuristic for identifying optimization-relevant subgraphs, and production Augur event wiring once GAPS-104 lands.

### §11.2 Test infrastructure (test/) — implicit only

The 10 agents read tests to understand what IS verified. None audited the test infrastructure ITSELF. Open questions: the 600+ ctest entries — are they all green on every preset? Is the neg-compile harness consistent in diagnostic regex specificity? Does the property-fuzz infrastructure (Philox-seeded inputs) cover every load-bearing invariant? Are the 30 negative-compile fixtures from fixy.md §24.12 v1 corpus all shipped (current count is 10 v0.1)? Do tsan + asan presets cover every concurrent primitive's stress test?

### §11.3 Bench infrastructure (bench/) — only the BPF subset audited

`bench/` was touched by Agent 9 (BPF SenseHub consumer) and Agent 10 (DoNotOptimize asm gotcha mentioned in passing). Open questions: the bench harness's noise floor (variance discipline per CLAUDE.md §VIII), whether bench results are persisted across runs for regression detection, whether the bench fleet has the right hardware coverage (NV/AMD/CPU baseline vs the 5-vendor target).

### §11.4 Build system (CMake, presets) — not audited

`CMakeLists.txt`, `CMakePresets.json`, the verify preset's Z3 gating, the per-TU contract semantic switching, the warning-error promotion list, the GCC 16 + libstdc++ 16 dependency surface. Open questions: does `verify` preset actually run today? Are all TUs covered by the appropriate semantic? Does the build system gate on the cheat probe / neg-compile fixtures correctly?

### §11.5 Documentation system (misc/) meta-organization

The 30+ files under `misc/` (CRUCIBLE.md, FORGE.md, MIMIC.md, fixy.md, 24_04_2026_safety_integration.md, 25_04_2026.md, 27_04_csl_permission_session_wiring.md, 28_04_2026_effects.md, 02_05_2026.md, this doc, ...) — are they internally consistent? When fixy.md says "ALGEBRA-15", does that task ID exist? When CRUCIBLE.md cites §11.9, does that section say what other docs claim it says? Cross-doc consistency was not audited.

### §11.6 vessel/torch/ frontend tree

`vessel/torch/` (the PyTorch adapter) is the production consumer of TraceRing. Agent 10 #10.2 noted vessel_api.cpp's opaque-handle ABI as a HIGH-severity production substrate adoption gap, but did not audit the rest of vessel/. Open questions: how does vessel handle ATen op-set drift (PyTorch upgrades)? Does the schema-table fallback (crucible_fallback.cpp) correctly cover the long tail of ATen ops? Is there per-op coverage tracking?

### §11.7 The augur/ tree — does not exist

Agent 9 #9.5 noted that 5+ headers reference an `Augur::*` namespace that doesn't exist. The audit identified this gap but could not audit the (non-existent) augur/ tree itself. When Phase 6 ships the augur scaffold, a dedicated augur-audit agent should examine: API surface, drift-detection correctness, recommendation engine ranking, telemetry consumer wiring.

### §11.8 Cross-language interop (Python frontend, JAX frontend)

Agent 10 mentioned vessel_api.cpp + vessel_api.h + crucible_native.py + crucible_fallback.cpp but did not audit the Python-side surface. The Fixy researcher path (`@cr.kernel.attention(...)`, `cr.fleet("local")`, etc.) per fixy.md §2.1 is a major UX surface. Open questions: do the .pyi type stubs cover every public API? Does the Cursor autocompletion experience work as described? Are the diagnostic translations (C++ template error → Python traceback) correct?

### §11.9 CI/CD pipelines — not audited

The cross-vendor numerics CI matrix from fixy.md §24.11 is the operational discharge mechanism for numerical-equivalence claims. Open questions: is this CI matrix actually running? What's the per-PR cell count today? Are there platforms that consistently fail (NV-only / AMD-only / CPU-only fixtures)? Does the fleet-wide numerics dashboard exist?

### §11.10 Verdict on coverage limitations

The 10 agents partitioned ~80% of the substrate-and-runtime surface; the remaining ~20% (vis/, test/, bench/, build, docs, vessel, Python, CI) is acknowledged as future-audit scope. The findings reported here are SOUND (every cited file:line is real, every cited gap is genuine), but they are not COMPLETE — the next audit run should explicitly target the §11.1-§11.9 areas.

## §15. Risk register

What could go wrong with each rank-1-through-10 move? This section names the failure modes and the mitigation for each.

### §15.1 Rank 1 — P0-2 CollisionCatalog risks

**R1.1 — A rule in §24.2 is genuinely uncheckable at compile time.** Possible: some collisions (e.g., L002 borrow × Async) require flow-sensitive analysis the C++ type system doesn't have natively. Mitigation: ship the rule as a runtime contract (`pre()`) where compile-time check is impossible; document the limitation; route through `safety/Diagnostic.h` Catalog with a `runtime` discriminator.

**R1.2 — Activating `static_assert(ValidComposition<Fn>)` breaks existing examples/fn/* code.** Likely: the 5 worked examples were written against a placeholder ValidComposition; they may use compositions the catalog rejects. Mitigation: PR 14 (activation) is gated on running every example through the catalog; failures result in either example update or rule scope adjustment.

**R1.3 — A rule's diagnostic is too generic to be actionable.** Possible: "constraints not satisfied" is the default, and the per-rule Catalog wrapper could fail to surface the named diagnostic. Mitigation: each per-rule PR includes the neg-compile fixture verifying the diagnostic regex matches the named rule code.

### §15.2 Rank 2 — SwmrSession risks

**R2.1 — SwmrSession's permission-fold pattern doesn't generalize to `is_permission_balanced_v<Γ, …>`.** Possible: SWMR is a special case (1 writer, N readers); the Γ-level closure may need machinery SWMR doesn't exercise. Mitigation: ship #5.2 (the ghost API closure) as a SEPARATE PR after SwmrSession lands; if the pattern doesn't generalize, the ghost API needs its own design pass.

**R2.2 — KernelCache R32 migration reveals layout-incompatibility.** Possible: KernelCache slots are 24 bytes with FOUND-I05 wire-format pin; the SwmrSession-typed access pattern may not map cleanly to the existing slot layout. Mitigation: PR 5 (R32 migration) is gated on `static_assert(sizeof(KernelCacheSlot) == 24)` + federation wire-format compatibility test. If incompatible, KernelCache R32 stays parked and only AtomicSnapshot consumes SwmrSession initially.

**R2.3 — SharedPermissionPool's atomic refcount becomes a contention hot spot under N-reader load.** Possible: at ~1000 readers, the refcount CAS could serialize. Mitigation: the SAFEINT-B18 task description specifies SharedPermissionPool as the synchronization primitive; if contention shows, layer in a per-NUMA-node refcount sharding.

### §15.3 Rank 3 — Stage/Pipeline row-check risks

**R3.1 — Existing call sites of mint_stage / mint_pipeline break.** Likely: stages and pipelines today are written without payload row awareness; the row-check will reject some that are semantically correct but under-typed. Mitigation: grep all call sites first; for each broken site, decide whether (a) the site needs explicit row annotation, or (b) a compatibility path that auto-deduces row from payload.

**R3.2 — Variadic fold over `row_union_t` exceeds template depth limits.** Unlikely but possible at very long pipelines (>100 stages). Mitigation: the existing concurrent/Stage.h supports up to ~16 stages per Pipeline; fold complexity is O(N²) in stages; stays well within compiler limits.

### §15.4 Rank 4 — SessionGlobal bugfix risks

**R4.1 — The corrected projection rule rejects code that previously compiled.** Likely: existing test_session_global.cpp tests may use the buggy rule's output; refining the rule makes them fail. Mitigation: the corrected rule is STRICTER — it surfaces previously-silent crash-safety holes. Existing tests that fail are SHOULD-FAIL tests that were passing because of the bug; their failure is a feature.

**R4.2 — Downstream consumers of the projection rule break.** Possible: code that relied on the buggy `End` projection now sees `Stop` and may not handle it. Mitigation: the corrected rule emits a clearer diagnostic (per Agent 1 #1.3 description) so downstream breakage is well-localized.

### §15.5 Rank 5 — row_hash_contribution risks

**R5.1 — The wrapper specializations introduce cycle in template instantiation.** Unlikely (each specialization is a leaf in the wrapper-nesting hierarchy). Mitigation: existing 2-spec coverage proves the recursion depth is bounded; the new specs follow the same pattern.

**R5.2 — A federation cache key collision shows up in production after the migration.** Possible: existing (pre-migration) entries hashed under the old `value=0` rule; post-migration code queries with the new hashed values; cache misses everywhere. Mitigation: federation cache is content-addressed by content_hash, NOT by row_hash; row_hash is the cache SLOT, not the cache KEY. Migration is forward-compatible.

### §15.6 Rank 6 — CrashWatchedPSH risks

**R6.1 — The permission-inheritance lattice is the WRONG abstraction.** Likely: deciding which dead peers' permissions inherit to which survivors is non-trivial; the lattice may need to be data-flow-sensitive (which depends on the protocol). Mitigation: §8.1 names this as an open question. Ship the lattice for the simple cases (1-to-1 inheritance) first; punt the data-flow-sensitive cases to a follow-up.

**R6.2 — The tsan test (PR 3) is flaky.** Possible: peer-death races are inherently nondeterministic. Mitigation: per CLAUDE.md HS5, run the test 50× in stress preset; if variance > 5%, the bug is in the test, not the implementation.

### §15.7 Rank 7 — Vigil.h migration risks

**R7.1 — PublishOnce's release-publish semantics differ from std::atomic's release-store.** Unlikely (the substrate review of PublishOnce per #1082 confirmed the semantics). Mitigation: the existing `pending_region_` callers use `std::memory_order_release` on store and `std::memory_order_acquire` on load — exactly PublishOnce's contract.

### §15.8 Rank 8 — AdaptiveScheduler risks

**R8.1 — The cache-tier rule's threshold (256KB) is wrong for non-Tiger Lake hardware.** Possible: AMD Zen 4 has different L2 sizes; ARM Graviton has different cache hierarchies. Mitigation: the cache-tier-rule lookup is parameterized on Topology (sysfs probe); each platform gets its own threshold.

**R8.2 — BackgroundThread rewrite breaks bit-exact replay.** Critical: BackgroundThread builds RegionNodes from TraceRing entries; any nondeterminism in the new pipeline implementation breaks replay. Mitigation: PR 4 (bench harness) explicitly includes the bit_exact_replay_invariant test from CLAUDE.md §XIII.

### §15.9 Rank 9 — BPF SenseHub risks

**R9.1 — BPF program loading fails on older kernel versions.** Likely: the 6 currently-orphaned BPF programs use BTF tracepoints that require kernel 5.5+. Mitigation: SenseHub's `bpf_load<Program>(target_tgid)` template returns std::expected; missing kernel features result in graceful degradation (telemetry simply unavailable).

**R9.2 — The deadline watchdog reacts too aggressively to telemetry noise.** Likely: any threshold-based reaction can oscillate. Mitigation: per CRUCIBLE.md §16.1, the watchdog uses hysteresis (deadline_miss_budget over watchdog_window_sec); SenseHub provides the windowed counters needed.

### §15.10 Rank 10 — Cipher sweep risks

**R10.1 — Federation MPST design (sub-PR 1) is wrong.** Possible: FederationProtocol's actual usage may not be 1-writer-N-readers; it might be N-writer-N-reader. Mitigation: prototype the protocol type FIRST, then run the existing federation tests against it; only ship when tests pass.

**R10.2 — ContentAddressed wrap (sub-PR 2) breaks Cipher tier promotion.** Unlikely (ContentAddressed is structurally transparent), but possible if existing tier promotion relies on wire bytes being raw. Mitigation: each Cipher integration sub-PR is independently testable; sub-PR 3 (tier promotion) lands AFTER sub-PR 2 with full test coverage.

**R10.3 — Unified event log (sub-PR 4) increases Cipher write latency.** Possible: if SessionEvent is larger than Cipher.LogEntry (currently small POD), tier-write throughput drops. Mitigation: bench Cipher write latency before and after; if regression > 5%, profile and shrink the SessionEvent variant.

**R10.4 — FederatedPermission (sub-PR 5) introduces a TCB element.** Possible: if peer admittance uses crypto signatures, HACL* dependency expands. Mitigation: HACL* is already in the TCB inventory per fixy.md §2.2; sub-PR 5 is an additive use, not a new TCB.

### §15.11 Cross-cutting risks

**R-X.1 — The audit findings are already stale by the time PRs land.** Likely: substrate evolves; new findings emerge. Mitigation: re-run the 10-agent audit at 4-month intervals (per §11 closing). Each run produces a delta against this baseline.

**R-X.2 — Engineer parallelization (§9.7) creates merge conflicts.** Possible: engineers 1+2+3 working in parallel may touch overlapping headers. Mitigation: the §9.7 partition is along independent file groups (engineer 1: substrate keystone, engineer 2: substrate-internal, engineer 3: production + bugs); if conflicts arise, merge daily.

**R-X.3 — The §8 open questions block work in progress.** Possible: §8.1 (permission inheritance lattice design) blocks rank 6. Mitigation: §8 questions are identified as POLICY decisions; each has a recommendation. The project lead should resolve them in a single sitting before rank 6 starts.

## §16. Decision log

Implicit decisions in this audit's structure and ranking. Surfacing them so they can be challenged.

### §16.1 Why Pattern P1 (SwmrSession) outranks Pattern P2 (Cipher) despite Cipher having more findings

**Decision:** SwmrSession is rank 2; Cipher sweep is rank 10.

**Rationale:** SwmrSession is the PREREQUISITE for Cipher sub-PRs 2+3+4 (ContentAddressed, tier promotion, unified event log). Cipher sweep cannot ship before SwmrSession ships. Severity (Cipher: 5 findings vs SwmrSession: 1) does not equal sequencing rank; dependency order does.

**Counter-argument:** Cipher sweep could start in parallel on the sub-PRs that don't depend on SwmrSession (sub-PR 1 Federation MPST, sub-PR 5 FederatedPermission). The §9.7 parallel arrangement allows this.

### §16.2 Why bugs are interleaved with forward gaps, not bulk-first

**Decision:** Bug fixes (#5.1, #5.2, #5.3, #5.4) are at ranks 4, 2 (bundled), 6, 9 — not all at the top.

**Rationale:** Each bug gates a SPECIFIC Fixy phase. Bug #5.1 (SessionGlobal projection) gates Phase 4; not fixing it before Phase 4 is fine. Bug #5.4 (BPF dead code) gates Phase 6; not fixing it before Phase 6 is fine. Bulk-first bug-fixing creates a "clean substrate" optical illusion without accelerating Fixy phase delivery.

**Counter-argument:** A clean substrate has compounding value (audits stay shorter, technical debt accumulates more slowly, new contributors trust the surface). The audit could re-rank with bugs first.

### §16.3 Why per-vendor backend wirings (Pattern P8) are bundled into Phase 4 prereq, not a separate rank

**Decision:** Pattern P8 (Lattices × Sessions) appears as a Phase 4 prerequisite (§6.5) but does NOT appear as a top-10 ranked move.

**Rationale:** Phase 4 (Mimic-NV first backend) is months 6-10. The lattice × session wirings are needed when the per-vendor backend pattern matures, not before. Ranking them in the top 10 would imply they should ship in the next 12 weeks; they don't need to.

**Counter-argument:** The wirings could ship now as substrate quality improvements, decoupled from Phase 4. Agent 3's strategic takeaway specifically picks NumericalTier × Send/Recv as the highest-payoff first wiring.

### §16.4 Why production substrate adoption (Pattern P7) gets only rank 7, not higher

**Decision:** Vigil.h migration (Agent 10 #10.1) is rank 7. The broader Pattern P7 (production substrate adoption) is the §6.4 Phase 3 prerequisite, not a top-tier move.

**Rationale:** Per misc/02_05_2026.md Bucket 3 explicit deferral, R29-R36 production refactor tasks are PARKED pending the fixy/ directory creation (Phase 3 reorganization). Doing them before Phase 3 means doing them twice (once now, once after the 70/30 split). Vigil.h #10.1 is the exception because it's layout-neutral.

**Counter-argument:** The 200+ WRAP-* tasks represent real technical debt; deferring them indefinitely accumulates more debt as the substrate evolves.

### §16.5 Why P0-2 (#1096) is rank 1, despite Phase 0 being "almost done"

**Decision:** P0-2 CollisionCatalog is rank 1 despite being only one of six Phase 0 deliverables and the others all being shipped.

**Rationale:** "Almost done" doesn't unblock Phase 1. Phase 1 codegen targets the COMPLETE Phase 0 substrate; without CollisionCatalog, the activated `ValidComposition<Fn>` gate is a placeholder. The strategic doc (misc/02_05_2026.md "Sequencing rule") explicitly puts P0-2 first.

**Counter-argument:** Phase 1 itself is months 1-3, so P0-2 could ship anytime in the next 6 weeks without delaying Phase 1 start. SwmrSession (rank 2) could go first if it has higher cascade.

### §16.6 Why the audit is structured per-agent (§3) before cross-cutting (§4), not the reverse

**Decision:** §3 preserves each agent's full report verbatim; §4 synthesizes patterns.

**Rationale:** Each agent's report is the load-bearing primary source; the synthesis is editorial. Future audits can compare agent-by-agent against this baseline; cross-cutting patterns may evolve as the substrate matures and agents refocus.

**Counter-argument:** Most readers will skip §3 (10 × ~150 lines = 1500 lines of dense detail) and read only §4. The structure could lead with patterns and put per-agent details in an appendix.

### §16.7 Why "what success looks like" is not an explicit section

**Decision:** Each rank in §7 names its deliverables but no separate "success criteria" subsection.

**Rationale:** Success for a substrate primitive is "ships + green tests + documented diagnostic + neg-compile fixtures + zero regression in bench." This is uniform across the 10 ranks; calling it out per-rank would be ceremony.

**Counter-argument:** Cross-vendor numerics CI matrix passing rate is a different success metric than green ctest. For Phase-6-gating items (Cipher sweep, BPF SenseHub), federated test coverage is a different dimension.
