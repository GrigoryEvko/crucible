# Fixy — Crucible's reject-by-default discipline layer

*16 May 2026. Companion to `misc/fixy.md` (deferred DSL design) and `misc/02_05_2026.md`, `misc/03_05_2026.md`, `misc/08_05_2026_harness.md`. Author: superintelligence pair.*

**Thesis.** Fixy is not a DSL. Fixy is a thin **defaults-flipping discipline layer** over Crucible's existing substrate that enforces the FX paradigm (reject-by-default, explicit-grant) at the C++26 template level. ~1000 LoC of new headers turn the permissive `safety::` substrate into a strict surface that refuses to compile any binding whose grade vector has not been explicitly engaged with on every dimension. The substrate stays where it is; fixy/ is a namespace + concept layer on top.

This document specifies the design, the roadmap, and — critically — the **adoption discipline** that says every new Crucible primitive composes through `fixy::` rather than raw substrate. Past `safety::` direct calls stay as the implementation; future construction goes through fixy.

---

## 0. The frame: what fixy IS and ISN'T

**Fixy IS:**

- A C++26 header tree under `include/crucible/fixy/`, ~13 headers, ~1000 LoC, zero new dependencies.
- A **strict surface** where the default of every binding is the most restrictive setting per the FX 20-dimension grade vector (fixy.md §24.1), and every relaxation is annotated.
- A **discipline boundary**: code written against `fixy::fn<...>` cannot accidentally do IO, allocate, mutate, leak, copy a linear handle, branch on a secret, or escape a region — unless the binding's `Grant` pack explicitly admits the capability.
- A **graduation path**: existing production code stays on `safety::*`. New code, new vendor backends, new compiler phases, new transport headers, new bench harnesses, new examples — every greenfield site — composes through `fixy::*`.
- A **publishing format**: the `Fn<...>` instantiation IS the manifest. Federation caches key on the canonical wrapper-nesting hash (already shipped via `safety/diag/RowHashFold.h` per GAPS-028); fixy::fn produces stricter, more discriminating cache keys.

**Fixy IS NOT:**

- A parser. No `.fixy` source files. No `fixyc`. No transpiler.
- A new compiler tier. Uses GCC 16 patched build, same `default` / `release` / `tsan` / `sketch` presets.
- A separate dialect. Every `.cpp` file parses identically; only what the elaborator's concept gates accept differs.
- A replacement for `safety::`. The substrate stays as the implementation; `fixy::` is a strict-discipline surface that re-exports under different defaults.
- An SMT-backed verifier. Discharge is via `Decide.h` + Pre/Post + cross-vendor CI + neg-compile fixtures, per fixy.md §24.13 and `08_05_2026_harness.md` Tier 0/Tier 1.

## 1. Why now, and why this shape

### 1.1 The substrate is done; the discipline is not

Audit of `include/crucible/` (16 May 2026): 530 files, 57 directories. Every FX 22-dimension wrapper ships at production grade. Every FX kernel construct (Fn aggregator, CollisionCatalog 12 rules, DimensionTraits S/L/T/F/V, Reflected, Pre/Post/Decide) ships. Every FX surface construct that survives the fixy.md §24.14 inheritance map (sessions: 33 headers; machines: Machine.h; contracts: Contract.h + Cipher migration; effects: 12 headers; permissions: 8 headers) ships. The 23-axiom kernel is mapped to operational discharge per fixy.md §24.13.

What does **not** ship is the **discipline that enforces reject-by-default**. Today a developer can write:

```cpp
safety::Fn<int> double_it = [](int x) { return x * 2; };
```

and the substrate accepts it: no `Linear` annotation, no `Tot` effect declaration, no security grade, no trust assertion — every dimension defaults to a permissive value that the C++26 compiler simply accepts. This is the **opposite** of FX §1.2 "capabilities denied by default and granted explicitly." The wrappers exist; the rejection does not.

### 1.2 The 5%-not-yet-done is the load-bearing 5%

What's missing is the gate that takes every Fn instantiation and says: "did the author engage with every one of the 20 dimensions, either by explicit `grant::*` or by explicit `default_strict_for<Dim>`?" Without that gate, the substrate is a library of types that nobody is forced to use. With that gate, every fixy:: binding is a complete capability manifest readable in isolation.

The gate is structurally simple: one concept (`IsAccepted<F>`) that combines the existing `ValidComposition<F>` from `safety/CollisionCatalog.h` (the 12 collision rules, all shipped) with a new **engagement check** that requires each dim to appear in the `Grant` pack — either as a relaxation or as an explicit default-acceptance.

### 1.3 The vertical-integration story

Crucible's stated goal: "robust substrate for distributed HPC and ML that was never seen before with vertical integration from graphs and opses to own proprietary binary compilers." The vertical stack is:

```
Vessel (PyTorch / JAX / native) → IR000 → IR001 (TraceRing/MerkleDag) → Forge (12 phases)
  → IR002 (KernelGraph, NumericalRecipe) → Mimic (per-vendor backends)
  → IR003* (NV/AM/TPU/TRN/CPU) → vendor binary (ioctl emit, no libcuda/libtpu/libnrt)
  → Cipher (3-tier persistence) → CNTP (AF_XDP transport) → Canopy (SWIM+Raft mesh)
```

Every layer of this stack composes against the substrate. Today the composition is at the `safety::` level — each header instantiates wrappers directly. **The fixy/ discipline turns "compose against substrate" into "compose against capability manifest."** A Mimic backend that secretly allocates on the hot path can't pass `fixy::fn<..., grant::with_alloc>` if its row doesn't admit Alloc. A Forge phase that secretly does IO can't pass `fixy::fn<..., grant::with_io>` without that grant. A CNTP transport that mishandles a Permission can't pass `fixy::fn<..., grant::permission<RemoteWorkerPerm>>` without the inheritance lattice being checked.

This is the vertical integration: every layer that gets built in fixy:: produces a binary whose capabilities are visible in the type signature of every public entry point.

## 3. The IsAccepted gate — the load-bearing concept

The whole of fixy/ pivots on one concept. Sketched:

```cpp
namespace crucible::fixy {

  template <typename F>
  concept IsAccepted =
       safety::ValidComposition<F>            // existing 12-rule §6.8 catalog
    && Engaged<F, Dim::Type>                  // every dim engaged either via
    && Engaged<F, Dim::Refinement>            // grant or via explicit default
    && Engaged<F, Dim::Usage>
    && Engaged<F, Dim::Effect>
    && Engaged<F, Dim::Security>
    && Engaged<F, Dim::Protocol>
    && Engaged<F, Dim::Lifetime>
    && Engaged<F, Dim::Provenance>
    && Engaged<F, Dim::Trust>
    && Engaged<F, Dim::Representation>
    && Engaged<F, Dim::Observability>
    && Engaged<F, Dim::Complexity>
    && Engaged<F, Dim::Precision>
    && Engaged<F, Dim::Space>
    && Engaged<F, Dim::Overflow>
    && Engaged<F, Dim::Mutation>
    && Engaged<F, Dim::Reentrancy>
    && Engaged<F, Dim::Size>
    && Engaged<F, Dim::Version>
    && Engaged<F, Dim::Staleness>
    && NotInTheoryCorpus<F>;                  // §30.14 rejection set
}
```

`Engaged<F, Dim::X>` returns true iff `F::Grants` contains either:
- A relaxation tag for X (e.g., `grant::with<Effect::IO>` engages `Dim::Effect` with a granted IO row);
- Or `grant::accept_default_strict_for<Dim::X>` — the explicit acknowledgment that the strict default is the chosen value.

The discipline is: **the author has read the discipline and made a choice on every dim.** A wrong choice fails `ValidComposition` (collision catalog). No choice fails `Engaged` (not yet engaged). Both produce structured diagnostics through the existing `safety/Diagnostic.h` Catalog framework.

`NotInTheoryCorpus<F>` is a finite enumeration check: if `F`'s shape matches one of the FX §30.14 known-unsoundness patterns (e.g., Atkey-2018 Lam: linear capture in unrestricted closure with body using it twice), it rejects with a citation to the source paper. The corpus grows monotonically per the §30.14 discipline; the check is O(|corpus|) at compile time and discharges via reflection-driven pattern matching on `F`'s `Grants` pack and `Body` type signature.

### 3.1 What this looks like at the use site

```cpp
// REJECTED — no engagement on any dim
fixy::fn<int> double_it = [](int x) { return x * 2; };
// error: IsAccepted<fn<int, /* empty Grants */>> not satisfied
//   Dim::Usage not engaged — add grant::accept_default_strict_for<Dim::Usage>
//                            OR grant::affine / grant::copy / grant::ghost
//   Dim::Effect not engaged — add grant::accept_default_strict_for<Dim::Effect>
//                             OR grant::with<Effect::IO, ...>
//   ... 18 more dims ...

// ACCEPTED — author has acknowledged every dim
fixy::fn<int,
    grant::accept_default_strict_for<Dim::Type>,
    grant::accept_default_strict_for<Dim::Refinement>,
    grant::copy,                                          // Usage: grade w
    grant::accept_default_strict_for<Dim::Effect>,        // Effect: Tot
    grant::accept_default_strict_for<Dim::Security>,
    /* ... 14 more ... */
> double_it = [](int x) { return x * 2; };

// In practice the common case is encoded by a STANCE alias:
fixy::fn<int, fixy::stance::PureCopy> double_it = [](int x) { return x * 2; };
//   — stance::PureCopy expands to the explicit-accept-all-strict-defaults pack
//     with `grant::copy` swapped in for the Usage default.

// Other canonical stances (each ~3 lines in fixy::stance namespace):
//   PureLinear     — strict on every dim, Linear usage (the canonical default)
//   IoFunction     — strict + grant::with<Effect::IO>
//   BgWorker       — strict + grant::with<Effect::Bg>
//   CtCrypto       — strict + grant::with<Effect::CT> + Linear
//   SecretConsumer — strict + Linear + retains Classified
//   PublicEmit     — strict + grant::declassify<Policy>
//   AsyncEndpoint  — strict + grant::with<Effect::Async> + Linear (no borrow)
```

The stance aliases are the ergonomic shortcut: 90% of code picks one of ~8 canonical stances. The explicit-engagement form is what stances expand to, but the author who reaches for it has already thought about which stance fits — the discipline holds.

### 3.2 The cost model

`sizeof(fixy::fn<int, stance::PureCopy>) == sizeof(int)` — every grant tag is an empty type, every default is regime-1 EBO-collapsed per `algebra/Graded.h`. No runtime cost; the gate fires only at template instantiation.

Compile-time cost: one extra concept evaluation per `fixy::fn<...>` instantiation. Empirically (estimated from `safety/Fn.h`'s 19-axis aggregator already shipping in CONTRACT-030 fuzz harness): ~0.5ms per instantiation. A 1000-fn translation unit adds ~0.5s to the build. Within the existing `-fconstexpr-ops-limit=100000000` ceiling.

## 4. The Roadmap — six phases over four weeks

### Phase A — Foundation (3 days, ~350 LoC)

Ship `Dim.h`, `Default.h`, `Grant.h`, `Reject.h`. Validate end-to-end with:

- 8 positive-compile tests: one per Tier × stance combination (PureLinear × S, PureCopy × S, IoFunction × S, CtCrypto × S, SecretConsumer × T, PureLinear × L, PureLinear × F, PureLinear × V)
- 12 neg-compile fixtures: one per missing-dim engagement
- 1 cheat-probe round: 5 attempts to bypass `IsAccepted` (e.g., trait-spec injection on `Engaged`, default-template specialization, friend-class circumvention)

**Acceptance:** `IsAccepted<>` rejects every grant-empty `fn<T>` and accepts every fully-engaged stance. `static_assert(sizeof(fixy::fn<int, stance::PureCopy>) == sizeof(int))` holds.

**Status gate:** without Phase A nothing else compiles; without the cheat probe nothing else is trusted.

### Phase B — Fn aggregator + Rules (5 days, ~260 LoC)

Ship `Fn.h`, `Rules.h`. The Fn aggregator maps `Grants` → `safety::Fn<Type, /* resolved params */>` instantiation through Default+Grant resolution metafunctions. `Rules.h` exposes the 12 collision rules under `fixy::rule::*` for diagnostic clarity.

- 12 neg-compile fixtures: one per §6.8 rule, each triggered via the fixy:: path
- 1 round-trip test: `fixy::fn<...>` instantiates the underlying `safety::Fn<...>` with identical type-parameter list (witnessed via `std::is_same_v`)
- 2 ergonomics tests: stance aliases (`stance::PureCopy`, `stance::IoFunction`) produce identical Fn types as their fully-expanded equivalents

**Acceptance:** every neg-compile fixture in `safety_neg/` that targeted a `safety::Fn` rejection has a `fixy_neg/` mirror that fails for the same reason via `fixy::fn`.

### Phase C — Sess + Mach + Contract aliases (5 days, ~180 LoC)

Ship `Sess.h`, `Mach.h`, `Contract.h`. These are namespace aliases over existing substrate; the work is in choosing the right surface names per FX inheritance (fixy.md §24.14).

- Sess: `fixy::sess::Send / Recv / Loop / Select / Offer / End / Stop` for the 7 surface combinators; `fixy::phi::{safe, df, term, nterm, live, live_plus, live_pp}` for the 7 safety predicates; `fixy::sess::mint_*` for the 5 mint factories (per CLAUDE.md §XXI Universal Mint Pattern, mint discipline is preserved).
- Mach: `fixy::mach::{state, transition, inverse, on_fail, atomic, idempotent, commutative}` and LTL aliases for the 4 std/ltl operators.
- Contract: `fixy::contract::{version, migration, compatibility, format, access, invariant}` over Cipher's existing serialization + migration machinery.

**Acceptance:** every existing session in `sessions/SessionPatterns.h` (15 patterns) compiles unchanged with `using namespace fixy::sess;`. The aliases are inert — they cannot accept code that the underlying header rejects.

### Phase D — Profile + Theory + Fixy umbrella (3 days, ~200 LoC)

Ship `Profile.h`, `Theory.h`, `Fixy.h`.

- Profile: maps to CMake `default` / `release` / `sketch` (new) presets. Sketch flips `IsAccepted` to `IsAcceptedWithWarning` via P2900 contract-evaluation-semantic `observe`. Release stays `enforce` per existing CMakePresets default. The CMake var `CRUCIBLE_FIXY_STRICT` toggles per-target.
- Theory: ships the §30.14 Known-Unsoundness corpus as 15 entries. Each is a `static_assert(!compiles<Pattern>)` template + a doc-comment citing the source paper (Atkey 2018, Caires-Pfenning 2010, BSYZ22, Capecchi-Castellani-Dezani-Rezk 2014, etc.). The corpus grows monotonically per §30.14 discipline — no entry ever deleted.
- Fixy.h: umbrella. Sets `CRUCIBLE_FIXY=1`. Single include.

**Acceptance:** sketch preset compiles all 12 rule violations as warnings, not errors. Release preset rejects them all. The 15 §30.14 entries each have a paired test that proves the corpus catches them.

### Phase E — Worked examples (3 days, ~600 LoC of example code, not in budget)

Ship 5 worked examples mirroring `examples/fn/*`:

```
examples/fixy/
├── example_custom_kernel.cpp      — user GEMM (smallest stance: PureLinear + tier::BITEXACT)
├── example_custom_optimizer.cpp   — Adam step (BgWorker + Alloc + Linear<param>)
├── example_cntp_frame.cpp         — wire frame (External + sanitize + retag → Sanitized)
├── example_forge_phase.cpp        — pure IR transform (PureLinear + Verified trust)
└── example_mimic_backend_hook.cpp — vendor emit (BgWorker + vendor<NV> + IO for ioctl)
```

Each example:
- Carries heavy per-axis rationale comments explaining the stance choice
- Includes `static_assert(sizeof(BoundFn) == sizeof(Body))` proving EBO collapse
- Registers in ctest with `-L examples` so `ctest -L examples` runs all 5

**Acceptance:** the five examples demonstrate every stance in `fixy::stance::`. A new contributor reading them understands the discipline without reading any other doc.

### Phase F — Discipline gate (2 days, ~50 LoC)

Ship `scripts/check-fixy-discipline.sh`. CI grep linter that flags raw `safety::Fn<` use in any file under a configured set of greenfield directories (those declared as "fixy-only" in `CMakeLists.txt`).

- The greenfield list is **explicit, not implicit**: a directory opts INTO fixy-only via `set_target_properties(... PROPERTIES CRUCIBLE_FIXY_ONLY ON)`. No directory is fixy-only by default; existing code never breaks.
- Initial fixy-only opt-ins: `examples/fixy/`, `test/fixy_neg/`, and (over time) any new `cog/`, `mimic/`, `forge/`, `cntp/`, `canopy/`, `topology/` headers added after the fixy/ ship date.

**Acceptance:** the linter rejects `safety::Fn<` in `examples/fixy/`; accepts it everywhere else. The CMake property gates the linter per-target. A new contributor adding a header in a fixy-only directory who reaches for `safety::Fn<` gets a CI fail with the specific line.

## 5. The adoption discipline — vertical-integration policy

This is the load-bearing policy section. Without it, fixy/ becomes a parallel substrate nobody uses. With it, Crucible's vertical-integration goal is structurally achieved over time.

### 5.1 Three-band classification

Every Crucible header lives in one of three bands:

**Band 1 — Substrate primitives (~150 headers).** `safety/*`, `effects/*`, `permissions/*`, `algebra/*`, `concurrent/*`, `handles/*`, `bridges/*`, `sessions/*`. These are the wrappers and primitives the rest of the system composes against. They live at the lowest layer and are **the implementation of fixy itself** — fixy/ types alias and constrain them, but their construction is permitted to use raw `safety::*` and raw C++26. New band-1 primitives (rare) follow existing discipline.

**Band 2 — Production runtime (~80 headers).** `Vigil.h`, `BackgroundThread.h`, `TraceRing.h`, `MetaLog.h`, `Cipher.h`, `MerkleDag.h`, `Arena.h`, `PoolAllocator.h`, `ReplayEngine.h`, `Transaction.h`, `TraceGraph.h`, the entire Forge IR001 surface, the existing Mimic CogMimic scaffolding. These are the production-data-path headers. They **stay on safety::** indefinitely. The WRAP-* migration backlog (~110 tasks, ~100 closed) continues to migrate raw types to safety wrappers; that work is unchanged. fixy/ doesn't touch band 2.

**Band 3 — Greenfield (everything new from 17 May 2026 onward).** Every new header added to `include/crucible/` after fixy/ ships is **fixy-only** by default — i.e., composes against `fixy::` types, not raw `safety::`. The CMake property `CRUCIBLE_FIXY_ONLY ON` is set on greenfield directories at the time of creation, and the discipline linter enforces it.

Concretely, the greenfield surface as of 16 May 2026 covers:

- Every new `mimic/{nv,am,intel,mellanox,broadcom,cpu}/network/Backend.h` per-vendor backend (currently `_wip/`, ~6 headers projected)
- Every new `cntp/*` transport (in-flight: KtlsOffload, QuicTransport, WireguardTransport, Sharp, Doca, P4 — ~8 headers)
- Every new `canopy/*` membership header (in-flight: HyParView, Plumtree, Lifeguard — 3 headers; already shipped: Crdt, VectorClock, Hlc, Scuttlebutt, Swim)
- Every new `topology/*` discovery header (in-flight: Discovery, Telemetry, Ptp — 3 headers; shipped: AsymmetricFailure, CongestionTelemetry, Pingmesh, TopologyGraph)
- Every new `cog/*` calibration header (shipped: 9 headers)
- Every new `forge/Ir001/*`, `forge/Phases/*`, `forge/recipes/*` Forge phase (in-flight: Comm, Phases/Comm, recipes/Network)
- Every new `observe/*`, `warden/*` runtime-observation header (shipped: ~10 headers each; future ones fixy-only)
- Every new example, every new bench harness

**Band-3 is where vertical integration happens.** A vendor backend that lands under fixy:: discipline has a public surface that declares — through its `fixy::fn<...>` signatures — exactly which effects, allocations, vendor pin, recipe tier, NUMA affinity, and CSL permissions it requires. Composing five Mimic backends (NV, AM, TPU, TRN, CPU) into a cross-vendor numerics CI run becomes a row-union check (already shipped via `Subrow` + GAPS-025 `CtxFitsPipeline`); the row union IS the cross-product audit.

### 5.2 Concrete vertical-integration cases

Six load-bearing cases where fixy/ discipline pays off across the vertical stack:

**(a) Vendor backends — Mimic/{nv,am,intel,mellanox,broadcom,cpu}.** Today the `_wip/mimic/.../network/Backend.h` headers are scaffolds with no enforced surface. Under fixy/, every backend's `emit_kernel<KernelKind, Recipe>(...)` entry point is a `fixy::fn<CompiledKernel, stance::BgWorker, grant::vendor<V>, grant::tier<R>, grant::with<Effect::IO>, ...>`. The composition gate verifies: NV backend with BITEXACT_TC recipe can only be composed against a context whose row admits IO and whose vendor lattice contains NV. A CNTP layer that calls an NV-only backend from an AM-only host fails at template instantiation — the existing `VendorLattice` + `mint_substrate_session` (GAPS-067/068) does the actual check; fixy:: is the surface that surfaces it as a single stance rejection.

**(b) Forge phases A-L.** Each phase has a wall-clock budget per FORGE.md §5. Under fixy/, each phase is a `fixy::fn<Out, stance::PureLinear, grant::tier<R>, grant::with<Effect::Bg>>` where Out is the next-phase input type. Composition of phases is `mint_pipeline<...>` (GAPS-026 already shipped); the pipeline's row union is the phase budget. A phase that secretly allocates beyond its declared budget fails the row admission check.

**(c) CNTP transport stack.** Each transport (AfXdp, RoceConfig, MtlsTransport, KtlsOffload, Sharp, Doca, P4, WireguardTransport, QuicTransport, etc.) is a `fixy::fn<Result, stance::AsyncEndpoint, grant::vendor<V>, grant::permission<Tag>>`. CrashWatchedHandle composition (GAPS-003/045/046 shipped) inherits permissions on peer death; fixy:: surfaces the inheritance through the stance signature. A new transport added under fixy:: cannot accidentally drop a permission on crash because the signature declares the permission tag.

**(d) Canopy membership protocols.** HyParView / Plumtree / Lifeguard (in-flight) compose with existing Crdt / VectorClock / Hlc / Scuttlebutt / Swim. Under fixy/, each protocol is `fixy::fn<MembershipUpdate, stance::BgWorker, grant::permission<MembershipTag>, grant::epoch_required>` — the epoch_required grant maps to `EpochVersioned<T>` (GAPS-070/071 shipped). Membership transitions that race against reshard fail because EpochedDelegate's well-formedness check fires at the fixy:: composition site.

**(e) Cipher migration + federation.** Cipher's three-tier persistence with content-addressed federation (GAPS-039 through GAPS-046 shipped) becomes a `fixy::contract::version<N> migration<from, to>` per CipherTier promotion path. Under fixy/, the `fixy::contract::*` aliases force every Cipher boundary crossing to be a verified total function — matching FX §14.2 modal-univalence-equivalence per fixy.md §24.14 (we keep migration as verified total function, drop the cubical-evaluation overhead).

**(f) Observe + Warden runtime-observation loop.** SenseHub + PmuSample + SchedSwitch + LockContention + SyscallLatency (GAPS-004a..i shipped) feed into WorkloadProfiler + Policy deadline watchdog. Under fixy/, each telemetry pipeline is `fixy::fn<Sample, stance::BgWorker, grant::with<Effect::IO, Effect::Block>>`. The fixy:: surface forces the runtime-observation loop to declare its blocking behavior — making it impossible to wire a syscall-blocking BPF loader into a hot-path TU by accident.

In every case, the discipline pays off the same way: **fixy:: types make the binding's full capability set visible in the signature.** Reading a function's signature tells the next contributor what it does. No type aliasing, no opaque template parameter packs, no hidden defaults.

### 5.3 The migration policy

**Existing code (Bands 1+2): no migration.** WRAP-* backlog continues. Production runtime stays on `safety::` indefinitely. The audit trail of safety::wrappers in production code is the audit trail.

**New code (Band 3): fixy-only from 17 May 2026.** Every new header under `mimic/`, `forge/`, `cntp/`, `canopy/`, `topology/`, `observe/`, `warden/`, `cog/`, and every new example, every new bench harness composes through `fixy::`. The CMake property + CI linter enforces.

**Exception process:** a band-3 directory that genuinely needs raw `safety::` access (rare; usually because it implements a new substrate primitive that fixy/ itself depends on) opts out by setting `CRUCIBLE_FIXY_ONLY OFF` on its target. The opt-out requires a comment in the CMakeLists.txt citing the substrate primitive being implemented and a reference to the relevant Decide.h discipline. CI does not enforce on opt-out targets but logs them in `build/fixy_audit.txt` for review.

**Backward compat:** zero. `fixy::` is purely additive. No existing code breaks. No existing wrapper is deleted, renamed, or moved.

### 5.4 Federation cache key implications

`row_hash_contribution` (GAPS-028/029 shipped) folds canonical-order wrapper nesting into a federation-cache key. Under fixy/, the fold gains a new contribution: the `Grants` pack. A `fixy::fn<int, stance::PureCopy>` and a `fixy::fn<int, stance::PureLinear>` produce different cache keys even though their underlying `safety::Fn<int, ...>` differs only in Usage grade. The federation cache becomes more discriminating: fewer false-positive cache hits, more reproducible cross-organization builds.

Concretely, the federation cache key under fixy/ encodes:
- The 20-dim grade vector (via the existing canonical wrapper-nesting fold)
- The §6.8 collision-rule classification (each `Grants` pack maps to a 12-bit subset of rules that ARE relaxations of the strict default; the federation cache slot keys on this subset)
- The Theory-corpus version (so a §30.14 update invalidates exactly the cache entries that match the new pattern)

This is the operational analog of FX §25.5 Minimal Version Selection + §25.10 reproducibility theorem: the same fixy:: source compiles to the same `.fixy-pkg`-equivalent artifact across organizations because every dimension's value is in the type signature, hence in the cache key.

## 7. The numerics — federation, cross-vendor CI, vertical correctness

Crucible's correctness story per fixy.md §24.11 + MIMIC.md §41: every (KernelKind × NumericalRecipe × vendor) tuple compiles, runs, outputs are compared pairwise against CPU scalar-FMA oracle, tolerance per the recipe's declared tier (BITEXACT_STRICT / BITEXACT_TC / ORDERED / UNORDERED). A backend that violates tolerance fails the build.

Under fixy/, this story tightens in two ways:

**(a) Surface obligation.** Every Mimic backend's `emit_kernel<K, R>(...)` signature under fixy:: declares its recipe tier as a grant: `grant::tier<R>`. Composition of a backend with the cross-vendor CI fixture instantiates `mint_pipeline<...>` (GAPS-026 shipped) whose row admission verifies the recipe ID. A backend that compiles to BITEXACT_TC but secretly emits ORDERED-tier instructions cannot pass the row check at template instantiation — even before runtime numerics CI fires.

**(b) Federation discharge.** The federation cache key encodes the recipe tier. A kernel compiled to BITEXACT_TC at org A and downloaded by org B via Cipher cold-tier federation (GAPS-039/040 shipped) inherits the tolerance guarantee. The downstream verification at org B doesn't re-run numerics CI — it trusts the cache key. The trust is grounded in fixy:: surface obligation + federation peer trust model (GAPS-107 DECISION: self-signed for v1).

Together this is the load-bearing claim of Crucible-as-substrate-for-distributed-HPC-and-ML: **the type signature of every public entry point is a complete capability manifest**, federation is content-addressed by that manifest, and cross-vendor numerics correctness is enforced before the binary leaves the publishing organization.

## 9. Risks and what could go wrong

**R1 — Discipline drift.** Future code is supposed to live under fixy::, but a contributor reaches for `safety::Fn<` because it's familiar. The CI linter catches this only on directories with `CRUCIBLE_FIXY_ONLY ON`. Mitigation: the linter runs at PR-merge time, not at developer-build time, so the friction is at review-not-build. The greenfield directory list is appended to whenever a new band-3 directory is created (governance: CMake property is the canonical record).

**R2 — Stance explosion.** Five canonical stances (PureLinear, PureCopy, IoFunction, BgWorker, CtCrypto, SecretConsumer, PublicEmit, AsyncEndpoint) cover most cases. But edge cases — a function that needs `grant::declassify<P>` AND `grant::with<Effect::Network>` AND `grant::vendor<NV>` AND `grant::tier<BITEXACT_TC>` — need explicit grant packs. Mitigation: stance aliases compose. `fixy::stance::compose<CtCrypto, grant::vendor<NV>>` produces the CtCrypto stance with the NV vendor pin added. The compose metafunction is ~30 LoC.

**R3 — Compile-time cost.** Every `fixy::fn<...>` instantiation runs the `IsAccepted` concept which evaluates 20+ engagement checks + the 12 collision rules + the 15 §30.14 corpus checks. On a 1000-fn TU this might add 1-2 seconds to the build. Mitigation: the existing `safety::Fn` already runs `ValidComposition` (GAPS-018); fixy::fn adds the 20-dim engagement check, ~0.2-0.5ms per instantiation. Within budget on the patched GCC 16 + `-fconstexpr-ops-limit=100000000`.

**R4 — Diagnostic verbosity.** A failed `IsAccepted<>` with 20 unsatisfied dims produces a wall of text. Mitigation: structured diagnostics via the existing `safety/Diagnostic.h` Catalog framework (FOUND-E01 shipped). The first failing dim gets the structured Goal/Have/Gap/Suggestion form; the remaining 19 are summarized in one line. Plus stance aliases mean most failures are "you wrote `fixy::fn<int>` — pick a stance: PureLinear / PureCopy / IoFunction / BgWorker / CtCrypto."

**R5 — The strict-defaults disagreement.** What does "strict-default for Mutation" mean? Per fixy.md §24.10 it's `Immutable`. But `Mutation.h` has `Immutable / AppendOnly / Monotonic / ReadWrite` — four levels. Different authors may disagree on which is "strict." Mitigation: the `Default.h` header is the canonical answer. Disagreement gets resolved at fixy/ ship time, not at use time. Strict = the FX §6.3 dim's annotated default. For Mutation: `Immutable`. For Effect: `Tot` (empty row). For Trust: `Verified` (top of lattice). For Reentrancy: `NonReentrant`. Etc.

**R6 — §30.14 corpus maintenance.** Type-theory bugs accumulate; the corpus grows monotonically. Each new entry is one neg-compile fixture + one `static_assert` rejection. Mitigation: the corpus is data, not code. New entries are 10-line additions to `Theory.h`. The discipline is the same as adding a new Decide.h procedure or a new CollisionCatalog rule.

### A.8 — Coordination notes for the clean reimplementation

The substrate is ready as-is (modulo the 52 LoC of A.7 mandatory additions). The clean fixy reimplementation is purely DELETE-and-alias work — no new algebra, no new lattices, no new modality enums, no new collision-rule catalogs (Agents B and C cover the Rules and Modality redundancies). Every grant tag becomes either (a) a 2–4 line engagement marker on a substrate dim, (b) a one-line `using` alias to a shipped `safety::*` wrapper, or (c) a resolver routing rule that picks the right wrapper based on the grant's NTTP payload.

The discipline this appendix encodes: **fixy is the engagement-and-rejection veneer, not a second algebra**. If a clean-reimplementation diff adds a new lattice file under `algebra/`, or a new wrapper file under `safety/`, the addition needs an external justification (covers a payload semantic the existing 32 lattices / 37 wrappers don't reach). The default answer is "no — alias the existing primitive."
