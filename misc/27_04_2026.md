# Crucible — 27 April 2026 Foundation Spec for the Smart Substrate

*This document specifies the lower-level foundation Crucible needs to wire and glue together so that, later, a smart user-facing API can be built on top with zero compromise on type discipline, CSL ownership, session-type protocol enforcement, or runtime cost. It is not the user-facing API. It is the substrate the user-facing API will dispatch into. Last updated 2026-04-27. Companion to `25_04_2026.md` (new primitives), `24_04_2026_safety_integration.md` (safety × sessions × permissions integration) and `THREADING.md` (the concurrency framework). Read alongside `CLAUDE.md`.*

---

## 0. Scope, non-scope, voice

### 0.1 What this document is

A specification of the components — types, traits, concepts, primitives, glue — required to bring Crucible's existing accreted substrates (Graded algebra, safety wrappers, CSL permissions, session types, lock-free concurrency primitives, Cipher persistence, P2996 reflection) into a single load-bearing foundation. The foundation is what the *next* layer above it (the smart user API of choice — sketched in `25_04_2026.md` §2 onwards, surveyed across three architectural angles in the 27 April working notes) will dispatch into.

This document treats the foundation itself as the deliverable. The user-facing surface is treated as a downstream consumer whose specifics we deliberately leave open. What matters here is that everything the smart layer might want to derive — concurrency shape, permission tree, fusion eligibility, NUMA placement, effect-row enforcement, content-addressable caching — is already encoded in the type system and the wrappers, with zero abstraction overhead, before any user-facing call site exists.

### 0.2 What this document is not

It is not a user-facing API design. The earlier rounds of dataflow / annotated-imperative / type-spec / signature-as-spec exploration produced consensus that the user surface can be deferred; the load-bearing question is whether the substrate carries enough information that any of those four user surfaces could read what it needs. This document focuses on that prior question.

It does not specify abstract cost dimensions of any kind. Cost as ns-per-item, FLOP intensity, bytes-bandwidth-per-second, complexity classes such as O(N log N) — none of these appear in the foundation as a budget the user is asked to declare or that Crucible promises. Every cost-shaped decision is driven by directly-measurable hardware facts: cache hierarchy sizes (L1d, L2, L3, DRAM), NUMA distances, cache-line widths, alignment requirements, working-set bytes. The previous-round cleanup pass (commit landed ahead of this document) removed every abstract-cost reference from headers and docs; this document continues that discipline by specifying the foundation entirely without them.

### 0.3 Voice

Direct, opinionated, paper-citing where the literature applies, no hedging on architectural commitments. Every spec in this document either exists today, has a tracked implementation task, or is sketched concretely enough that an engineer can begin. Where a design choice has alternatives, the rejected alternatives are named and the reason for rejection is given. Open questions are flagged as such; everything else is committed.

## 1. Design principles

These six principles bind every component spec in §5 and every wiring decision in §6.

### 1.1 Zero abstraction overhead

The foundation must not impose runtime cost above the underlying primitive. A `Permission<Tag>` parameter is one byte (zero via EBO) at runtime; the type system carries the proof. A `Refined<positive, int>` over an integer compiles to an integer with a `pre()` clause that under hot-TU semantics collapses to `[[assume]]`. A `Linear<T>` over a value occupies the same bytes as `T`. The dispatch layer above the foundation will be consteval — concept-overloaded specialization, no virtual dispatch, no runtime registry — and will produce machine code byte-for-byte identical to a hand-written equivalent. This is not aspiration; it is verified per primitive in the cross-vendor numerics CI sweep (task #508, GRADED-HEALTH).

### 1.2 Concrete over abstract

The foundation models cost via concrete hardware facts, not abstract dimensions. Allowed: working-set bytes, cache sizes per level, NUMA distance matrices, cache-line widths, alignment caps. Disallowed: ns-per-item budgets, FLOP/byte intensity targets, complexity-class promises about Crucible primitives. The cache-tier rule (THREADING.md §5.4 and `concurrent/CostModel.h::recommend_parallelism`) is the canonical example: it decides Sequential vs Parallel(N) from working-set bytes alone, against a topology probe at process startup. No nanosecond appears in the rule.

The previous round's documentation cleanup pass (commit 2026-04-27) removed every `compute_ns(N)`, every "p99 target X ns" table, every "per-op cost ~X ns" header docstring, and every aspirational throughput claim that was framed as a guarantee. The foundation continues that discipline. Where a primitive's structural per-call shape must be documented (one acquire-load + one release-store + one cell touch, e.g.), the documentation describes the SHAPE, never the magnitude.

### 1.3 Type-as-spec

The user's parameter types ARE the specification. A function declared `void f(safety::OwnedRegion<float, MyTag>&&)` carries — without any annotation — the information that it consumes (`&&`) a single contiguous region of `float` tagged `MyTag`. A function declared `void g(fx::Bg, safety::Stale<GradShard, 16>&&)` declares it operates in a Bg context and accepts a gradient shard with bounded staleness ≤ 16. Each declared parameter type is a load-bearing fact that the dispatch layer reads.

This principle has a corollary: every Crucible primitive whose state the dispatch layer might want to inspect must carry that state in its type, not in a separately-declared annotation. Capacities, Tags, Sources, Predicates, Lattices, Sessions, Modalities — all live in the type system. There is no `[[crucible::cost(...)]]` attribute in this foundation; there is no `compute_ns_per_item`; there is no `LatencyBudget` lattice. (The latter was already dropped per `25_04_2026.md` §2.4 — the rejection is now codified in this principle.)

### 1.4 CSL discipline everywhere

Every cross-thread channel, every shared region, every fork-join, every multi-party protocol carries `Permission<Tag>` at the type level. Bare atomic counters and bare std::atomic-typed fields are not foundation primitives; they are implementation details inside foundation primitives. Foundation primitives expose Permission-typed handles at their boundaries.

This principle constrains every primitive we build. `SpscRing<T, N>` exposes a `Permission<spsc_tag::Producer<UserTag>>` and `Permission<spsc_tag::Consumer<UserTag>>` at its handle boundaries; the bare `try_push` / `try_pop` calls that exist for migration purposes are explicitly subordinate to the Permissioned variants. The same holds for MpscRing, MpmcRing, ShardedSpscGrid, ChaseLevDeque, and AtomicSnapshot. Today this is partially shipped (PermissionedSpscChannel, PermissionedSnapshot); the foundation requires completing it (see §5.3).

### 1.5 Signature dispatch, not annotation

The foundation enables the dispatch layer to read everything from function signatures and parameter types — never from separately-declared attributes the user has to remember to add. If a function takes a `safety::OwnedRegion<T, Tag>&&`, the dispatcher reads "consumes a tagged region," derives the splits_into_pack tree, and lowers to `parallel_for_views<N>`. The user did not write `[[crucible::parallel]]`; the parameter type already said it.

This principle constrains what the foundation must publish. Wrapper types must be reflectable enough that the dispatch layer can extract their grade, lattice, and modality at compile time. Attributes (where they exist at all in the foundation, primarily for debugging diagnostics) are derived, not authored.

### 1.6 C++26 native — no DSL, no plugin, no JIT

The foundation is plain C++26 headers, compiled by GCC 16, linked into the user's binary. No clang plugin, no embedded DSL the user must learn, no JIT compilation step, no separate compilation phase. P2996 reflection at consteval time, P2900 contracts on boundaries, P1306 expansion statements for iterating reflected members, P3491 `define_static_array` for memoizing canonical forms — all standard. The foundation must work entirely within the C++26 surface area.

This principle has a hard consequence: anything that requires inspecting the body of a user function is not in the foundation. Auto-vectorization, loop fusion across statements, automatic tile-sizing of nested loops, body-AST-driven optimization — all defer to the compiler's auto-vectorizer or to the user writing them by hand using `safety::parallel_for_views` directly. The foundation captures everything at the parameter / type level.

## 3. The parameter-shape protocol

The dispatch layer reads function signatures. For it to do so reliably, the foundation publishes a small, fixed catalog of canonical parameter shapes. Each shape is a recognizable pattern of parameter types that maps to a concrete lowering. Functions that follow the protocol can be dispatched automatically; functions that don't fall through to manual orchestration (the existing `safety::parallel_for_views`, `safety::permission_fork`, `concurrent::Queue` direct usage).

The catalog has seven shapes. The dispatch layer recognizes them by reflection over parameter types; it does not require attributes. The foundation's job is to ensure each shape is unambiguously identifiable from the type system alone.

### 3.1 Unary transform

A function consuming one `OwnedRegion<T, Tag>&&` and producing one (the same or a transformed) region. Shape signature:

```cpp
auto f(safety::OwnedRegion<T, Tag>&&) -> safety::OwnedRegion<U, OutTag>;
// or
void f(safety::OwnedRegion<T, Tag>&&);   // in-place
```

Lowering: per-element parallel via `parallel_for_views<N>` over the consumed region; permission tree auto-generated as `splits_into_pack<Tag, Slice<Tag, 0..N-1>>`; `N` chosen by the cache-tier rule from `Tag`'s region size.

### 3.2 Binary transform / map-pair

Two `OwnedRegion`s consumed; one (or two) produced. Shape signature:

```cpp
auto f(safety::OwnedRegion<T, A>&&, safety::OwnedRegion<U, B>&&) -> safety::OwnedRegion<V, OutTag>;
```

Lowering: `parallel_apply_pair<N>(rA, rB, body)` (the existing pair-iterator from THREADING.md §5.3). Permission tree generated for both Tags. Element-count match enforced by a contract (`pre`).

### 3.3 Reduction

One `OwnedRegion<T, Tag>&&` consumed; an accumulator wrapped in `safety::reduce_into<R, Op>` parameter.

```cpp
R f(safety::OwnedRegion<T, Tag>&&, safety::reduce_into<R, Op>&);
```

Lowering: `parallel_reduce_views<N, R>(region, init, mapper, reducer)`. The `Op` parameter on `reduce_into` carries the associative reducer (sum, max, min, custom). The framework folds N partials into the final accumulator; `Op` must be associative for parallelism (commutative for any-order parallelism). The foundation enforces associativity via concept; non-associative Ops fall through to sequential.

### 3.4 Producer endpoint

A function holding a producer-side handle of a Permissioned channel (`Queue<T, K>::ProducerHandle` or equivalent). Shape signature:

```cpp
void f(concurrent::ProducerHandle<T, K, UserTag>&&,
       safety::OwnedRegion<T, In>&&);
```

Lowering: producer-thread dispatch into the channel; the framework supplies the right Kind based on `K`, splits the input region into batches, calls `try_push_batch` if available else loops `try_push`. The producer's Permission tag is an auto-generated child of the channel's whole-channel tag.

### 3.5 Consumer endpoint

Symmetric to §3.4 — a function holding a consumer-side handle.

```cpp
void f(concurrent::ConsumerHandle<T, K, UserTag>&&,
       safety::OwnedRegion<T, Out>&&);
```

Lowering: consumer-thread loop; `try_pop_batch` where available; framework handles drain-on-shutdown semantics.

### 3.6 SWMR reader / writer pair

An `AtomicSnapshot<T>` exposed via two function shapes — one writer (publish) and N readers (load).

```cpp
void publisher(concurrent::SwmrWriterHandle<T, UserTag>&&, T value);
T    reader(concurrent::SwmrReaderHandle<T, UserTag>&&);
```

Lowering: writer maps to `AtomicSnapshot::publish`; reader maps to `AtomicSnapshot::load`. Permission tree: one `Writer<UserTag>` (linear), N `Reader<UserTag>` (fractional via `SharedPermissionPool`). The foundation provides the Pool wrapping; the dispatch layer routes the calls.

### 3.7 Pipeline stage

A function whose first parameter is a `ConsumerHandle` for the previous stage's output and whose second parameter is a `ProducerHandle` for the next stage's input.

```cpp
void stage(concurrent::ConsumerHandle<In, K1, S1>&&,
           concurrent::ProducerHandle<Out, K2, S2>&&);
```

Lowering: `safety::parallel_pipeline<Stages...>` (THREADING.md §5.3) — each stage runs in its own jthread; inter-stage queues materialized at framework level; permission tree auto-generated as a chain.

### 3.8 The catch-all

Any function that does not match a canonical shape falls through to manual orchestration. The user calls `parallel_for_views`, `permission_fork`, `Queue<T, K>::producer_handle()`, `AtomicSnapshot::load()` directly. This is the documented escape hatch; nothing in the foundation forces shape conformance. The dispatch layer will refuse to auto-dispatch a non-canonical function and emit a diagnostic naming the unrecognized parameter type.

## 4. Signature dispatcher mechanics (the abstract surface)

The dispatcher is the layer above the foundation that consumes the parameter-shape protocol and routes to lowerings. This document does not specify the dispatcher itself — that is the user-facing API question deliberately deferred. But it specifies what the dispatcher must be able to read from the foundation, so the foundation's specs in §5 are sized to match.

A canonical dispatcher entry point looks like:

```cpp
namespace crucible {
template <auto FnPtr, typename... Args>
auto dispatch(Args&&... args);
}
```

For this entry point to work, the foundation must publish:

1. **Reflection on `FnPtr`'s parameter list.** P2996R13 `parameters_of_v<^^FnPtr>` plus `template for` iteration. The foundation must guarantee that every parameter type whose presence is meaningful (every wrapper from substrate 2, every Permission tag, every effect token, every reduce_into) is reflectable down to its template arguments.

2. **A parameter-shape recognizer.** A `consteval` predicate `is_unary_transform<Sig>`, `is_reduction<Sig>`, etc., one per shape in §3. These are concepts over the parameters_of result.

3. **An auto-permission-tree generator.** Given a parameter list containing `OwnedRegion<T, Tag1>&&, OwnedRegion<T, Tag2>&&`, emit a `splits_into_pack` specialization for the inferred Whole tag. See §5.4.

4. **A cache-tier-driven N selector.** Given a parameter list containing one or more `OwnedRegion<T, Tag>` plus an optional working-set hint (the region's runtime size), call `recommend_parallelism(WorkBudget{...})` and dispatch to the right factor. See §5.7.

5. **An effect-row checker.** Given the parameter list and the caller's enclosing effect context, verify that the callee's row is a Subrow of the caller's. Compile error otherwise. See §5.1.

6. **Concept-overloaded specialization.** Once the shape is recognized, a `requires` clause selects one of `invoke_inline`, `invoke_sequential`, `invoke_parallel<N>`, `invoke_pipelined<Stages>`. These are foundation-level functions; the dispatcher just routes.

7. **Diagnostic infrastructure.** When recognition fails, the diagnostic must name the parameter type that couldn't be classified. See §5.8.

The dispatcher itself ships in a future document; the foundation must enable all seven of the above. Each is specified in §5.

## 5. Component specs

Each of the following sections specifies one component the foundation needs. Each section contains: the gap, the concrete API or trait surface, the integration with existing substrates, the implementation cost in lines and headers, and dependencies on other components in this document.

### 5.1 Met(X) effect rows — the Computation carrier

**Gap.** The legacy `fx::*` capability tokens (`fx::Alloc`, `fx::IO`, `fx::Block`, `fx::Bg`, `fx::Init`, `fx::Test`) are zero-byte phantom structs passed by value. They work as compile-time capabilities, but they do not compose: a function that needs both `fx::Alloc` and `fx::Bg` must take both as parameters, and there is no row arithmetic — no union, no subrow check, no quantification. The foundation's signature dispatcher (§4 step 5) needs to verify that a callee's effect row is a subrow of the caller's; this requires Met(X) row arithmetic, not the current ad-hoc `static_assert(has<fx::Bg, Caps...>)` pattern scattered through the codebase.

`25_04_2026.md` §3 already specified the Met(X) construction (Tang & Lindley POPL 2026). Tasks #473–479 (METX-1 through METX-7) track the implementation. This section restates what the foundation requires and what the migration path looks like.

**Foundation requirement.**

```cpp
namespace crucible::effects {

// Effect tags — one per fx::* capability today, plus future additions.
enum class Effect : uint8_t {
    Alloc, IO, Block, Bg, Init, Test, Refute, /* future... */
};

// The row — a compile-time set of Effects.
template <Effect... Es>
struct Row {
    static constexpr std::array elements = {Es...};
    static consteval bool contains(Effect e) noexcept;
    static consteval std::size_t size() noexcept;
    // ... canonical sort + dedup applied at construction
};

using EmptyRow = Row<>;

// Subrow inclusion: R1 is a subrow of R2 if every element of R1 is in R2.
template <typename R1, typename R2>
inline constexpr bool Subrow = /* fold over R1::elements */;

// Row union (for composing chained Computations).
template <typename R1, typename R2>
using row_union_t = /* sorted dedup of R1 ∪ R2 */;

// The Computation carrier — wraps a value with its effect row.
template <typename Row, typename T>
class [[nodiscard]] Computation {
    T value_;
public:
    constexpr Computation(T v) noexcept : value_{std::move(v)} {}
    constexpr T extract() && noexcept requires (Row::size() == 0);
    template <typename Row2>
    constexpr Computation<Row2, T> weaken() && noexcept requires (Subrow<Row, Row2>);
    template <typename Row2, typename U>
    constexpr Computation<row_union_t<Row, Row2>, U>
        then(Computation<Row2, U>(*k)(T)) && noexcept;
};

// Backward-compat aliases — drop in for fx::* uses during migration.
namespace fx_legacy {
    using Alloc = Effect::Alloc;
    using IO    = Effect::IO;
    // ... etc.
}

}  // namespace crucible::effects
```

**Integration.** Every existing Crucible function that takes `fx::Bg` etc. as a parameter today migrates to taking the `Computation<Row<Bg>, T>` carrier as a return type or to declaring its row in the function's annotation. The legacy `fx::*` parameter pattern stays compilable via a backward-compat shim (`effects/compat/Fx.h`, task METX-4) for one release, then removes (METX-8/9).

The foundation requires the row arithmetic (`Subrow`, `row_union_t`) for the dispatcher's effect-row check. Without these, the dispatcher cannot verify a callee against a caller's context at compile time — it can only check exact-match, which forbids legitimate row narrowing.

**Lines, headers.** Per `25_04_2026.md` §3.5: ~1–2 KLoC for the Computation carrier and row arithmetic, plus ~500 lines of static_assert tests. Five new headers under `effects/`: `EffectRow.h`, `Computation.h`, `Capabilities.h`, `compat/Fx.h`, plus a verification harness `effects/Verify.h`.

**Dependencies.** None — Met(X) is bottom-of-stack. Required by §5.5 (signature dispatcher's effect check), §5.4 (auto-permission-tree generator must be agnostic to effect rows but compose with them), §5.11 (the fusion analyzer needs `Subrow` to decide whether two adjacent computations can fuse).

### 5.2 PermissionedSessionHandle — the unfinished thesis

**Status: ✅ SHIPPED as FOUND-C v1, 2026-04-27.** Implementation plan: `misc/27_04_csl_permission_session_wiring.md`. Three new headers: `permissions/PermSet.h`, `sessions/SessionPermPayloads.h`, `sessions/PermissionedSession.h`. 15/15 tracked tasks complete (#605–#619). 221/221 ctest green at landing.

**Gap (closed).** `sessions/Session*.h` defines binary and multiparty session types over twelve layers (Honda 1998, HYC 2008, BSYZ22 crash-stop, Gay-Hole subtyping, etc.) — about 8400 lines and 102 tests. After FOUND-C v1, the framework has a `PermissionedSessionHandle<Proto, PS, Resource, LoopCtx>` that CRTP-inherits `SessionHandleBase` and threads CSL permissions alongside protocol state. Production wiring (K-series tasks #355–#358 / SAFEINT-R31 #413) remains pending.

**Foundation requirement.** The thesis primitive is exhaustively specified in `24_04_2026_safety_integration.md` §5 and the FOUND-C wiring plan; this document does not duplicate either. Foundation summary:

**Integration.** Every Crucible channel that crosses thread or process boundaries is a candidate consumer: TraceRing → MetaLog drain (§31 of the safety-integration doc), KernelCache publication (§32), CNTP Layer 1 (§34), Cipher hot-tier reads (§24). The thesis closes the L9 layer of the session-type stack and, alongside it, gives the dispatcher a uniform protocol-handle type to consume in shapes §3.4, §3.5, §3.7.

The foundation cannot defer this work. The signature dispatcher cannot route the producer/consumer/pipeline shapes without knowing what handle types to expect; the only reasonable canonical handle is `PermissionedSessionHandle`. Until task #333 ships, the dispatcher's consumer/producer routing must accept the bare `Queue<T, K>::ProducerHandle` from the existing Permission-extension in `concurrent/Queue.h` as a temporary bridge.

**Lines, headers.** Per `24_04_2026_safety_integration.md` §5.4: ~600 lines for the new header (`safety/PermissionedSession.h`), plus negative-compile tests. Builds on existing `safety/Session*.h` foundation; no new substrates introduced.

**Dependencies.** Met(X) (§5.1) is helpful but not required — PermissionedSessionHandle works with the legacy `fx::*` model during the transition. It is required by every Permissioned* wrapper in §5.3 and by the signature dispatcher's protocol-handle recognition in §5.5.

### 5.3 PermissionedMpsc / PermissionedMpmc / PermissionedShardedGrid — the CSL coverage gap

**Gap.** Among the six concurrency primitives in substrate 5, only two ship a Permissioned wrapper today: PermissionedSpscChannel and PermissionedSnapshot. The other four — MpscRing, MpmcRing, ShardedSpscGrid, ChaseLevDeque — expose only bare-handle factories. The foundation principle (§1.4) demands that every concurrent primitive carry CSL-typed handles at its boundary. Tasks #309 (PermissionedShardedGrid), #327 (PermissionedMpmcChannel core), #562/#563/#564 (the new tracking tasks) cover this. `THREADING.md` §5.5.1 specifies the design; this section restates the foundation requirement.

**Foundation requirement.** Each of the four primitives gets a Permissioned* wrapper that:

1. Holds the underlying lock-free primitive as a Pinned member.
2. Publishes one or two `SharedPermissionPool` instances (one per role: producer, consumer, or both for MPMC).
3. Hands out role-typed handles via `lend()`-style factory methods: `producer()`, `consumer()`, `producer_share()`, etc.
4. Each handle carries `[[no_unique_address]] SharedPermissionGuard<role_tag>` plus a pointer to the channel — so `sizeof(Handle) == sizeof(void*)` post-EBO.
5. Each handle exposes only the operations valid for its role: `ProducerHandle` has `try_push` / `try_push_batch` only; `ConsumerHandle` has `try_pop` / `try_pop_batch` only. Compile error on cross-role calls.
6. Mode-transition primitive `with_drained_access(body)`: atomic upgrade of all pools to exclusive, body runs with zero live handles, RAII deposit on body return.

For PermissionedMpmcChannel specifically (THREADING.md §5.5.1), the wrapper has TWO independent pools — one for `Producer<UserTag>` and one for `Consumer<UserTag>`. Each refcounts its side's live handles. `with_drained_access` atomically upgrades both pools.

For PermissionedShardedGrid (M producers × N consumers), the permission tree is auto-generated:

```cpp
template <typename UserTag, std::size_t M, std::size_t N>
struct shard_tag {
    template <std::size_t I> struct Producer {};
    template <std::size_t I> struct Consumer {};
    struct Whole {};
};

// splits_into_pack synthesized via reflection + expansion statements
template <typename UserTag, std::size_t M, std::size_t N>
struct splits_into_pack<shard_tag<UserTag, M, N>::Whole,
                        shard_tag<UserTag, M, N>::template Producer<0..M-1>...,
                        shard_tag<UserTag, M, N>::template Consumer<0..N-1>...>
    : std::true_type {};
```

The synthesis uses P2996R13 `define_static_array` plus `template for` to generate the pack at compile time without macros (THREADING.md §3.2 details the technique).

**Integration.** The dispatcher's producer/consumer/SWMR shapes (§3.4–§3.6) all consume Permissioned handles. The TraceRing migration (SAFEINT-R31, task #413) and KernelCache migration (SAFEINT-R32, task #414) and several CNTP/Cipher sites (SAFEINT-R33/34/35/36, tasks #415–418) all become straightforward once the four wrappers exist.

**Lines, headers.** Each wrapper is approximately 200–300 lines, mirroring the existing PermissionedSpscChannel template. Total ~1000 lines plus tests. Four new headers under `concurrent/`: `PermissionedMpscChannel.h`, `PermissionedMpmcChannel.h`, `PermissionedShardedGrid.h`, `PermissionedChaseLevDeque.h`.

**Dependencies.** PermissionedSessionHandle (§5.2) optional — the wrappers can ship without it as bare-Permission wrappers, then upgrade to PermissionedSessionHandle later. The auto-permission-tree generator (§5.4) is required for the M×N case in PermissionedShardedGrid.

### 5.4 Auto-permission-tree generator

**Gap.** Every Permission tag tree in Crucible today is hand-declared via a `splits_into` or `splits_into_pack` specialization adjacent to the tag struct definition. This works for the half-dozen concrete trees that exist (TraceRing producer/consumer, MetaLog producer/consumer, AugurReader, kernel-compile pool worker shards), but for the dispatcher to generate trees from `OwnedRegion<T, Tag>&&` parameters automatically, the foundation needs a generator: given a Whole tag and a parallelism factor N, emit `splits_into_pack<Whole, Slice<Whole, 0>, ..., Slice<Whole, N-1>>`.

**Foundation requirement.** A new header `safety/PermissionTreeGenerator.h` providing:

```cpp
namespace crucible::safety {

// Slice<Parent, I> — phantom indexed sub-tag.
template <typename Parent, std::size_t I>
struct Slice {
    using parent_tag = Parent;
    static constexpr std::size_t index = I;
};

// Auto-generate splits_into_pack for any (Parent, N) pair.
template <typename Parent, std::size_t N>
struct auto_split_n {
    template <std::size_t... Is>
    struct impl {
        // The specialization that splits_into_pack consults.
        // Template-for over Is generates the pack; the
        // partial specialization fires the trait.
    };
    using type = impl<std::make_index_sequence<N>>;
};

// Fully-generated specialization — implementation detail; users
// just pass Parent + N and the trait works.
template <typename Parent, std::size_t N, std::size_t... Is>
struct splits_into_pack<Parent, Slice<Parent, Is>...> : std::true_type {};

// Convenience trait the dispatcher consumes.
template <typename Parent, std::size_t N>
inline constexpr bool can_split_n_v =
    splits_into_pack_v<Parent, /* generated pack */>;

}  // namespace crucible::safety
```

For the M×N grid case (PermissionedShardedGrid in §5.3), the generator extends naturally to two-dimensional packs:

```cpp
template <typename Whole, std::size_t M, std::size_t N>
struct auto_split_grid {
    // Generate 2D pack: ProducerSlice<0..M-1> + ConsumerSlice<0..N-1>
    // via nested template-for.
};
```

The generator must be reflection-friendly: P2996R13 `display_string_of(^^Slice<Parent, I>)` should produce a readable name for diagnostics ("Slice<MyData, 3>" not a mangled symbol).

**Integration.** Consumed by every Permissioned* wrapper in §5.3 (so users don't write per-tag `splits_into` declarations), by the signature dispatcher (so unary/binary transforms get auto-generated trees from `OwnedRegion<T, Tag>` parameters), and by `parallel_for_views<N>` itself (so the same N value triggers the same auto-split everywhere).

**Lines, headers.** ~150 lines in one new header. The reflection machinery is the bulk; the actual specialization is short.

**Dependencies.** None on other §5 specs. Required by §5.3 (Permissioned wrappers) and §5.5 (signature dispatcher).

### 5.6 Parameter-shape concepts (the protocol's enforcement)

**Gap.** The seven parameter shapes in §3 are intuition; the foundation needs them as enforceable concepts so the dispatcher's overloading resolves correctly. §5.5's shape recognizers are the concept names; this section fills in their specifications.

**Foundation requirement.** Each shape concept is a `requires` clause over the parameter list. Examples:

```cpp
namespace crucible::safety::extract {

template <auto FnPtr>
concept UnaryTransform =
    signature_traits<FnPtr>::arity == 1
    && is_owned_region_v<param_type_t<FnPtr, 0>>
    && std::is_rvalue_reference_v<param_type_t<FnPtr, 0>>
    && (std::is_void_v<return_type_t<FnPtr>>
        || is_owned_region_v<return_type_t<FnPtr>>);

template <auto FnPtr>
concept Reduction =
    signature_traits<FnPtr>::arity == 2
    && is_owned_region_v<param_type_t<FnPtr, 0>>
    && std::is_rvalue_reference_v<param_type_t<FnPtr, 0>>
    && is_reduce_into_v<param_type_t<FnPtr, 1>>
    && std::is_lvalue_reference_v<param_type_t<FnPtr, 1>>;

template <auto FnPtr>
concept ProducerEndpoint =
    signature_traits<FnPtr>::arity == 2
    && is_producer_handle_v<param_type_t<FnPtr, 0>>
    && std::is_rvalue_reference_v<param_type_t<FnPtr, 0>>
    && is_owned_region_v<param_type_t<FnPtr, 1>>;

// ... etc., one concept per shape in §3
}
```

The concepts must be mutually exclusive on the canonical shapes — a function cannot satisfy both UnaryTransform and Reduction. They must be ordered (most specific first) so concept-overload resolution picks the right lowering. The foundation publishes the ordering.

For the catch-all (§3.8), the foundation provides:

```cpp
template <auto FnPtr>
concept CanonicalShape =
    UnaryTransform<FnPtr>
    || BinaryTransform<FnPtr>
    || Reduction<FnPtr>
    || ProducerEndpoint<FnPtr>
    || ConsumerEndpoint<FnPtr>
    || SwmrWriter<FnPtr>
    || SwmrReader<FnPtr>
    || PipelineStage<FnPtr>;

template <auto FnPtr>
concept NonCanonical = !CanonicalShape<FnPtr>;
```

The dispatcher refuses to handle `NonCanonical` functions and emits a diagnostic naming the parameter at index 0 and citing which canonical shape would have admitted it.

**Lines, headers.** ~300 lines (one concept per shape, ~30 lines each). One header `safety/extract/Shapes.h` (or absorbed into §5.5).

**Dependencies.** §5.5's extraction traits.

### 5.7 Cache-tier-only cost model (refresh)

**Gap.** `concurrent/CostModel.h` already publishes the cache-tier rule (THREADING.md §5.4). Its `WorkBudget` struct has a `per_item_compute_ns` field that the cleanup pass (commit 2026-04-27) reframed as a caller-supplied hint, not a runtime promise. The foundation principle (§1.2) says abstract cost dimensions are out of scope. The cost model needs a refresh that drops the abstract dimension entirely and operates on bytes + topology only.

**Foundation requirement.** A revised `WorkBudget` and `recommend_parallelism` API that takes only what's measurable:

```cpp
namespace crucible::concurrent {

struct WorkBudget {
    std::size_t read_bytes  = 0;
    std::size_t write_bytes = 0;
    std::size_t item_count  = 0;
    // No per_item_compute_ns. No per_item_compute_class.
    // The dispatcher chooses Sequential vs Parallel(N) from
    // working_set = read_bytes + write_bytes against the cache hierarchy.
};

enum class Tier : std::uint8_t {
    L1Resident,    // working_set < L1d_per_core
    L2Resident,    // working_set < L2_per_core
    L3Resident,    // working_set < L3_per_socket
    DRAMBound,     // anything else
};

enum class NumaPolicy : std::uint8_t {
    NumaIgnore, NumaLocal, NumaSpread,
};

struct ParallelismDecision {
    enum Kind { Sequential, Parallel } kind;
    std::size_t factor = 1;       // 1, 2, 4, 8, 16, ... up to core count
    NumaPolicy  numa  = NumaPolicy::NumaIgnore;
    Tier        tier  = Tier::L1Resident;
};

// The decision rule — depends only on bytes + topology, never on ns.
[[nodiscard]] ParallelismDecision recommend_parallelism(WorkBudget) noexcept;

}  // namespace crucible::concurrent
```

The decision rule (concrete, no abstract cost):

```
ws = read_bytes + write_bytes
core_count = Topology::instance().num_cores()

if ws < L1d_per_core:   Sequential
elif ws < L2_per_core:  Sequential
elif ws < L3_per_socket: Parallel(min(4, cores_per_socket), NumaLocal)
else (DRAM-bound):       Parallel(min(core_count, ws / L2_per_core), NumaLocal)
```

The "compute-bound override" branch in the current code (which raises N when `per_item_compute_ns > 100`) is removed. Without ns hints, the rule is purely cache-driven; that is its honest scope.

**Migration.** The existing `WorkBudget::per_item_compute_ns` field is removed. Callers passing it today (none in production code; a few test fixtures) update to omit it. The cleanup pass already reframed the field's docstring as a hint; this spec completes the work by removing the field.

**Lines, headers.** Net negative: deletes `per_item_compute_ns` field and the compute-bound override branch (~30 lines). One header touched. Adds tests verifying the decision rule produces expected outputs across (working_set × core_count × NUMA topology) inputs (~100 lines).

**Dependencies.** None — cost model is bottom-of-stack. Required by §5.5 dispatcher's N selection.

### 5.9 Reflection-based stable type naming

**Gap.** P2996R13 provides `display_string_of(^^T)` which produces a readable type name. ### 5.11 Fusion analyzer (call-graph level)

**Gap.** The dispatcher should be able to detect when two adjacent dispatch calls compose two pure functions, and inline them into one parallel_for_views body. This is call-graph fusion (one function inlined into another), not loop-level fusion (two loops combined into one tile). The foundation must publish the predicate the dispatcher consumes.

**Foundation requirement.** A header `safety/Fusion.h`:

```cpp
namespace crucible::safety::fusion {

// Two functions are fusable iff:
//   1. Both have shape UnaryTransform (or compatible chain shapes)
//   2. Both have empty effect rows (Pure)
//   3. The output region of the first matches the input region of the second
//   4. Neither has a Pinned wrapper in its parameters (Pinned implies side
//      effects through a stable address)
//   5. Neither carries a non-Absolute modality grade (e.g., Stale<T>'s
//      staleness can't compose freely across pure boundaries)
template <auto Fn1, auto Fn2>
inline constexpr bool can_fuse_v =
       extract::CanonicalShape<Fn1>
    && extract::CanonicalShape<Fn2>
    && Subrow<inferred_row_t<Fn1>, EmptyRow>
    && Subrow<inferred_row_t<Fn2>, EmptyRow>
    && /* output of Fn1 matches input of Fn2 */
    && /* no Pinned in either */
    && /* no non-Absolute grades */;

// Composed function — produces a callable that runs Fn1 then Fn2 on the
// same iteration body.
template <auto Fn1, auto Fn2>
    requires can_fuse_v<Fn1, Fn2>
constexpr auto fuse() noexcept -> /* generated lambda */;

}  // namespace crucible::safety::fusion
```

The generated lambda from `fuse<Fn1, Fn2>()` has the shape of Fn1's input → Fn2's output, and its body is `Fn2(Fn1(x))` for each element. The compiler inlines through both, producing one tight loop.

**Integration.** The dispatcher detects adjacent dispatches and calls `fuse` to compose them. This is the meaningful magic the user gets without writing fusion themselves.

**Lines, headers.** ~200 lines. One new header.

**Dependencies.** §5.1 (Met(X) for Subrow), §5.5 (extraction traits), §5.6 (shape concepts).

### 5.12 Session-type composition with CSL

**Gap.** Session types and CSL permissions today live in adjacent directories but compose only in three places (`PermissionedSpscChannel`, `PermissionedSnapshot`, `concurrent/Queue.h`'s Permission-typed handle support). The signature-as-spec dispatcher needs the third axis — protocols — to be a first-class compositional citizen. Tasks #339 (SessionGlobal.h with global types + projection), #345 (SessionAssoc.h association invariant), #346 (SessionSafety.h φ family), #347 (SessionCrash.h crash-stop) are tracked. This document specifies what the foundation requires.

**Foundation requirement.** Beyond what `safety/Session*.h` already publishes, the foundation requires:

1. **Global types + projection** (task #339, SessionGlobal.h). Honda-Yoshida-Carbone 2008 multi-party protocols. Given a global type G and a role R, `project_t<G, R>` produces R's local protocol. Used by the dispatcher's pipeline-stage shape (§3.7) to verify each stage's local protocol is the projection of the pipeline's global protocol.

2. **Association invariant Δ ⊑_s G** (task #345, SessionAssoc.h). Hou-Yoshida-Kuhn 2024 master theorem. Verifies that a local context Δ is consistent with the global type G modulo subtyping. Required for the dispatcher to compose multiple PermissionedSessionHandles into a coherent multi-party protocol.

3. **The φ family** (task #346, SessionSafety.h). Communication safety (no message stuck), session fidelity (every send is matched by a receive), deadlock freedom, termination. Crucible's L7 layer of the session-type stack. The dispatcher does not directly consume φ but the type system carries them — every protocol the user can express is φ-safe by construction.

4. **Crash-stop extensions** (task #347, SessionCrash.h). BSYZ22 / BHYZ23. Stop, Crash<Peer>, UnavailableQueue<Peer> primitives. Plus the runtime crash-flag wiring per `24_04_2026_safety_integration.md` §11 (OneShotFlag-driven crash transitions). Required for the dispatcher's SwmrWriter / SwmrReader shapes (§3.6) and any cross-process protocol.

5. **Content-addressed quotient combinator** (SEPLOG-L1, task #361). For a protocol whose payload is the same up to content hash, a `ContentAddressed<T>` payload marker that lets the type system recognize equivalent payloads and dedup them. The KernelCache publication path consumes this.

6. **Checkpointed session combinator** (SEPLOG-L2, task #362). For a session that needs to be checkpointed and resumed (Cipher integration), a `CheckpointedSession<CommitPath, RollbackPath>` combinator. Consumed by Transaction.h's refactor (SAFEINT-R30, task #412).

**Integration.** The dispatcher's protocol-handle recognition (§5.5 `is_session_handle_v`) is satisfied by any of these handle types. The composition rules from §5.11 (fusion) extend to "two pipeline stages whose protocols are composable" — same predicate, broader scope.

**Lines, headers.** Per `session_types.md` §IX and the SAFEINT roadmap: ~3000 lines across the six items above. Tracked in tasks #339, #345, #346, #347, #361, #362.

**Dependencies.** PermissionedSessionHandle (§5.2). Met(X) (§5.1) for the row composition.

### 5.13 Cipher integration for compiled-body cache

**Gap.** Cipher today persists DAG state, weight snapshots, and a content-addressed kernel cache (CRUCIBLE.md §L7, Phase 1 already shipped). The foundation needs an additional integration: when the dispatcher generates a lowering for a `dispatch(fn, args)` call, it should be able to look up the compiled body in Cipher by `stable_function_id<fn>` (§5.9), and if found, use it directly. If not found, generate, optionally store. This is computation-result caching at the function-handle level, distinct from kernel-bytecode caching.

**Foundation requirement.** A new header `Cipher/ComputationCache.h` providing:

```cpp
namespace crucible::cipher {

// Look up a compiled body by stable function ID + parameter type IDs.
// Returns a pointer to the compiled body if found, nullptr otherwise.
template <auto FnPtr, typename... Args>
[[nodiscard]] CompiledBody* lookup_computation_cache() noexcept;

// Insert a compiled body. Idempotent: if the same (fn, params) is
// inserted twice, the second insert is a no-op.
template <auto FnPtr, typename... Args>
void insert_computation_cache(CompiledBody*) noexcept;

// Drain the cache (cold-tier eviction).
void drain_computation_cache(std::chrono::seconds max_age) noexcept;

}  // namespace crucible::cipher
```

`CompiledBody` is implementation-defined; the foundation publishes the API but the dispatcher decides what shape the cached body takes (a function pointer, a code blob, a serialized computation graph).

**Integration.** The dispatcher consults the cache before generating a lowering. Stable function ID + stable parameter type IDs = cache key. Hit ⇒ use cached body; miss ⇒ generate, insert, use. This is the dispatcher analog of the KernelCache for kernel bytecode.

**Lines, headers.** ~250 lines. One new header. Plus one Cipher tier extension to support content-addressed function IDs alongside the existing content-addressed merkle hashes.

**Dependencies.** §5.9 (stable type/function IDs).

**Integration.** Consumed by every concurrency primitive in §5.3. Validated by the bench harness (`bench/bench_concurrent_queues.cpp` — 0.075 ns/item L1d-saturation floor at N=512–1024 for the SPSC ring's batched API).

### 5.16 OwnedRegion<T, Tag> — the canonical region type

**Gap.** `safety/OwnedRegion.h` is partially shipped. The foundation requires it as the canonical region type for every UnaryTransform / BinaryTransform shape. Concrete needs:

1. `OwnedRegion<T, Tag>` with `[[no_unique_address]] Permission<Tag> perm_` so sizeof equals (T*, size_t).
2. `OwnedRegion::adopt(Arena&, std::size_t count, Permission<Tag>&&)` for arena-backed construction.
3. `OwnedRegion::wrap(T* ptr, std::size_t count, Permission<Tag>&&)` for adopting externally-allocated memory (the FFI boundary case).
4. `OwnedRegion::span()` and `cspan()` for direct iteration.
5. `OwnedRegion::split_into<N>()` returning `array<OwnedRegion<T, Slice<Tag, I>>, N>`.
6. `OwnedRegion::recombine<N>(array<...>&&)` for the inverse.
7. Linear discipline: deleted copy, default move with reason strings.
8. Pinned discipline (no move once distributed across threads — but movable while still under sole ownership).

**Foundation requirement.** Complete the partial shipment to satisfy all 8 requirements above. Most exist; this is consolidation work.

**Lines, headers.** ~300 lines for completion. One header (extends existing).

**Dependencies.** §5.4 (auto-permission-tree generator) for split_into / recombine.

### 5.17 ReplayLog<Ordering> for async-safe replay (Cipher extension)

**Gap.** `25_04_2026.md` §4 specifies the async-safe replay extension to Cipher. The foundation requires it as a substrate component because Stale-graded computations (substrate 2's `Stale<T>` wrapper) are unsound under bit-exact replay without it. The dispatcher's recognition of `Stale<T>` parameters (§5.5) implicitly requires that ReplayLog be available so the dispatcher can route async-grade dispatches correctly.

**Foundation requirement.** Implement the design from `25_04_2026.md` §4. ~3–5 KLoC for the log writer/reader, integration with existing Cipher tiers, and admission-policy hooks.

**Integration.** Composes with `Stale<T>` — every Stale-graded value's admission to a synchronous Cipher tier is recorded in the ReplayLog with the staleness tag. Replay extracts the tag and re-applies the same admission decision.

**Lines, headers.** ~3000 lines. Tracked separately; not in critical path for foundation.

**Dependencies.** Cipher (substrate 7).

### 5.18 Topology — extended for cache-line and NUMA hint propagation

**Gap.** `concurrent/Topology.h` already publishes L1/L2/L3 sizes, NUMA distances, core counts. The foundation needs a small extension: a way for parameter types to declare "this region prefers NUMA-local placement" via a tag-side hint, propagated to the dispatcher's NUMA policy selection.

**Foundation requirement.** A new tag concept `safety::NumaPreference` that specific Tag types can opt into:

```cpp
namespace crucible::safety {

template <typename Tag>
concept HasNumaPreference = requires {
    typename Tag::numa_preference;
    requires std::is_same_v<typename Tag::numa_preference, NumaLocal_t>
          || std::is_same_v<typename Tag::numa_preference, NumaSpread_t>
          || std::is_same_v<typename Tag::numa_preference, NumaIgnore_t>;
};

struct NumaLocal_t  {};
struct NumaSpread_t {};
struct NumaIgnore_t {};

}  // namespace crucible::safety
```

A user defines a tag like `struct MyData { using numa_preference = NumaLocal_t; }` and the dispatcher reads the preference via reflection. Tags without the preference inherit from the cache-tier rule's default.

**Lines, headers.** ~50 lines. One header (extends Topology or sits adjacent).

**Dependencies.** §5.7 (cost model consumes the preference).

### 5.19 Runtime hardening policy + bench integration

**Gap.** `rt/Policy.h` publishes `crucible::rt::Policy` with sched-FIFO, mlock, THP flags. The bench harness already integrates via `Run::hardening(p)`. The foundation requires that the dispatcher can also apply a hardening policy at lowering time, so production callers get the same isolation the bench gets.

**Foundation requirement.** Extend the dispatcher entry point to accept an optional Policy:

```cpp
namespace crucible {
template <auto FnPtr, typename... Args>
auto dispatch(rt::Policy const&, Args&&... args);
}
```

The Policy is applied via `rt::apply()` which returns an RAII guard; the dispatcher holds the guard for the duration of the lowering. Defaults to no hardening.

**Lines, headers.** ~50 lines (dispatcher extension + Policy plumbing).

**Dependencies.** §5.5.

### 5.20 The eight-axiom verification harness, extended

**Gap.** Crucible's eight safety axioms (CLAUDE.md §II) — InitSafe, TypeSafe, NullSafe, MemSafe, BorrowSafe, ThreadSafe, LeakSafe, DetSafe — are verified per primitive via the `test/test_migration_verification.cpp` harness and per-axiom test files. The foundation requires that every new primitive in §5.1–§5.13 has matching verification entries.

**Foundation requirement.** Per primitive, a test in `test/safety_axioms/` that exercises the primitive's contribution to each of the eight axioms. Plus the cross-cutting `test_signature_dispatch.cpp` that validates the dispatcher routes correctly for every shape × every primitive combination.

**Lines, headers.** ~200 lines per primitive × 8 primitives = ~1600 lines. Plus ~500 lines for the cross-cutting harness.

**Dependencies.** All §5 specs.

## 6. Composition matrix

This matrix specifies how each pair of foundation substrates composes. A cell entry of "✓" indicates the composition is well-defined and used in production. "→" indicates one-way derivation. "✗" indicates the composition is forbidden by foundation discipline. A blank indicates the composition is irrelevant.

|  | Graded (1) | Wrappers (2) | Met(X) (3) | CSL (4) | Concur (5) | Sessions (6) | Cipher (7) | Reflect (8) |
|---|---|---|---|---|---|---|---|---|
| **Graded (1)** | ✓ | → | composes via Row | grade per Permission | grade per Cell | grade per payload | grade per persisted value | reflected for diagnostics |
| **Wrappers (2)** | ✓ | ✓ | wraps Computation | wraps Permission | wraps Handle | wraps Resource | wraps Cipher entry | reflected for type names |
| **Met(X) (3)** | ✓ | ✓ | ✓ | annotates Permission ops | annotates ring ops | composes with Session row | persists with row tag | row introspection |
| **CSL (4)** | ✓ | ✓ | ✓ | ✓ | every channel CSL-typed | PermissionedSessionHandle | persists with permission tag | reflected for tree generation |
| **Concur (5)** | ✓ | ✓ | ✓ | ✓ | ✓ | sessions-typed channels | ring → Cipher tier | reflected for Kind selection |
| **Sessions (6)** | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | session log → Cipher | reflected for protocol display |
| **Cipher (7)** | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | reflected for cache keys |
| **Reflect (8)** | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ | ✓ |

The matrix encodes the foundation's compositional discipline. Every wrapper composes with every other wrapper; every wrapper carries its own grade; every grade composes with every other grade via lattice join; every primitive carries CSL permissions at its boundary; every persistent operation goes through Cipher.

The dispatcher's job (out of foundation scope) is to route a specific user-call site through the right cells of this matrix. The foundation's job is to ensure each cell is well-defined.

## 8. What we explicitly do NOT do

Documenting non-decisions is as important as documenting decisions. These prevent future iterations from re-litigating settled choices.

### 8.1 No DSL

The foundation is plain C++26. Users write regular C++26 functions. Parameter types are the spec. There is no embedded DSL the user must learn — no `[[crucible::parallel]]` attribute syntax to memorize, no `pipe(...) | map(...) | filter(...)` combinator chain, no `make_computation<Spec...>(body)` spec pack.

This is not because DSLs are bad; it is because Crucible's existing wrappers already constitute an implicit spec language. Adding an explicit DSL on top would force users to learn TWO vocabularies.

### 8.2 No abstract cost dimension

No `Cost<bytes, ns_per_item>`, no `[[crucible::compute_ns(N)]]`, no `LatencyBudget`, no FLOP-intensity targets. Cost is modeled via concrete hardware facts only — bytes, cache sizes, NUMA distances. This was committed by the cleanup pass on 2026-04-27 and is restated as foundation discipline.

The cost-model decision (`recommend_parallelism`) is honest about its scope: it picks Sequential vs Parallel(N) from working-set bytes against the cache hierarchy, nothing more. It does not promise nanoseconds, does not estimate complexity, does not model FLOP intensity. Users who need finer-grained control fall through to manual orchestration.

### 8.3 No JIT, no compiler plugin, no body inspection

The foundation works entirely within the C++26 surface that GCC 16 provides. P2996 reflection sees declarations, not statement bodies. We do not parse user function bodies, do not generate code at runtime, do not invoke clang plugins. Anything that depends on body-AST inspection (auto-vectorization, loop fusion across statements, automatic tile sizing) is handled by the compiler's auto-vectorizer or by users writing manual `parallel_for_views` calls.

### 8.4 No latency promises

The foundation publishes structural per-call shape (number of atomic ops, number of cache-line touches, branch counts) but no latency guarantees. Users observe whatever latency the hardware delivers under their workload. Bench files report observed latencies on specific hardware/dates as data, not as promises.

### 8.5 No optimistic API surface

Every foundation API is conservative: it requires explicit type-level commitments (Permission tags, OwnedRegion lifetimes, Refined predicates, Effect rows) and refuses to dispatch when those commitments aren't in place. The dispatcher emits a diagnostic, not a default-best-effort. Users who want default-best-effort fall through to manual orchestration via the existing direct primitives.

### 8.6 No dynamic shape changes (within a single dispatch)

A `dispatch(fn, args)` call with parameter types fixed at compile time produces a lowering with a fixed shape. The dispatcher does not adapt to runtime inputs (e.g., picking a different N based on observed throughput). Adaptation happens between dispatch calls — the next call can use a different fn with different parameter types — not within a call.

### 8.7 No global mutable scheduler state

The dispatcher does not register call sites with a global scheduler, does not maintain a global thread pool that must be initialized at process startup, does not have hidden lifetime dependencies on a singleton. Each dispatch is a self-contained computation; the underlying primitives (jthreads, Permission roots, OwnedRegions) are constructed and destroyed within the call's lifetime.

The exception is the topology probe (`Topology::instance()`) which is a process-wide cached singleton — but it is read-only and constructed at first access, so it has no initialization-order issues.

## 9. Open questions

These are the design questions the foundation specification leaves open. Each requires either further research or a deliberate choice before the affected component can ship.

### 9.1 Effect-row inference from opaque callables

When a user's function calls a third-party library function that has no Met(X) row annotation (e.g., `printf`, vendor math library, std::filesystem), the foundation cannot infer the row. Three options:

1. **Conservative pessimism**: any non-annotated callee is assumed to have row `Row<Alloc, IO, Block>`, forcing the calling pipeline to require `fx::Bg`. Pipelines using third-party code lose the "purely inline run" path.

2. **Annotation-only**: the foundation refuses to dispatch any function whose body calls a non-annotated callee. Forces users to wrap third-party calls in annotation shims. High discipline cost.

3. **External annotation file**: the foundation reads a sidecar `effect_rows.json` mapping symbol names to declared rows. Users maintain the file; framework consumes it.

Recommendation: ship (1) for v1. Revisit if users demand finer control.

### 9.2 Refinement implication propagation

`safety/Refined.h` has `implies_v<P, Q>` for cross-predicate refinement (positive ⇒ non_negative). The dispatcher could use this to safely narrow a function's input refinement when composing two stages. Open: should the dispatcher apply implication transitively (positive ⇒ non_negative ⇒ bounded_below_by<-MAX>)? Transitivity might surprise users.

Recommendation: ship implications as direct (one hop) for v1. Add transitive closure later if a use case emerges.

### 9.3 Ordering of canonical shapes

When a user function satisfies multiple shape concepts (e.g., a function that takes `OwnedRegion` and `reduce_into` could be either Reduction or BinaryTransform), which lowering wins? The foundation publishes an ordering (most specific first), but the choice is somewhat arbitrary. Reduction is more specific than BinaryTransform; SwmrWriter is more specific than ProducerEndpoint.

Recommendation: document the ordering in CLAUDE.md and treat it as load-bearing — changing it would silently reroute existing code. Cite the discipline by analogy to `[[gnu::flatten]]` where order matters.

### 9.4 Parameter shape ambiguity in Cipher cache keys

If two functions have the same `stable_function_id` but different bodies (e.g., two functions with identical signatures but different implementations), the cache key collides. The foundation could either include a body hash (impossible — body unreflectable) or a build-time content hash of the source file (possible but complex).

Recommendation: include the `stable_function_id` plus the source file hash plus the build hash plus a timestamp in the cache key. Cache hits require all to match; misses are silent regenerations.

### 9.5 Cross-process Permission flow

When a session crosses a process boundary (CNTP Layer 1), the receiving process cannot literally hold the same `Permission<Tag>` token as the sending process. The thesis from `24_04_2026_safety_integration.md` Appendix C question 2 covers this: the receiving process mints fresh Permissions locally; the cross-process bytes never carry permissions directly. Open: how does the foundation publish this discipline? Is it a documented pattern, or a typed primitive (`CrossProcessPermissionEstablishment<Tag>`)?

Recommendation: document the pattern for v1; add the typed primitive when CNTP Layer 1's typed-session migration (SEPLOG-K4, task #358) is in flight.

### 9.6 Lean obligation generation

Each foundation primitive could in principle emit a Lean obligation file alongside its compile output, encoding the type-level invariants the primitive promises. The `lean/Crucible/` development (1300+ theorems, 0 sorry) is the home for these. Open: should the foundation auto-generate the obligation files via reflection, or do humans write them by hand?

Recommendation: human-written for v1. The reflection-driven auto-generation would be a Phase 6+ research item.

### 9.7 What the dispatcher itself looks like

This document deliberately defers the dispatcher API. The foundation makes any dispatcher possible — signature-as-spec, type-spec, dataflow combinators, annotated-imperative — because all four can read the same foundation primitives. Open: which dispatcher does Crucible ship first?

Recommendation: prototype signature-as-spec (the simplest) first; let production callers reveal whether more vocabulary is needed; decide.

## 10. Closing

The foundation is the substrate that makes a smart user-facing API possible without compromise. Eight substrates, twenty component specs, six phased deliverables, ~12 KLoC of new code. Every component composes with every other; every cell of the composition matrix is well-defined; every primitive carries CSL permissions at its boundary; every grade is a lattice operation; every effect row is a Subrow check; every protocol is a session type; every persistent operation goes through Cipher; every type is reflectable down to its template arguments.

The user-facing API question — how the user actually expresses computations on top of this foundation — is deliberately deferred. Three architectural angles (dataflow, annotated, type-spec) and one synthesis (signature-as-spec) have been surveyed; the foundation supports any of them. The choice between them is a user-experience decision, made later when we have production callers to design backwards from.

The foundation's discipline:

- Zero abstraction overhead, verified per primitive.
- Concrete hardware facts as the cost model; no abstract latency or complexity dimensions.
- Type-as-spec — the user's parameter types ARE the specification.
- CSL permissions at every boundary — every channel, every region, every fork-join.
- Signature dispatch — the framework reads, doesn't ask for, the spec.
- C++26 native — no DSL, no plugin, no JIT.

When the foundation ships in full (28 weeks at one engineer; 14 weeks at three), Crucible has the substrate to support arbitrarily ambitious user-facing API surfaces. The foundation is the load-bearing pour; the user-facing API is the building shape we choose to put on top.
