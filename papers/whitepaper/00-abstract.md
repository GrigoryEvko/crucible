# Abstract

Machine learning runtimes execute computation without recording, adapting, or formally verifying it. Each operation dispatches independently --- the same kernel runs regardless of shape, device, or surrounding context. Memory is allocated dynamically per operation. Parallelism strategy is configured manually and remains fixed throughout training. When hardware fails, work is lost back to the last checkpoint. No runtime invariant is proved; correctness depends entirely on testing.

We present Crucible, an adaptive ML runtime built around a content-addressed Merkle DAG that serves as the single representation for compilation, memory planning, distribution, and persistence. Crucible interposes on the framework dispatch layer (the *Vessel*) to record a complete execution trace, constructs a dataflow graph with content-based hashing, and compiles execution plans where identical sub-computations produce identical hashes --- enabling kernel reuse across models, runs, and organizations. We propose SMT-based kernel configuration synthesis that encodes hardware constraints and cost models as satisfiability problems, yielding configurations that are optimal within the analytical model and carry formal safety guarantees (no out-of-bounds access, no shared memory bank conflicts, coalesced memory access). Static memory planning from dataflow lifetimes eliminates runtime allocation entirely. An event-sourced persistence model (the *Cipher*) enables deterministic replay and hardware migration.

We describe the system architecture spanning seventeen layers from formal verification (FX) to ecosystem-wide kernel sharing. We report on our prototype implementation covering the recording pipeline, graph construction, Merkle DAG, memory planning, and compiled replay with divergence recovery. We accompany the implementation with a Lean 4 formalization proving properties of every core data structure and algorithm, with no unresolved proof obligations.
